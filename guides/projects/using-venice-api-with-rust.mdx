If you've been curious about Rust but haven't built anything real with it yet, an LLM integration is a friendly place to start. The scope is small, you finish with a single binary, and the compiler catches a lot of the silly mistakes before you ever hit "run".

In this article, we'll build a small Rust app that talks to the Venice API through [Rig](https://github.com/0xPlaygrounds/rig), a Rust library for LLM-powered applications. By the end of this tutorial, you'll have a working agent you can prompt from the command line, give it Rust functions to call as tools, stream tokens as they arrive, and harden the HTTP client with retries for production use.

## Pre-requisites

- Rust 1.75+ (install via [rustup](https://rustup.rs/) if you don't have it).
- A Venice API key. You can generate one from the [Venice dashboard](https://venice.ai/settings/api). Export it as an environment variable before running anything:

```bash
export VENICE_API_KEY=<my-key>
```

## Why Rust is worth your time for AI apps

Before we get building, it's worth a quick word on why Rust is a good fit here. You don't need to "rewrite everything in Rust" to get value out of it; for LLM tooling specifically, a few things line up well.

The compiler does a lot of the boring review work for you. Streaming chunks, tool-call payloads, and structured outputs all have a shape, and Rust will refuse to compile until you handle them. You spend less time debugging "what was in that JSON again?" and more time shipping.

You also get small, predictable binaries. A Rust app calling Venice compiles down to a single file you can drop on a server or into a container without dragging a language runtime along with it. For CLIs, sidecars, or gateways that sit in front of Venice, that's a real operational win.

Async I/O is the third big one. Rust's `async/await` with [Tokio](https://tokio.rs) is built for network-bound work: calling Venice, fanning out to tools, streaming tokens back to a client. You stay close to the metal when you need to, without giving up readable async code.

Finally, Rust is a great language for AI agents to be building on. The syntax that gets generated by compile-time errors are significantly helpful for humans, because they point at exactly where the error is, what it is and you don't need to follow a 100-line call stack to figure out where your problem is. This means that it's very easy for an agent to follow up on the error.

## Project overview

We'll build the project up in small steps:

| Step | What we'll do |
| --- | --- |
| 1. Set up the client | Point Rig at Venice's OpenAI-compatible endpoint with our API key |
| 2. Build the agent | Wrap the model in a Rig `Agent` so we can prompt it with one line |
| 3. Add Venice features | Pass Venice-only parameters (like web search) through the agent |
| 4. Give the agent tools | Define a Rust function the model can call when it needs to |
| 5. Stream responses | Print tokens as they arrive so we never block on a slow reply |
| 6. Harden the HTTP client | Swap in a custom HTTP client with retries for production use |

One detail worth flagging up front: Rig's default `openai::Client` targets OpenAI's newer Responses API. Venice's compatibility surface is the classic Chat Completions API ([OpenAI migration guide](/guides/getting-started/openai-migration)), so we'll use Rig's `CompletionsClient` instead. That single choice is what makes everything else line up.

## Getting started

Create a new Rust project and add the dependencies we'll need. `rig-core` is the library itself, `tokio` is the async runtime, and `serde_json` lets us build the Venice-specific JSON payloads later:

```bash
cargo new venice-rig-demo
cd venice-rig-demo
cargo add rig-core serde_json
cargo add tokio --features macros,rt-multi-thread
```

Check [crates.io/crates/rig-core](https://crates.io/crates/rig-core) for the latest version of `rig-core` if you want to pin to a specific release.

## Setting up the Venice client

Now that we've got the project scaffolded, we can wire up the client. We need three things: the API key, Venice's OpenAI-compatible base URL, and the right Rig client type.

Open `src/main.rs` and replace its contents with:

```rust
use rig_core::{
    client::CompletionClient,
    completion::Prompt,
    providers::openai::CompletionsClient,
};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let client = CompletionsClient::builder()
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    Ok(())
}
```

A quick tour of what's happening here:

- `CompletionClient` is the trait that gives our client the `.agent()` method we'll use next, and `Prompt` is the trait that gives the agent its `.prompt()` method. Bringing them into scope makes those methods available.
- `CompletionsClient::builder()` is the Chat Completions client we mentioned earlier. We pass it our key, point it at Venice's base URL (`https://api.venice.ai/api/v1`), and call `.build()`.
- `#[tokio::main]` turns our `main` into an async function so we can `.await` HTTP calls.

If you run `cargo run` now, the program will compile and exit successfully without doing anything visible. That's fine; we just confirmed the client builds.

## Building the agent

Next, we'll wrap the client in an `Agent` and send our first prompt. An agent in Rig is just a model plus a bit of configuration; it's the easiest way to get from "I have a client" to "I have a response on stdout".

Update `main` so it looks like this:

```rust
use rig_core::{
    client::CompletionClient,
    completion::Prompt,
    providers::openai::CompletionsClient,
};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let client = CompletionsClient::builder()
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    let agent = client.agent("zai-org-glm-5").build();

    let response = agent.prompt("Say hello in one short sentence.").await?;
    println!("{response}");

    Ok(())
}
```

All we added were two lines: one that builds the agent around the `zai-org-glm-5` model, and one that sends a prompt and prints the reply. The model name is a plain string because Venice's model IDs aren't the same as OpenAI's; you can swap in any text model from the [Venice model catalog](/models/text).

## Running the project

With the agent in place, we can run it:

```bash
cargo run
```

The first build will take a minute while `rig-core` and its dependencies compile. After that, you should see a one-line reply from the model, something like:

```text
Hello! How can I help you today?
```

If you get a 401, double-check that `VENICE_API_KEY` is exported in the same shell. If you get an "unknown model" error, look up a current model ID from the [text models](/models/text) page; IDs do change over time.

## Adding Venice-specific features

Now that we've done the hard part, we can layer Venice features on top. Venice supports extra fields on chat requests through a `venice_parameters` object; one of the most useful is `enable_web_search`, which lets the model pull in live context before answering.

Rig's OpenAI provider flattens any extra JSON we hand it into the outgoing request body, so all we need to do is attach a `venice_parameters` payload to the agent via `additional_params`.

Update the agent block:

```rust
use rig_core::{
    client::CompletionClient,
    completion::Prompt,
    providers::openai::CompletionsClient,
};
use serde_json::json;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let client = CompletionsClient::builder()
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    let agent = client
        .agent("zai-org-glm-5")
        .additional_params(json!({
            "venice_parameters": {
                "enable_web_search": "auto"
            }
        }))
        .build();

    let response = agent
        .prompt("Summarize one credible news headline from today.")
        .await?;
    println!("{response}");

    Ok(())
}
```

When you run this, the model will reach out for fresh context before answering, so you should see a response that reflects something timely rather than a generic placeholder.

Not every model supports every Venice feature. If a parameter is ignored or you get an error, cross-reference your model against the [text models](/models/text) catalog or check the [model feature suffixes](/api-reference/endpoint/chat/model_feature_suffix) guide for the per-model details.

## Giving the agent tools

A prompt-in, text-out agent is a fine start, but the real power of an LLM library shows up when the model can call your code. Rig's tool API lets us define a Rust function with a typed signature, hand it to the agent, and let the model decide when to invoke it.

We'll build a tiny `add` tool the model can call to do arithmetic for us. Before we touch the code, switch to a tool-calling model — Venice lists `zai-org-glm-5-1` among its [function-calling models](/guides/getting-started/openai-migration#feature-compatibility), so we'll use that.

Add two more crates:

```bash
cargo add serde --features derive
cargo add thiserror
```

Then replace `main.rs` with:

```rust
use rig_core::{
    client::CompletionClient,
    completion::{Prompt, ToolDefinition},
    providers::openai::CompletionsClient,
    tool::Tool,
};
use serde::Deserialize;
use serde_json::json;

#[derive(Deserialize)]
struct AddArgs {
    x: i32,
    y: i32,
}

#[derive(Debug, thiserror::Error)]
#[error("math error")]
struct MathError;

struct Adder;

impl Tool for Adder {
    const NAME: &'static str = "add";

    type Error = MathError;
    type Args = AddArgs;
    type Output = i32;

    async fn definition(&self, _prompt: String) -> ToolDefinition {
        ToolDefinition {
            name: Self::NAME.to_string(),
            description: "Add two integers together".to_string(),
            parameters: json!({
                "type": "object",
                "properties": {
                    "x": { "type": "integer", "description": "First number" },
                    "y": { "type": "integer", "description": "Second number" }
                },
                "required": ["x", "y"]
            }),
        }
    }

    async fn call(&self, args: Self::Args) -> Result<Self::Output, Self::Error> {
        Ok(args.x + args.y)
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let client = CompletionsClient::builder()
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    let agent = client
        .agent("zai-org-glm-5-1")
        .preamble("You are a helpful assistant. Use the `add` tool whenever the user asks for arithmetic.")
        .tool(Adder)
        .build();

    let response = agent.prompt("What is 137 + 256?").await?;
    println!("{response}");

    Ok(())
}
```

There's a bit going on here, but it breaks down cleanly:

- `AddArgs` is the input the model will fill in. Rig deserializes the JSON the model produces into this struct, so you get typed arguments instead of stringly-typed JSON.
- The `Tool` impl is the contract Rig needs: a unique `NAME`, the `definition()` describing the tool to the model in JSON Schema form, and `call()` which is the actual Rust code that runs when the model invokes it.
- `.tool(Adder)` attaches the tool to the agent, and `.preamble(...)` nudges the model toward using it.

Run it:

```bash
cargo run
```

You should see a reply that includes `393`. Under the hood, the model produced a tool call, Rig executed `Adder::call`, fed the result back into the conversation, and the model used that result to write the final answer.

From here, the same pattern works for anything you can write in Rust: hit a database, call your own API, look up data in a vector store, or run a shell command. Just remember to keep `call()` doing the *minimum* the model asked for — let the model handle the orchestration.

## Streaming the response

So far we've been calling `.prompt(...).await`, which waits for the whole response before returning. That's fine for short replies, but it has two real problems as your usage grows: long replies can run into HTTP timeouts (yours, or whatever proxy sits between you and Venice), and the user is staring at nothing while the model is busy generating.

Streaming fixes both. Rather than buffering the entire response on the server, Venice sends it back as a sequence of SSE chunks and Rig hands them to us as they arrive. The connection stays active the whole time, and you can start showing tokens immediately. Venice supports OpenAI-style SSE streaming as a drop-in ([feature compatibility](/guides/getting-started/openai-migration#feature-compatibility)), so on our side it's mostly a matter of using a different method on the agent.

Add the `futures` crate:

```bash
cargo add futures
```

Then swap `main.rs` to use `stream_prompt` instead of `prompt`. Rig ships a `stream_to_stdout` helper that handles the chunk loop for us:

```rust
use rig_core::{
    agent::stream_to_stdout,
    client::CompletionClient,
    providers::openai::CompletionsClient,
    streaming::StreamingPrompt,
};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let client = CompletionsClient::builder()
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    let agent = client.agent("zai-org-glm-5").build();

    let mut stream = agent
        .stream_prompt("Write a short haiku about Rust.")
        .await;

    let _final = stream_to_stdout(&mut stream).await?;
    println!();

    Ok(())
}
```

A few things changed:

- We brought the `StreamingPrompt` trait into scope so the agent picks up the `.stream_prompt()` method.
- `stream_prompt` returns a `StreamingPromptRequest` immediately; awaiting it gives us the live `StreamingResult` we can iterate over.
- `stream_to_stdout` consumes the stream and prints each text chunk as it arrives, returning a `FinalResponse` with token usage and the full message at the end.

Run it:

```bash
cargo run
```

You should see the haiku appear word by word in your terminal instead of in one go. That same chunk-by-chunk delivery is what makes streaming such a strong fit for production: if the model takes 40 seconds to finish, the connection is sending data the whole time rather than sitting idle waiting to time out.

If you're plugging this into a web server (or anywhere `stdout` isn't the destination), you'll want to handle chunks yourself instead of using `stream_to_stdout`. The pattern looks like this:

```rust
use futures::StreamExt;
use rig_core::{
    agent::MultiTurnStreamItem,
    streaming::{StreamedAssistantContent, StreamingPrompt},
};

let mut stream = agent.stream_prompt("Write a short haiku about Rust.").await;

while let Some(event) = stream.next().await {
    match event? {
        MultiTurnStreamItem::StreamAssistantItem(StreamedAssistantContent::Text(text)) => {
            // Forward `text.text` to your client, websocket, channel, etc.
            print!("{}", text.text);
        }
        MultiTurnStreamItem::FinalResponse(_final_res) => {
            // Use this to record token usage or finalize state.
        }
        _ => {}
    }
}
```

The `MultiTurnStreamItem` enum covers everything you might see during a turn — text chunks, reasoning chunks (for reasoning-capable models), tool calls, tool results, and the final response. For a basic chat UI, matching on the `Text` variant is enough; if you're using a reasoning model like `qwen3-4b`, add a branch for `StreamedAssistantContent::Reasoning` so you can render it separately.

## Adding retries with a custom HTTP client

By default, Rig builds its own `reqwest::Client` under the hood. That's fine for local development, but for anything that runs unattended you'll want to handle the inevitable transient errors — momentary 5xx responses, dropped TCP connections, rate-limit blips. Rig lets us supply our own HTTP client, so we can layer retries on top using [`reqwest-middleware`](https://crates.io/crates/reqwest-middleware) and [`reqwest-retry`](https://crates.io/crates/reqwest-retry).

First, enable Rig's `reqwest-middleware` feature and add the middleware crates:

```bash
cargo add rig-core --features reqwest-middleware
cargo add reqwest reqwest-middleware reqwest-retry
```

The first command updates the existing `rig-core` entry to turn on the `reqwest-middleware` feature, and the second pulls in the middleware ecosystem.

Then build a `ClientWithMiddleware` with an exponential-backoff retry policy and hand it to Rig via `.http_client()`:

```rust
use reqwest_middleware::ClientBuilder as MiddlewareBuilder;
use reqwest_retry::{policies::ExponentialBackoff, RetryTransientMiddleware};
use rig_core::{
    client::CompletionClient,
    completion::Prompt,
    providers::openai::CompletionsClient,
};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    let api_key = std::env::var("VENICE_API_KEY")?;

    let retry_policy = ExponentialBackoff::builder().build_with_max_retries(3);

    let http_client = MiddlewareBuilder::new(reqwest::Client::new())
        .with(RetryTransientMiddleware::new_with_policy(retry_policy))
        .build();

    let client = CompletionsClient::builder()
        .http_client(http_client)
        .api_key(api_key)
        .base_url("https://api.venice.ai/api/v1")
        .build()?;

    let agent = client.agent("zai-org-glm-5").build();
    let response = agent.prompt("Say hello in one short sentence.").await?;
    println!("{response}");

    Ok(())
}
```

A couple of things worth pointing out:

- `RetryTransientMiddleware` only retries on responses that look retryable: connection errors, 5xx responses, and 429s. It will *not* retry on a 401 or a 400, which is what you want — those are mistakes you should fix, not paper over.
- The exponential backoff (300 ms, then doubling) keeps you from hammering Venice during an outage and plays nicely with their [rate limits](/api-reference/rate-limiting).
- The same `http_client` builder accepts any middleware in the ecosystem, so you can stack logging, tracing, or auth refresh middleware here too.

For long-running workers or services that sit in front of Venice, this is the small change that separates "works on my machine" from "stays up overnight".

## Extending this example

Looking to take this further? Here are a couple of ways you can do that:

- **More interesting tools.** Swap the `Adder` for something that does real work — hit your own API, query a database, or wrap a CLI. The `Tool` trait scales up the same way regardless of what's behind it.
- **Reasoning models.** Try `qwen3-4b` or another [reasoning model](/guides/features/reasoning-models) and handle the `StreamedAssistantContent::Reasoning` branch separately so you can render the model's thinking in a different UI panel.
- **Embeddings and RAG.** `CompletionsClient` also exposes embeddings via the same OpenAI-compatible surface. Pair it with a vector store and you've got the bones of a private RAG bot in Rust.
- **Web service.** Wrap this in an [Axum](https://github.com/tokio-rs/axum) handler that streams chunks back to the browser over SSE. The retry middleware keeps the upstream connection resilient while the streaming response keeps your own response from timing out.
