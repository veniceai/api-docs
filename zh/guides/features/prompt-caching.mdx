提示缓存会存储已处理过的输入 token，使后续具有相同前缀的请求可以复用它们，而无需重新处理。这可以减少延迟（长提示最多可减少 80%）并降低成本（缓存 token 最多可享 90% 折扣）。

Venice 会为受支持的模型自动处理缓存，但了解每家提供商的缓存实现方式有助于你最大化缓存命中率并尽可能降低成本。

## 缓存的工作方式

缓存基于**前缀匹配**：系统存储已处理的 token，并在后续请求以相同内容开头时复用它们。

考虑一个具有 2,000 token 系统提示的聊天机器人：

<Steps>
  <Step title="请求 1">
    系统提示（2,000 token）+ 用户消息（50 token）

    **已处理**：2,050 token · **来自缓存**：0 token

    前缀写入缓存。
  </Step>
  <Step title="请求 2">
    系统提示（2,000 token）+ 用户消息（80 token）

    **已处理**：80 token · **来自缓存**：2,000 token
  </Step>
  <Step title="请求 3">
    系统提示（2,000 token）+ 用户消息（120 token）

    **已处理**：120 token · **来自缓存**：2,000 token
  </Step>
</Steps>

**未启用缓存的总量**：2,050 + 2,080 + 2,120 = 6,250 token，全部按全价计费

**启用缓存的总量**：2,050 + 80 + 120 = 2,250 token 按全价，4,000 token 按折扣价

<Warning>
缓存仅作用于**前缀**。提示开头的任何更改都会让其后的全部内容缓存失效。请始终把静态内容（系统提示、文档、示例）放在动态内容（用户消息）之前。
</Warning>

## 各提供商的特定行为

Venice 在不同提供商之间对缓存进行了归一化。对于大多数模型，缓存是自动的。你只需正常发送请求并在响应中查看缓存统计信息。**Claude** 需要在协议层显式标记缓存断点，但 Venice 会为系统提示和对话历史自动添加这些标记。

缓存行为最终由各提供商控制，可能会变化，请查阅各提供商文档获取最新细节。

| 模型 | 提供商 | 最小 token | 缓存生命周期 | 写入成本 | 读取折扣 | 显式标记 |
|-------|----------|------------|----------------|------------|---------------|------------------|
| Claude Opus 4.5 | Anthropic | ~4,000 | 5 分钟 | +25% | 90% | 必需 |
| GPT-5.2 | OpenAI | 1,024 | 5-10 分钟 | 无 | 90% | 无需 |
| Gemini | Google | ~1,024 | 1 小时 | 无 | 75-90% | 无需 |
| Grok | xAI | ~1,024 | 5 分钟 | 无 | 75-88% | 无需 |
| DeepSeek | DeepSeek | ~1,024 | 5 分钟 | 无 | 50% | 无需 |
| MiniMax | MiniMax | ~1,024 | 5 分钟 | 无 | 90% | 无需 |
| Kimi | Moonshot | ~1,024 | 5 分钟 | 无 | 50% | 无需 |

### Claude Opus 4.5（Anthropic）

Claude 需要在协议层显式声明缓存断点。Venice 会自动处理：

- **系统提示**会自动缓存
- **对话历史**通过在倒数第二条用户消息上放置断点来缓存

这意味着对话历史会从缓存中读取，只有最新一轮被作为新的输入处理：

| 轮次 | 提示 token | 缓存读 | 缓存写 | 节省 |
|------|---------------|------------|-------------|---------|
| 1 | 10,979 | 0 | 10,938 | 首次写入 |
| 2 | 11,031 | 10,938 | 31 | 99.7% 命中 |
| 3 | 11,062 | 10,969 | 52 | 99.5% 命中 |

**其他细节：**

- **每个请求最多 4 个断点**：系统会使用最长匹配前缀
- **缓存键按字节精确匹配**：空白字符变化、不同的图像编码或重新排序的工具都会破坏缓存命中
- **缓存感知的速率限制**：缓存 token 不计入 ITPM 限制，实际吞吐更高
- **25% 写入溢价**：首次请求成本更高，但后续读取可节省 90%

#### 手动缓存控制

对于特殊场景（例如在第一轮就缓存一份大文档），你可以添加显式断点：

```json
{
  "messages": [
    {
      "role": "system",
      "content": [{
        "type": "text",
        "text": "You are a legal assistant...",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    {
      "role": "user", 
      "content": [{
        "type": "text",
        "text": "[Long contract document...]",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    { "role": "assistant", "content": "I've reviewed the contract." },
    { "role": "user", "content": "What are the termination clauses?" }
  ]
}
```

这样可以确保系统提示与文档从第一次请求就被缓存。对于典型对话，你不需要手动标记。

### 所有其他模型

缓存是**自动**的。无需特殊参数。只需确保提示超过约 1,024 token，并使用 `prompt_cache_key` 实现一致路由。

## 请求参数

| 参数 | 类型 | 模型 | 说明 |
|-----------|------|--------|-------------|
| `prompt_cache_key` | string | 全部 | 用于缓存亲和性的路由提示。具有相同 key 的请求更可能命中同一台带有热缓存的服务器。 |
| `cache_control` | object | Claude | 标记内容块以进行缓存。参见 Claude Opus 4.5 章节。 |

### prompt_cache_key

对于会话或代理型工作流，使用一致的 `prompt_cache_key` 来提高缓存命中率：

```json
{
  "model": "claude-opus-4-5",
  "prompt_cache_key": "session-abc-123",
  "messages": [...]
}
```

这会将请求路由到可能已缓存你上下文的服务器。可使用 session ID、会话 ID 或用户 ID 作为 key。

## 响应字段

响应的 `usage` 对象包含缓存统计：

```json
{
  "usage": {
    "prompt_tokens": 5500,
    "completion_tokens": 200,
    "total_tokens": 5700,
    "prompt_tokens_details": {
      "cached_tokens": 5000,
      "cache_creation_input_tokens": 0
    }
  }
}
```

| 字段 | 说明 |
|-------|-------------|
| `prompt_tokens` | 请求中的输入 token 总数 |
| `prompt_tokens_details.cached_tokens` | 来自缓存的 token（按折扣价计费） |
| `prompt_tokens_details.cache_creation_input_tokens` | 写入缓存的 token（在 Claude 上可能产生溢价） |

**账单分解**（以 Claude Opus 4.5 为例）：
- 5000 个缓存 token × \$0.60/100 万 = \$0.003
- 500 个未缓存 token × \$6.00/100 万 = \$0.003
- 总计：\$0.006（相对未启用缓存时的 \$0.033，节省 82%）

## 最佳实践

### 为缓存而组织提示

将静态内容放在开头，动态内容放在末尾。

**良好结构**

| 位置 | 内容 | 是否缓存？ |
|----------|---------|---------|
| 1 | 系统指令 | 是 |
| 2 | 参考文档 | 是 |
| 3 | Few-shot 示例 | 是 |
| 4 | 用户查询 | 否 |

**糟糕结构**

| 位置 | 内容 | 是否缓存？ |
|----------|---------|---------|
| 1 | 当前时间戳 | 否（导致后续所有内容失效） |
| 2 | 系统指令 | 否 |
| 3 | 用户查询 | 否 |

### 保持前缀按字节一致

缓存键由精确的字节序列计算得出。即便是微小差异也会破坏缓存命中：

- 不同的空白或换行
- 提示中的时间戳或请求 ID
- 随机化的 few-shot 示例顺序
- 同一内容的不同格式

### 满足最小 token 阈值

如果提示低于最小值（通常为 1,024 token），缓存不会启用。对于较小的提示，可以考虑：

- 增加更多上下文或示例以达到阈值
- 将多个小请求合并为批量提示
- 接受简单查询无法命中缓存的事实

### 在对话中使用 prompt_cache_key

对于持续的对话，请设置一致的 `prompt_cache_key`：

```json
// 第 1 轮
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// 第 2 轮
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// 第 3 轮
{ "prompt_cache_key": "conv-xyz", "messages": [...] }
```

这能提高所有轮次命中同一台带热缓存服务器的概率。

### 监控缓存表现

跟踪以下指标：
- **缓存命中率**：`cached_tokens / prompt_tokens`
- **节省成本**：比较实际成本与未缓存时的成本
- **延迟降低**：有/无缓存命中时的首 token 时间

如果 `cached_tokens` 始终为 0：
1. 提示可能低于最小 token 阈值
2. 提示可能在请求间发生了变化
3. 请求可能命中了不同服务器（使用 `prompt_cache_key`）
4. 缓存可能已过期（请求过于稀疏）

### 考虑缓存经济性

**Claude Opus 4.5 缓存写入溢价**：首次请求成本高 25%，但后续读取可节省 90%。

| 场景 | 缓存写入溢价是否值得？ |
|----------|-------------------------------|
| 该提示仅 1 次请求 | 否（多付 25%，没有收益） |
| 同前缀的 2 次以上请求 | 是（第 2 次请求时回本） |
| 提示快速变化 | 否（持续的写入成本） |
| 稳定的系统提示、大量查询 | 是（在大量读取上摊销） |

## 缓存生命周期

缓存在一段不活跃时间后过期（通常 5-10 分钟）。这意味着：

| 流量模式 | 缓存收益 |
|-----------------|-----------------|
| 持续请求（间隔 < 5 分钟） | 高：缓存保持热 |
| 突发流量（间隔 > 10 分钟） | 有限：缓存在突发之间过期 |
| 偶发请求（相隔数小时） | 无：缓存始终冷 |

## 工具与函数的缓存

函数定义可以与系统提示一起被缓存：

```json
{
  "model": "claude-opus-4-5",
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_database",
        "description": "Search the product database",
        "parameters": { ... }
      }
    }
  ],
  "messages": [
    {
      "role": "system",
      "content": [
        {
          "type": "text",
          "text": "You are a shopping assistant...",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    ...
  ]
}
```

工具定义会成为被缓存前缀的一部分。如果你有很多工具，这可以显著降低每个请求的成本。

## 图像与文档的缓存

对于视觉模型，图像可以包含在被缓存的内容中：

```json
{
  "model": "claude-opus-4-5",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": { "url": "data:image/png;base64,..." }
        },
        {
          "type": "text",
          "text": "This is the floor plan. I'll ask several questions about it.",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    {
      "role": "assistant",
      "content": "I can see the floor plan. What would you like to know?"
    },
    {
      "role": "user",
      "content": "How many bedrooms are there?"
    }
  ]
}
```

图像与初始上下文会被缓存，因此针对同一图像的后续问题不会再次重新处理它。

## 故障排查

<Accordion title="cached_tokens 始终为 0">

| 原因 | 解决方案 |
|-------|----------|
| 提示过短 | 确保提示超过约 1,024 token（Claude 为 4,000） |
| 前缀已更改 | 检查提示开头是否存在动态内容 |
| 首次请求 | 符合预期：首次请求写入缓存，后续请求读取 |
| 缓存已过期 | 将请求间隔缩短到 5 分钟以内 |
| 不同服务器 | 添加 `prompt_cache_key` 以实现一致路由 |

</Accordion>

<Accordion title="每次请求都有 cache_creation_input_tokens">

| 原因 | 解决方案 |
|-------|----------|
| 提示发生变化 | 从前缀中移除时间戳、请求 ID 或其他动态内容 |
| 缺少 cache_control | 对 Claude，确保内容块上存在 `cache_control` 标记 |
| 低于阈值 | 低于最小 token 数的提示不会触发缓存 |
| 单条用户消息 | 第一轮预期如此。缓存会随对话历史增长。 |

</Accordion>

<Accordion title="成本高于预期">

| 原因 | 解决方案 |
|-------|----------|
| 缓存写入溢价 | Claude 写入贵 25%。只有在复用提示时才划算。 |
| 复用率低 | 如果每条提示都唯一，会付写入成本却没有读取收益 |
| 提示结构不合理 | 将动态内容移到末尾，让前缀保持稳定 |

</Accordion>
