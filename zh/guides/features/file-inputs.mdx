文件输入让你可以将文档与源文件直接附加到 `/chat/completions` 请求中。Venice 会先将文件提取为文本，再发送给所选模型，因此你无需先构建自己的解析器，即可对文件内容进行提问、摘要、比较或转换。

当你的提示依赖文档、表格、markdown 文件、JSON 文件或代码文件的内容时，请使用文件输入。它们是请求级别的输入，并非持久化的文件存储，因此每个需要该文件的请求都需要包含该文件。

<Info>
文件输入使用 OpenAI 兼容的聊天 content 数组。增加一个 `type: "file"` 的内容块，并在 `file.file_data` 中提供文件内容。
</Info>

## 支持的文件类型

聊天 API 接受以 base64 data URL 或可公开访问的 URL 形式提供的文件输入。

每个文件的最大大小为 **25MB**，按 base64 解码后或抓取 URL 后的体积计算。

| 类别 | 格式 |
|----------|---------|
| 文档 | PDF、DOCX、PPTX |
| 表格 | XLSX、XLS、CSV |
| 文本与数据 | TXT、Markdown、JSON |
| 源代码 | 大多数常见代码文件，包括 `.py`、`.js`、`.ts`、`.c`、`.cpp`、`.java`、`.go`、`.rs`、`.ps1`、`.sh`、`.yaml` 和 `.sql` |

<Note>
文件在推理前会被提取为文本。提取出的文本会计入模型的输入上下文，请选择 `availableContextTokens` 足够容纳文件加上指令与预期回答的模型。
</Note>

## 基本用法

发送一个 `messages` 数组，其中用户消息的 `content` 是文本和文件块组成的数组：

<CodeGroup>
```python Python
import base64
import os
from pathlib import Path

from openai import OpenAI

client = OpenAI(
    api_key=os.environ["VENICE_API_KEY"],
    base_url="https://api.venice.ai/api/v1",
)

path = Path("q3-report.pdf")
file_data = "data:application/pdf;base64," + base64.b64encode(path.read_bytes()).decode("utf-8")

response = client.chat.completions.create(
    model="openai-gpt-55",
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": "Summarize this report in five bullets and list the main risks.",
                },
                {
                    "type": "file",
                    "file": {
                        "file_data": file_data,
                        "filename": "q3-report.pdf",
                    },
                },
            ],
        }
    ],
)

print(response.choices[0].message.content)
```

```javascript Node.js
import OpenAI from "openai";
import { readFile } from "node:fs/promises";

const client = new OpenAI({
  apiKey: process.env.VENICE_API_KEY,
  baseURL: "https://api.venice.ai/api/v1",
});

const pdf = await readFile("q3-report.pdf");
const fileData = `data:application/pdf;base64,${pdf.toString("base64")}`;

const response = await client.chat.completions.create({
  model: "openai-gpt-55",
  messages: [
    {
      role: "user",
      content: [
        {
          type: "text",
          text: "Summarize this report in five bullets and list the main risks.",
        },
        {
          type: "file",
          file: {
            file_data: fileData,
            filename: "q3-report.pdf",
          },
        },
      ],
    },
  ],
});

console.log(response.choices[0].message.content);
```

```bash cURL
          }

    }

```

## 文件 URL

如果文件已托管在可公开访问的 HTTP 或 HTTPS URL 上，可以直接将 URL 放入 `file_data`，无需进行 base64 编码：

```json
{
  "model": "openai-gpt-55",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "Identify the governing law, renewal terms, and termination rights in this agreement."
        },
        {
          "type": "file",
          "file": {
            "file_data": "https://example.com/contracts/vendor-agreement.pdf",
            "filename": "vendor-agreement.pdf"
          }
        }
      ]
    }
  ]
}
```

<Warning>
仅使用 Venice 可以在无认证条件下抓取的公开 URL。对于私有文件，请发送 base64 data URL。
</Warning>

## 多个文件

你可以在同一条消息中包含多个文件块。在文件之前先放一条简短的文本指令，让模型知道如何使用它们。

```json
{
  "model": "openai-gpt-55",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "Compare these two policy drafts. Return the material differences and recommend which version is clearer."
        },
        {
          "type": "file",
          "file": {
            "file_data": "data:application/pdf;base64,JVBERi0xLjQK...",
            "filename": "policy-v1.pdf"
          }
        },
        {
          "type": "file",
          "file": {
            "file_data": "data:application/pdf;base64,JVBERi0xLjQK...",
            "filename": "policy-v2.pdf"
          }
        }
      ]
    }
  ]
}
```

为获得最佳效果，请为每个文件清晰命名，并在你的提示中直接引用这些名称。

## Data URL

对于本地文件，将文件字节编码为 base64，并用正确的 MIME 类型作为前缀：

| 文件类型 | Data URL 前缀 |
|-----------|-----------------|
| PDF | `data:application/pdf;base64,` |
| DOCX | `data:application/vnd.openxmlformats-officedocument.wordprocessingml.document;base64,` |
| PPTX | `data:application/vnd.openxmlformats-officedocument.presentationml.presentation;base64,` |
| XLSX | `data:application/vnd.openxmlformats-officedocument.spreadsheetml.sheet;base64,` |
| CSV | `data:text/csv;base64,` |
| Markdown | `data:text/markdown;base64,` |
| 纯文本 | `data:text/plain;base64,` |
| JSON | `data:application/json;base64,` |

如果你不清楚确切的 MIME 类型，可以使用 `application/octet-stream`。即便如此，提供准确的 `filename` 仍有助于 Venice 识别和展示该文件。

## 处理大文件

由于文件会变成提示文本，大文件会增加延迟、token 使用量和成本。请留意模型的上下文窗口。

原始文件必须不大于 25MB。base64 编码会让请求体积增加约 33%，因此接近 25MB 上限的文件会产生更大的 JSON 请求体。

处理大文件的良好模式：

- 提出具体任务，而不是宽泛的"分析所有内容"式提示。
- 仅包含当前回答所需的文档。
- 对长报告或代码库使用具有更大 `availableContextTokens` 的模型。
- 如果你还在使用[提示缓存](/zh/guides/features/prompt-caching)，请将稳定、重复的文档放在动态用户问题之前。
- 当预期响应较长时，使用 `stream: true`。

## 文件输入 vs. 文本解析器

当你希望模型立即对文件进行推理时，使用聊天的文件输入。

当你希望先提取文本、检查 token 计数、将提取的文本存入自有系统，或将同一段提取文本发送给多个请求时，使用[文本解析器 API](/zh/api-reference/endpoint/augment/text-parser)。

| 需求 | 使用 |
|------|-----|
| 在一个请求中让模型针对文档作答 | 聊天文件输入 |
| 在不进行模型推理的情况下提取文本 | 文本解析器 API |
| 在提示前查看提取后的 token 计数 | 文本解析器 API |
| 在多个请求间复用提取文本 | 文本解析器 API，然后将文本嵌入提示 |

## 最佳实践

- 尽量提供 `filename`，尤其在发送多个文件时。
- 将指令放在文件块之前，让模型在读取提取内容前就知道任务。
- 公共 URL 只用于无需 cookie、请求头或签名会话状态即可访问的文件。
- 对于私有文件或应用内生成的文件，优先使用 base64 data URL。
- 提出聚焦的问题，并指定你希望的输出格式。
- 对于结构化抽取，可将文件输入与[结构化响应](/zh/guides/features/structured-responses)结合使用。

## 故障排查

<AccordionGroup>
  <Accordion title="模型称无法访问文件">
    确保消息 content 使用数组，并包含一个 `type: "file"` 块。如果使用了 URL，请确认该 URL 在无认证条件下可被访问。
  </Accordion>

  <Accordion title="请求缓慢或成本过高">
    文件可能被提取出大量文本。可使用更大上下文的模型、缩小任务范围、减少文件数量，或先通过文本解析器 API 进行预提取并裁剪文本。
  </Accordion>

  <Accordion title="响应忽略了我的某个文件">
    为每个文件赋予描述性的 `filename`，并在提示中直接引用这些文件名。例如："Compare `policy-v1.pdf` against `policy-v2.pdf`."
  </Accordion>

  <Accordion title="某个模型拒绝文件内容">
    文件输入在兼容的聊天模型上可用。请查看[模型页面](/zh/models/overview)了解当前模型的能力与上下文上限，或尝试一款当前的大上下文文本模型。
  </Accordion>
</AccordionGroup>
