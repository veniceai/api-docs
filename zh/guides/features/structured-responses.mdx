Venice 现已在 API 中通过 "response_format" 字段支持结构化输出。该字段使你可以生成遵循特定预定义格式的响应。通过这种新方式，模型在响应中编造错误的键或值的可能性更低，这在以往通过系统提示操控或函数调用实现时更为常见。

结构化输出的 "response_format" 字段采用 OpenAI API 的格式，更详细的说明见 OpenAI 的[指南](https://platform.openai.com/docs/guides/structured-outputs)。OpenAI 还专门发布了一篇关于在 API 中使用结构化输出的介绍文章，见[这里](https://openai.com/index/introducing-structured-outputs-in-the-api/)。由于这是高级功能，页面底部列出了一些需要注意的"陷阱"。

并非所有模型都原生支持该功能。请参阅 [模型部分](https://docs.venice.ai/api-reference/endpoint/models/list?playground=open)，查找 "supportsResponseSchema" 标识以确认适用的模型。

```json
    {
      "id": "venice-uncensored",
      "type": "text",
      "object": "model",
      "created": 1726869022,
      "owned_by": "venice.ai",
      "model_spec": {
        "availableContextTokens": 32768,
        "capabilities": {
          "supportsFunctionCalling": true,
          "supportsResponseSchema": true,
          "supportsWebSearch": true
        },
```

### 如何使用结构化响应

要正确使用 "response_format"，你可以定义包含多个 "properties" 的 schema，每个属性代表一类输出，并配置各自的数据类型。这些对象可以嵌套，以构建更高级的输出结构。

下面是一个使用 response_format 解释解方程逐步过程的 API 调用示例。

可以看到，properties 被配置为响应中必须同时包含 "steps" 与 "final_answer"。在嵌套中，steps 类目又由 "explanation" 和 "output" 组成，均为字符串。

```json
curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header 'Authorization: Bearer <api-key>' \
  --header 'Content-Type: application/json' \
  --data '{
  "model": "venice-uncensored",
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful math tutor."
    },
    {
      "role": "user",
      "content": "solve 8x + 31 = 2"
    }
  ],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "math_response",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "steps": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "explanation": {
                  "type": "string"
                },
                "output": {
                  "type": "string"
                }
              },
              "required": ["explanation", "output"],
              "additionalProperties": false
            }
          },
          "final_answer": {
            "type": "string"
          }
        },
        "required": ["steps", "final_answer"],
        "additionalProperties": false
      }
    }
  }
}

```

下面是模型返回的响应。可以看到结构严格遵循了要求：先按步骤给出 "steps"，每步包含 "explanation" 与 "output"，最后给出 "final answer"。

```json
{
  "steps": [
    {
      "explanation": "Subtract 31 from both sides to isolate the term with x.",
      "output": "8x + 31 - 31 = 2 - 31"
    },
    {
      "explanation": "This simplifies to 8x = -29.",
      "output": "8x = -29"
    },
    {
      "explanation": "Divide both sides by 8 to solve for x.",
      "output": "x = -29 / 8"
    }
  ],
  "final_answer": "x = -29 / 8"
}

```

虽然这是一个简单示例，但可以推广到更高级的用例，例如：数据提取、思维链练习、UI 生成、数据分类等。

### 陷阱

通过 response_format 使用结构化输出时，需要注意以下要点：

* 使用 response_format 的初次请求可能耗时较长。后续请求不会出现这种初次延迟。

* 对于较大的查询，如果达到 `max_tokens` 或模型超时、或违反任何速率限制，模型可能无法完成。

* 不正确的 schema 格式会在补全时报错，通常表现为超时。

* 虽然 response_format 能保证模型输出遵循特定方式，但并不保证模型在其中提供的内容正确。内容由提示和模型表现决定。

* 通过 response_format 实现的结构化输出与并行函数调用不兼容。

* 重要：所有字段或参数都必须包含 `required` 标签。如需将字段设为可选，需要在该字段的 `type` 中增加 `null` 选项，例如 `"type": ["string", "null"]`。

* 可以通过在 required 字段中提供 `null` 选项以允许空响应，从而实现字段可选。

* 重要：`additionalProperties` 必须设为 false，response_format 才能正常工作。

* 重要：`strict` 必须设为 true，response_format 才能正常工作。
