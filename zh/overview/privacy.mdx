Venice 的核心原则之一是用户隐私。平台的架构基于这一理念延伸而来，每个组件都围绕这一目标进行设计。

> 实现合理用户隐私的唯一方式，是一开始就避免收集这些信息。从工程角度而言，这更难做到，但我们相信这才是正确的方式。

Venice API 复用了与 Venice 平台相同的后端隐私架构：请求通过加密连接经由 Venice 代理传递；对于普通推理，Venice 不存储或记录提示与响应内容；所选模型在运行时层增加以下四种隐私模式之一：Anonymous、Private、TEE 或 E2EE。

## 隐私架构

Venice 代理是所有隐私模式的共同基础。请求通过 HTTPS/TLS 经由 Venice 中转，Venice 不存储提示或响应内容。所选模型的隐私模式决定了在提供商或模型运行时层之后会发生什么。

Venice 将模型的隐私呈现为四种模式。它们建立在相同的代理基础之上，逐级增强保护：从对提供商隐藏身份，到将提示端到端加密进入经过验证的飞地（enclave）。

`/models` 端点会告诉你每个模型的隐私级别。标记为 `anonymized` 的是 Anonymous 模型，标记为 `private` 的是 Private 模型。TEE 与 E2EE 在模型的能力字段中单独显示，例如 `supportsTeeAttestation` 和 `supportsE2EE`。

## TEE 与 E2EE

TEE 与 E2EE 模型在 Venice 默认的不留存内容方式之上，增加了基于加密和硬件的控制。

<CardGroup cols={2}>
  <Card title="何时使用 TEE" icon="shield-halved">
    你希望模型在经证明的硬件飞地内运行，但客户端仍可通过常规 API 请求以明文形式发送提示。
  </Card>
  <Card title="何时使用 E2EE" icon="lock">
    你希望提示在离开客户端之前就被加密，并且仅在经验证的 TEE 内被解密。
  </Card>
</CardGroup>

E2EE 流程在支持 E2EE 的模型上通过 `/chat/completions` 使用。你的客户端需要获取证明、验证 nonce 与飞地证据、加密 `user` 和 `system` 消息、发送 `X-Venice-TEE-*` 头、流式接收响应，并验证/解密响应内容。

E2EE 还会禁用那些需要在飞地之外存在明文的功能，例如网页搜索、记忆、摘要、部分工具流程以及其他服务端处理。

## 选择模型

发送请求前，使用 `/models` 查看每个模型支持的隐私保护。

每个模型有两个相关字段：

- `model_spec.privacy` 表示模型的基线隐私模式：
  - `anonymized`：Venice 对提供商隐藏你的身份，但提供商仍可能看到提示。
  - `private`：Venice 通过零数据留存基础设施路由请求。
- `model_spec.capabilities` 表示模型是否支持更强的保护：
  - `supportsTeeAttestation`：模型可在可验证的可信执行环境内运行。
  - `supportsE2EE`：模型可接受由客户端加密的提示，仅在 TEE 内部解密。

E2EE 是由客户端驱动的流程。你的应用必须加密请求、验证证明，并验证/解密响应。参见 [TEE 与 E2EE 模型指南](/zh/guides/features/tee-e2ee-models)。

<CodeGroup>
```bash cURL
curl https://api.venice.ai/api/v1/models \
  -H "Authorization: Bearer $API_KEY_VENICE" | \
  jq '.data[] | {
    id,
    privacy: .model_spec.privacy,
    tee: .model_spec.capabilities.supportsTeeAttestation,
    e2ee: .model_spec.capabilities.supportsE2EE
  }'
```
</CodeGroup>

一个简单的经验法则：需要零数据留存就选 `private`，需要硬件支持的隔离就选 `tee: true`，需要在提示离开客户端前就被加密就选 `e2ee: true`。

## 运营元数据

Venice 可能会处理用于认证、计费、防滥用、可靠性、分析与支持的元数据。根据你的使用方式，这些可能包括账户或钱包标识、API 密钥标识、请求时间戳、所选模型、token 数量、计费金额、速率限制状态、请求 ID、IP 地址、浏览器或设备信息以及产品事件日志。

这些元数据用于运行 API，与提示和响应内容相互独立。计费与使用记录跟踪模型、端点、token 数量、时间戳与账户标识等细节；它们不需要存储提示或补全内容。
