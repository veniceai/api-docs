Leiten Sie eine JSON-RPC-2.0-Anfrage (einzeln oder im Batch) an einen unterstützten Blockchain-Knoten weiter. Unterstützt sowohl API-Schlüssel- als auch x402-Wallet-Authentifizierung. Die Abrechnung erfolgt pro Credit und in Ihrem Venice-Guthaben — eine Credential, eine Rechnung, jede Chain unten.

## Authentifizierung

Dieser Endpoint unterstützt zwei Authentifizierungsmethoden:

- **API-Schlüssel**: Standard-Bearer-Token-Authentifizierung über den Header `Authorization: Bearer <key>`.
- **x402-Wallet**: Pay-as-you-go mit USDC-Credits aus einer Wallet auf Base oder Solana. Kein Venice-Konto erforderlich. Zur Einrichtung siehe den [x402-Leitfaden](/de/guides/integrations/x402-venice-api).

Beide Methoden teilen sich dieselben Rate Limits und Abrechnung (Venice-Credits).

## Anfrageformen

### Einzelne Anfrage

```json
{
  "jsonrpc": "2.0",
  "method": "eth_chainId",
  "params": [],
  "id": 1
}
```

### Batch-Anfrage

Ein Array aus bis zu **100** JSON-RPC-2.0-Objekten. Jedes Element wird unabhängig validiert; ist eine Methode nicht unterstützt, wird der gesamte Batch mit `400` zurückgewiesen, und jeder unzulässige Methodenname wird in der Fehlermeldung aufgelistet.

```json
[
  { "jsonrpc": "2.0", "method": "eth_chainId", "params": [], "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 2 }
]
```

## Unterstützte Methoden und Credit-Stufen

Methoden werden in drei Credit-Stufen klassifiziert. Verbrauchte Credits pro Aufruf = `baseCredits[chain] × methodTier`.

| Stufe | Multiplikator | Beispielmethoden |
|---|---|---|
| **Standard** | 1× | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas`, `net_version`, `web3_clientVersion`, ERC-4337-Bundler-Methoden (`eth_sendUserOperation`, `eth_estimateUserOperationGas` usw.), Chain-Family-Erweiterungen (`zks_*`, `linea_*`, `bor_*`, `starknet_*`) |
| **Advanced** | 2× | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction`, `debug_traceBlockByHash`, `txpool_inspect`, `txpool_status`, `arbtrace_*` |
| **Large** | 4× | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content`, `arbtrace_replayTransaction`, `arbtrace_replayBlockTransactions` |

### Basis-Credits pro Chain

| baseCredits | Chains |
|---|---|
| **20** | Ethereum + alle oben genannten EVM-L2s (Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast) und Starknet |
| **30** | zkSync Era |

### Kostenbeispiele

Bei Venice's `~6,25 × 10⁻⁷ $` pro Credit:

| Aufruf | Credits | USD-Kosten |
|---|---|---|
| `eth_call` auf Ethereum (20 × 1×) | 20 | 0,0000125 $ |
| `trace_transaction` auf Ethereum (20 × 2×) | 40 | 0,0000250 $ |
| `trace_replayTransaction` auf Ethereum (20 × 4×) | 80 | 0,0000500 $ |
| `eth_call` auf zkSync (30 × 1×) | 30 | 0,0000188 $ |

### Nicht unterstützt

- **Nur-WebSocket-Methoden** (`eth_subscribe`, `eth_unsubscribe`) — dieser Proxy ist HTTP-only. Pollen Sie stattdessen oder wechseln Sie zu einem direkten WebSocket-Anbieter.
- **Zustandsbehaftete Filter-Methoden** (`eth_newFilter`, `eth_getFilterChanges`, `eth_getFilterLogs`, `eth_uninstallFilter`, `eth_newBlockFilter`, `eth_newPendingTransactionFilter`) — der Filterzustand ist an ein einzelnes Upstream-Backend gebunden und bricht still auf einem lastverteilten HTTP-Proxy. Verwenden Sie stattdessen `eth_getLogs` (zustandslos).
- **Miner-/Key-haltende Methoden** (`eth_sign`, `eth_accounts`, `eth_mining`, `eth_hashrate`, `eth_getWork`, `eth_submitWork`) — gehostete Anbieter-Endpoints halten keine privaten Nutzerschlüssel, daher schlagen diese stets fehl. Signieren Sie Transaktionen client-seitig und senden Sie sie über `eth_sendRawTransaction`.
- **Nicht zugeordnete Methoden** — alles, was nicht explizit in der Allowlist steht, gibt `400` zurück. Wenden Sie sich an den Support, um Ergänzungen anzufragen.

## Abrechnung pro Batch-Element

Auch wenn die HTTP-Antwort `200` ist, können einzelne Batch-Elemente mit einem JSON-RPC-Feld `error` zurückkommen (z. B. ein Bad-Params-Fehler oder eine auf der Ziel-Chain nicht unterstützte Methode). Venice rechnet diese Elemente mit **5 Credits pro Stück** statt der vollen Methodenstufe ab — ein kleines Zugeständnis für normale "API-erkunden"-Fehler.

```json
// Batch-Anfrage:
[
  { "jsonrpc": "2.0", "method": "eth_chainId",   "params": [],         "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_getBalance","params": ["bad"],    "id": 2 }
]

// Antwort (HTTP 200, X-Venice-RPC-Credits: 25):
[
  { "jsonrpc": "2.0", "id": 1, "result": "0x1" },
  { "jsonrpc": "2.0", "id": 2, "error": { "code": -32602, "message": "invalid params" } }
]
```

Das erste Element (Erfolg) rechnet mit 20 Credits ab, das zweite (RPC-Level-Fehler) mit 5, Summe `= 25`.

## Rate Limits

Limit pro Minute pro authentifiziertem Aufrufer:

| Tier | Requests/Minute |
|---|---|
| Standard | 100 |
| Staff | 1.000 |

Wird das Limit überschritten, gibt der Endpoint `429` mit einer `customMessage` und den Standard-`X-RateLimit-*`-Response-Headern zurück.

## Idempotenz

Setzen Sie den Request-Header `Idempotency-Key` auf einen beliebigen String, der dem Muster `[A-Za-z0-9_-]{1,255}` entspricht, um sichere Retries zu ermöglichen. Die Antwort wird 24 Stunden lang anhand `(user, idempotency-key)` zwischengespeichert:

- Wird **derselbe Schlüssel mit demselben Body** erneut gesendet, wird die zwischengespeicherte Antwort zusammen mit einem `Idempotent-Replayed: true`-Response-Header zurückgegeben. Der Upstream wird nicht angesprochen und es werden keine neuen Credits berechnet.
- Wird **derselbe Schlüssel mit einem anderen Body** gesendet, wird `400` zurückgegeben, um stille Zustandsbeschädigungen zu verhindern. Wählen Sie für unterschiedliche Anfragen einen neuen Schlüssel.

## Response-Header

| Header | Beschreibung |
|---|---|
| `X-Venice-RPC-Credits` | Für diese Anfrage berechnete Credits. Bei Batch-Anfragen ist dies die Summe über alle Elemente. |
| `X-Venice-RPC-Cost-USD` | Dollar-Kosten auf 8 Dezimalstellen. Entspricht `X-Venice-RPC-Credits × Preis pro Credit`. |
| `X-Request-ID` | 32-Zeichen-Korrelations-ID. Fügen Sie diese jeder Supportkorrespondenz bei. |
| `Idempotent-Replayed` | Vorhanden mit Wert `"true"`, wenn die Antwort aus dem Idempotenz-Cache stammt. |
| `X-RateLimit-Limit` / `X-RateLimit-Remaining` / `X-RateLimit-Reset` | Nur bei 429-Antworten gesetzt. |

## Forensisches Logging für Transaktions-Relays

Jeder Aufruf von `eth_sendRawTransaction` wird serverseitig mit dem Tx-Hash (keccak256 der Rohbytes), dem Netzwerk-Slug, der Request-ID und der aufrufenden User-ID protokolliert. Wir bewahren die signierte Nutzlast selbst **nicht** auf — der Hash lässt sich aus dem On-Chain-Receipt rekonstruieren. Dieses Audit-Trail existiert, damit wir bei einem kompromittierten API-Schlüssel, der zum Relay illegaler Transaktionen über unsere Infrastruktur missbraucht wird, On-Chain-Aktivitäten dem verantwortlichen Konto zuordnen können.

## Beispiel

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Response-Header: `X-Venice-RPC-Credits: 20`, `X-Venice-RPC-Cost-USD: 0.00001250`, `X-Request-ID: <nanoid>`.
