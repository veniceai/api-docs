Venice bietet datenschutzverbesserte Modelle, die in Trusted Execution Environments (TEE) laufen und Ende-zu-Ende-Verschlüsselung (E2EE) unterstützen. Diese Modelle geben kryptografische Garantien, dass Ihre Daten privat bleiben — selbst gegenüber Venice.

## Die Datenschutzstufen verstehen

| Typ | Präfix | Was es bedeutet |
|------|--------|---------------|
| **TEE** | `tee-*` | Modell läuft in einer hardware-gesicherten Enklave. Venice kann nicht auf die Berechnung zugreifen. Sie können das per Attestation überprüfen. |
| **E2EE** | `e2ee-*` | Vollständige Ende-zu-Ende-Verschlüsselung. Ihre Prompts werden client-seitig verschlüsselt, bevor sie gesendet werden. Nur die TEE kann sie entschlüsseln. |

<Info>
E2EE-Modelle enthalten TEE-Schutz plus client-seitige Verschlüsselung. TEE-Modelle bieten Enklaven-Sicherheit, ohne client-seitige Verschlüsselung zu erfordern.
</Info>

## TEE-Modelle

TEE-Modelle laufen in hardware-gesicherten Enklaven (Intel TDX, NVIDIA Confidential Computing). Die Modellgewichte und Ihre Daten sind vor dem Host-System — einschließlich der Venice-Infrastruktur — geschützt.

### TEE-Attestation verifizieren

Sie können kryptografisch verifizieren, dass ein Modell in einer echten TEE läuft, indem Sie seinen Attestation-Report abrufen:

<CodeGroup>
```bash cURL
# Zufällige Nonce erzeugen (verhindert Replay-Attacken)
NONCE=$(openssl rand -hex 16)

# Attestation abrufen
curl "https://api.venice.ai/api/v1/tee/attestation?model=tee-qwen3-5-122b-a10b&nonce=$NONCE" \
  -H "Authorization: Bearer $API_KEY_VENICE"
```

```python Python
import secrets
import requests

nonce = secrets.token_hex(16)

response = requests.get(
    f"https://api.venice.ai/api/v1/tee/attestation",
    params={"model": "tee-qwen3-5-122b-a10b", "nonce": nonce},
    headers={"Authorization": f"Bearer {api_key}"}
)

attestation = response.json()
print(f"Verified: {attestation['verified']}")
print(f"TEE Provider: {attestation['tee_provider']}")
print(f"Model: {attestation['model']}")
```
</CodeGroup>

Die Attestation-Antwort umfasst:

| Feld | Beschreibung |
|-------|-------------|
| `verified` | Ob die Attestation die serverseitige Prüfung bestanden hat |
| `nonce` | Ihre Nonce, bestätigt die Frische |
| `model` | Die attestierte Modell-ID |
| `tee_provider` | TEE-Provider-Kennung |
| `intel_quote` | Roher Intel-TDX-Quote (base64) zur client-seitigen Verifikation |
| `nvidia_payload` | NVIDIA-GPU-Attestation-Daten (falls anwendbar) |
| `signing_key` | Öffentlicher Schlüssel zum Verifizieren von Response-Signaturen (typischerweise erforderlich für E2EE-Flows; kann für einige Plain-TEE-Modelle entfallen) |
| `signing_address` | Aus dem Signing-Key abgeleitete Ethereum-Adresse |

<Tip>
Für den Produktivbetrieb verifizieren Sie die Attestation client-seitig, indem Sie den Intel-TDX-Quote parsen und die NVIDIA-Attestation prüfen.
</Tip>

<Note>
Für die Verifikation von Plain-TEE-Modellen reichen `signing_address` und serverseitige Verifikationsfelder für die grundlegende Attestation-Prüfung aus. Ein `signing_key` ist erforderlich, wenn Sie client-seitige E2EE-Schlüsselvereinbarung und strikte Key-Binding-Prüfungen benötigen.
</Note>

## E2EE-Modelle

E2EE-Modelle fügen zur TEE-Sicherheit eine client-seitige Verschlüsselung hinzu. Ihre Prompts werden verschlüsselt, bevor sie Ihr Gerät verlassen, und nur die TEE kann sie entschlüsseln.

Venice-E2EE verwendet:

- **ECDH (Elliptic Curve Diffie-Hellman)** auf secp256k1 für den Schlüsselaustausch
- **HKDF-SHA256** zur Schlüsselableitung
- **AES-256-GCM** für symmetrische Verschlüsselung
- **TEE-Attestation** zur Verifikation, dass das Modell in einer sicheren Enklave läuft

<Warning>
E2EE erfordert eine client-seitige Implementierung. Die folgenden Beispiele zeigen das vollständige Protokoll.
</Warning>

### So funktioniert E2EE

<Steps>
  <Step title="Ephemeres Schlüsselpaar erzeugen">
    Der Client erzeugt ein secp256k1-Schlüsselpaar für diese Session.
  </Step>
  <Step title="TEE-Attestation abrufen">
    Der Client fordert `/api/v1/tee/attestation` an und erhält den öffentlichen Schlüssel des Modells, Attestation-Belege und eine Nonce.
  </Step>
  <Step title="Attestation verifizieren">
    Der Client prüft Nonce-Übereinstimmung, deaktivierten Debug-Modus und Attestation-Gültigkeit.
  </Step>
  <Step title="Nachrichten verschlüsseln">
    Der Client verschlüsselt Prompts mittels ECDH-Shared-Secret → HKDF → AES-GCM.
  </Step>
  <Step title="Request senden">
    Der Client sendet die Anfrage mit E2EE-Headern (`X-Venice-TEE-Client-Pub-Key`, `X-Venice-TEE-Model-Pub-Key`, `X-Venice-TEE-Signing-Algo`).
  </Step>
  <Step title="TEE-Verarbeitung">
    Die TEE entschlüsselt den Request, verarbeitet ihn und verschlüsselt die Antwort.
  </Step>
  <Step title="Antwort entschlüsseln">
    Der Client empfängt verschlüsselte Chunks und entschlüsselt sie mit dem Private Key.
  </Step>
</Steps>

### Schritt 2: Ephemeres Schlüsselpaar erzeugen

Erzeugen Sie pro Session ein neues Schlüsselpaar. Der Private Key sollte ausschließlich im Speicher gehalten und nach der Nutzung sicher mit Nullen überschrieben werden.

<CodeGroup>
```javascript JavaScript
import { ec as EC } from 'elliptic'

function generateEphemeralKeyPair() {
  const ec = new EC('secp256k1')
  const keyPair = ec.genKeyPair()

  return {
    privateKey: new Uint8Array(keyPair.getPrivate().toArray('be', 32)),
    publicKeyHex: keyPair.getPublic('hex'), // Uncompressed-Format (65 Bytes Hex)
  }
}

// Sicherheit: Private Key nach Gebrauch mit Nullen überschreiben
function zeroFill(arr) {
  arr.fill(0)
}
```

### Schritt 3: TEE-Attestation abrufen und verifizieren

Die Attestation beweist, dass das Modell in einer echten TEE läuft. Verifizieren Sie die Attestation immer, bevor Sie dem öffentlichen Schlüssel des Modells vertrauen.

<Info>
**Wichtig: Nonce-Länge** — Die Client-Nonce muss **32 Bytes (64 Hex-Zeichen)** lang sein. Einige TEE-Anbieter verlangen exakt 32 Bytes und lehnen kürzere Nonces ab.
</Info>

<CodeGroup>
```javascript JavaScript
import crypto from 'crypto'

async function fetchAndVerifyAttestation(modelId, apiKey) {
  // Client-Nonce zum Schutz vor Replay erzeugen (32 Bytes = 64 Hex-Zeichen)
  const clientNonce = crypto.randomBytes(32).toString('hex')

  const response = await fetch(
    `https://api.venice.ai/api/v1/tee/attestation?model=${encodeURIComponent(modelId)}&nonce=${clientNonce}`,
    { headers: { Authorization: `Bearer ${apiKey}` } }
  )

  const attestation = await response.json()

  // Attestation verifizieren
  if (attestation.verified !== true) {
    throw new Error('TEE attestation verification failed on server')
  }

  if (attestation.nonce !== clientNonce) {
    throw new Error('Attestation nonce mismatch - possible replay attack')
  }

  // Öffentlichen Schlüssel des Modells für die Verschlüsselung erhalten
  const modelPublicKey = attestation.signing_key || attestation.signing_public_key
  if (!modelPublicKey) {
    throw new Error('No signing key in attestation response')
  }

  return {
    modelPublicKey,
    signingAddress: attestation.signing_address,
    attestation,
  }
}
```

```python Python
import secrets
import requests

def fetch_and_verify_attestation(model_id: str, api_key: str) -> dict:
    """Fetch and verify TEE attestation for a model."""
    # Client-Nonce zum Schutz vor Replay erzeugen (32 Bytes = 64 Hex-Zeichen)
    client_nonce = secrets.token_hex(32)

    response = requests.get(
        f'https://api.venice.ai/api/v1/tee/attestation',
        params={'model': model_id, 'nonce': client_nonce},
        headers={'Authorization': f'Bearer {api_key}'}
    )
    attestation = response.json()

    # Attestation verifizieren
    if attestation.get('verified') != True:
        raise ValueError('TEE attestation verification failed on server')

    if attestation.get('nonce') != client_nonce:
        raise ValueError('Attestation nonce mismatch - possible replay attack')

    # Öffentlichen Schlüssel des Modells für die Verschlüsselung erhalten
    model_public_key = attestation.get('signing_key') or attestation.get('signing_public_key')
    if not model_public_key:
        raise ValueError('No signing key in attestation response')

    return {
        'model_public_key': model_public_key,
        'signing_address': attestation.get('signing_address'),
        'attestation': attestation
    }
```

### Schritt 4: Nachrichten verschlüsseln

Verschlüsseln Sie User- und System-Nachrichten vor dem Senden. Nur Nachrichten mit der Rolle `user` und `system` müssen verschlüsselt werden.

<Warning>
Wenn E2EE-Header gesetzt sind, müssen **alle** `user`- und `system`-Rollen-Nachrichten verschlüsselt werden. Klartext in diesen Rollen führt zum Fehler "Encrypted field is not valid hex".
</Warning>

<CodeGroup>
```javascript JavaScript
import { gcm } from '@noble/ciphers/aes.js'
import { hkdf } from '@noble/hashes/hkdf.js'
import { sha256 } from '@noble/hashes/sha2.js'
import { ec as EC } from 'elliptic'
import crypto from 'crypto'

const HKDF_INFO = new TextEncoder().encode('ecdsa_encryption')

function encryptMessage(plaintext, modelPublicKeyHex) {
  const ec = new EC('secp256k1')

  // Public Key normalisieren (04-Präfix ergänzen, falls nötig)
  let normalizedKey = modelPublicKeyHex
  if (!normalizedKey.startsWith('04') && normalizedKey.length === 128) {
    normalizedKey = '04' + normalizedKey
  }

  const modelPublicKey = ec.keyFromPublic(normalizedKey, 'hex')

  // Ephemeres Schlüsselpaar für diese Nachricht erzeugen
  const ephemeralKeyPair = ec.genKeyPair()

  // ECDH-Shared-Secret
  const sharedSecret = ephemeralKeyPair.derive(modelPublicKey.getPublic())
  const sharedSecretBytes = new Uint8Array(sharedSecret.toArray('be', 32))

  // AES-Key per HKDF ableiten
  const aesKey = hkdf(sha256, sharedSecretBytes, undefined, HKDF_INFO, 32)

  // Zufällige Nonce erzeugen
  const nonce = crypto.randomBytes(12)

  // Mit AES-GCM verschlüsseln
  const cipher = gcm(aesKey, nonce)
  const encrypted = cipher.encrypt(new TextEncoder().encode(plaintext))

  // Ephemeren Public Key abrufen (uncompressed)
  const ephemeralPublic = new Uint8Array(ephemeralKeyPair.getPublic(false, 'array'))

  // Zusammenfügen: ephemeral_public (65 Bytes) + nonce (12 Bytes) + ciphertext
  const result = new Uint8Array(65 + 12 + encrypted.length)
  result.set(ephemeralPublic, 0)
  result.set(nonce, 65)
  result.set(encrypted, 65 + 12)

  return Buffer.from(result).toString('hex')
}

function encryptMessagesForE2EE(messages, modelPublicKey) {
  return messages.map(msg => {
    if (msg.role === 'user' || msg.role === 'system') {
      return {
        ...msg,
        content: encryptMessage(msg.content, modelPublicKey),
      }
    }
    return msg
  })
}
```

```python Python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes
from ecdsa import SECP256k1, VerifyingKey, SigningKey
import os

HKDF_INFO = b'ecdsa_encryption'

def encrypt_message(plaintext: str, model_public_key_hex: str) -> str:
    """Encrypt a message using ECDH + HKDF + AES-GCM."""
    # Public Key normalisieren
    key_hex = model_public_key_hex
    if not key_hex.startswith('04') and len(key_hex) == 128:
        key_hex = '04' + key_hex

    model_public_key_bytes = bytes.fromhex(key_hex)

    # Public Key des Modells parsen (04-Präfix überspringen)
    model_verifying_key = VerifyingKey.from_string(
        model_public_key_bytes[1:],
        curve=SECP256k1
    )

    # Ephemeres Schlüsselpaar für diese Nachricht erzeugen
    ephemeral_private = SigningKey.generate(curve=SECP256k1)
    ephemeral_public = ephemeral_private.get_verifying_key()

    # ECDH: Shared Secret berechnen
    shared_point = model_verifying_key.pubkey.point * ephemeral_private.privkey.secret_multiplier
    shared_secret = shared_point.x().to_bytes(32, 'big')

    # AES-Key per HKDF ableiten
    hkdf = HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=None,
        info=HKDF_INFO,
    )
    aes_key = hkdf.derive(shared_secret)

    # Zufällige Nonce erzeugen
    nonce = os.urandom(12)

    # Mit AES-GCM verschlüsseln
    aesgcm = AESGCM(aes_key)
    ciphertext = aesgcm.encrypt(nonce, plaintext.encode('utf-8'), None)

    # Ephemeren Public Key abrufen (uncompressed: 04 || x || y)
    ephemeral_public_bytes = b'\x04' + ephemeral_public.to_string()

    # Zusammenfügen: ephemeral_public (65 Bytes) + nonce (12 Bytes) + ciphertext
    result = ephemeral_public_bytes + nonce + ciphertext

    return result.hex()

def encrypt_messages_for_e2ee(messages: list, model_public_key: str) -> list:
    """Encrypt user and system messages."""
    encrypted_messages = []
    for msg in messages:
        if msg['role'] in ('user', 'system'):
            encrypted_messages.append({
                **msg,
                'content': encrypt_message(msg['content'], model_public_key)
            })
        else:
            encrypted_messages.append(msg)
    return encrypted_messages
```

### Schritt 6: Antwort-Chunks entschlüsseln

Antworten von E2EE-Modellen sind hex-kodierte verschlüsselte Chunks. Entschlüsseln Sie jeden Chunk mit Ihrem Private Key.

<CodeGroup>
```javascript JavaScript
import { gcm } from '@noble/ciphers/aes.js'
import { hkdf } from '@noble/hashes/hkdf.js'
import { sha256 } from '@noble/hashes/sha2.js'
import { ec as EC } from 'elliptic'

const HKDF_INFO = new TextEncoder().encode('ecdsa_encryption')

function hexToBytes(hex) {
  const h = hex.startsWith('0x') ? hex.slice(2) : hex
  const bytes = new Uint8Array(h.length / 2)
  for (let i = 0; i < bytes.length; i++) {
    bytes[i] = parseInt(h.substring(i * 2, i * 2 + 2), 16)
  }
  return bytes
}

function isHexEncrypted(s) {
  // Minimum: ephemeral_pub (65) + nonce (12) + tag (16) = 93 bytes = 186 hex chars
  if (s.length < 186) return false
  return /^[0-9a-fA-F]+$/.test(s)
}

function decryptChunk(ciphertextHex, clientPrivateKey) {
  const raw = hexToBytes(ciphertextHex)

  // Komponenten parsen
  const serverEphemeralPubKey = raw.slice(0, 65)
  const nonce = raw.slice(65, 65 + 12)
  const ciphertext = raw.slice(65 + 12)

  // ECDH mit dem ephemeren Schlüssel des Servers
  const ec = new EC('secp256k1')
  const clientKey = ec.keyFromPrivate(Buffer.from(clientPrivateKey))
  const serverKey = ec.keyFromPublic(Buffer.from(serverEphemeralPubKey))
  const sharedSecret = clientKey.derive(serverKey.getPublic())
  const sharedSecretBytes = new Uint8Array(sharedSecret.toArray('be', 32))

  // AES-Key ableiten
  const aesKey = hkdf(sha256, sharedSecretBytes, undefined, HKDF_INFO, 32)

  // Entschlüsseln
  const cipher = gcm(aesKey, nonce)
  const plaintext = cipher.decrypt(ciphertext)

  return new TextDecoder().decode(plaintext)
}

// Streaming-Antwort verarbeiten
async function processE2EEStream(response, clientPrivateKey) {
  const reader = response.body.getReader()
  const decoder = new TextDecoder()
  let fullContent = ''

  while (true) {
    const { done, value } = await reader.read()
    if (done) break

    const text = decoder.decode(value)
    const lines = text.split('\n')

    for (const line of lines) {
      if (!line.startsWith('data: ')) continue
      const data = line.slice(6)
      if (data === '[DONE]') continue

      try {
        const chunk = JSON.parse(data)
        const content = chunk.choices?.[0]?.delta?.content

        if (content && isHexEncrypted(content)) {
          const decrypted = decryptChunk(content, clientPrivateKey)
          fullContent += decrypted
          process.stdout.write(decrypted) // Echtzeit-Ausgabe
        } else if (content) {
          fullContent += content
          process.stdout.write(content)
        }
      } catch (e) {
        // Fehlerhafte Chunks überspringen
      }
    }
  }

  return fullContent
}
```

```python Python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes
from ecdsa import SECP256k1, VerifyingKey, SigningKey
import json
import re

HKDF_INFO = b'ecdsa_encryption'

def is_hex_encrypted(s: str) -> bool:
    """Check if string looks like hex-encrypted content."""
    if len(s) < 186:  # Minimum: 65 + 12 + 16 = 93 bytes = 186 hex
        return False
    return bool(re.match(r'^[0-9a-fA-F]+$', s))

def decrypt_chunk(ciphertext_hex: str, client_private_key: bytes) -> str:
    """Decrypt an E2EE response chunk."""
    raw = bytes.fromhex(ciphertext_hex)

    # Komponenten parsen
    server_ephemeral_pub = raw[:65]
    nonce = raw[65:77]
    ciphertext = raw[77:]

    # Ephemeren Public Key des Servers parsen (04-Präfix überspringen)
    server_verifying_key = VerifyingKey.from_string(
        server_ephemeral_pub[1:],
        curve=SECP256k1
    )

    # Private Key des Clients rekonstruieren
    client_signing_key = SigningKey.from_string(client_private_key, curve=SECP256k1)

    # ECDH: Shared Secret berechnen
    shared_point = server_verifying_key.pubkey.point * client_signing_key.privkey.secret_multiplier
    shared_secret = shared_point.x().to_bytes(32, 'big')

    # AES-Key ableiten
    hkdf = HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=None,
        info=HKDF_INFO,
    )
    aes_key = hkdf.derive(shared_secret)

    # Entschlüsseln
    aesgcm = AESGCM(aes_key)
    plaintext = aesgcm.decrypt(nonce, ciphertext, None)

    return plaintext.decode('utf-8')

def process_e2ee_stream(response, client_private_key: bytes) -> str:
    """Process streaming E2EE response."""
    full_content = ''

    for line in response.iter_lines():
        if not line:
            continue

        line_str = line.decode('utf-8')
        if not line_str.startswith('data: '):
            continue

        data = line_str[6:]
        if data == '[DONE]':
            continue

        try:
            chunk = json.loads(data)
            content = chunk.get('choices', [{}])[0].get('delta', {}).get('content', '')

            if content and is_hex_encrypted(content):
                decrypted = decrypt_chunk(content, client_private_key)
                full_content += decrypted
                print(decrypted, end='', flush=True)  # Echtzeit-Ausgabe
            elif content:
                full_content += content
                print(content, end='', flush=True)
        except json.JSONDecodeError:
            pass

    print()  # Abschließender Zeilenumbruch
    return full_content
```

```python
    print('🔑 Generating ephemeral key pair...')
    private_key = SigningKey.generate(curve=SECP256k1)
    public_key = private_key.get_verifying_key()
    client_public_key_hex = (b'\x04' + public_key.to_string()).hex()

        raise ValueError('Attestation verification failed')

    model_public_key = attestation.get('signing_key') or attestation.get('signing_public_key')
    print(f'✅ TEE attestation verified (provider: {attestation.get("tee_provider", "unknown")})')
```

### E2EE-Einschränkungen

<Warning>
E2EE hat aufgrund der Verschlüsselungsanforderungen einige Einschränkungen:
</Warning>

| Funktion | Status |
|---------|--------|
| Streaming | **Erforderlich** (Nicht-Streaming nicht unterstützt) |
| Web-Suche | **Deaktiviert** (würde Inhalt offenlegen) |
| Datei-Uploads | **Nicht unterstützt** |
| Function Calling | **Nicht unterstützt** |
| Venice-System-Prompt | **Deaktiviert** (muss client-seitig verschlüsselt werden) |

### Sicherheits-Best-Practices

1. **Neue Schlüsselpaare pro Session erzeugen** — Keine ephemeren Schlüssel wiederverwenden
2. **Private Keys mit Nullen überschreiben** — Private-Key-Bytes nach Gebrauch aus dem Speicher entfernen
3. **Attestation verifizieren** — Immer `verified: true` und Nonce-Übereinstimmung prüfen
4. **Auf Debug-Modus prüfen** — Attestations von Debug-Enklaven ablehnen
5. **Streaming verwenden** — E2EE erfordert Streaming für sauberes Verschlüsselungs-Chunking
6. **Fehler robust behandeln** — Entschlüsselungsfehler nicht an Nutzende weitergeben
7. **32-Byte-Nonces verwenden** — TEE-Anbieter verlangen exakt 32 Bytes

## Best Practices

<AccordionGroup>
  <Accordion title="Attestation in Produktion immer verifizieren">
    Vertrauen Sie nicht nur der Antwort `verified: true`. Parsen Sie den Intel-TDX-Quote client-seitig und prüfen Sie, ob die Messwerte den erwarteten Werten entsprechen. Für NVIDIA-GPUs prüfen Sie die Attestation über den Verifikationsdienst von NVIDIA.
  </Accordion>
  
  <Accordion title="Frische Nonces verwenden">
    Erzeugen Sie für jede Attestation-Anfrage eine neue zufällige Nonce. Dadurch werden Replay-Attacken verhindert, bei denen ein Angreifer eine veraltete Attestation ausliefern könnte.
  </Accordion>
  
  <Accordion title="Key-Binding verifizieren">
    Der Signing-Key sollte an das TDX-`REPORTDATA`-Feld gebunden sein. Damit ist nachgewiesen, dass der Schlüssel innerhalb der Enklave erzeugt wurde.
  </Accordion>
  
  <Accordion title="Auf Debug-Modus prüfen">
    Prüfen Sie, dass die TDX-Attestation keine Debug-Flags gesetzt hat. Eine Debug-Enklave kann inspiziert werden und sollte in Produktion nicht vertraut werden.
  </Accordion>
  
  <Accordion title="Unsere SDKs für E2EE verwenden">
    E2EE erfordert eine sorgfältige kryptografische Implementierung. Verwenden Sie unsere offiziellen SDKs, anstatt das Protokoll selbst zu implementieren.
  </Accordion>
</AccordionGroup>

## Fehlerbehandlung

| Fehler | Ursache | Lösung |
|-------|-------|----------|
| `TEE attestation verification failed` | Attestation hat die Validierung nicht bestanden | Erneut versuchen oder Support kontaktieren |
| `Attestation nonce mismatch` | Möglicher Replay-Angriff | Frische Nonce erzeugen |
| `TDX debug mode detected` | Enklave im Debug-Modus | Nicht für Produktion verwenden |
| `Failed to decrypt field` | Serverseitige E2EE-Entschlüsselung fehlgeschlagen | Verschlüsselungs-Implementierung prüfen |
| `E2EE requires streaming` | Nicht-Streaming-Anfrage an ein E2EE-Modell | `stream: true` setzen |
| `Encrypted field is not valid hex` | Klartext mit E2EE-Headern gesendet | Alle user/system-Nachrichten verschlüsseln |
| `Invalid public key` | Falsches Schlüsselformat | 130 Hex-Zeichen, beginnend mit `04`, verwenden |

## Fehlersuche

<AccordionGroup>
  <Accordion title="502 Bad Gateway oder 'Nonce must be exactly 32 bytes'">
    Die Nonce-Länge ist falsch. TEE-Anbieter verlangen exakt **32 Bytes (64 Hex-Zeichen)**.
    
    - `crypto.randomBytes(32).toString('hex')` (JS) oder `secrets.token_hex(32)` (Python) verwenden
    - Häufiger Fehler: `secrets.token_hex(16)` ergibt 32 Hex-Zeichen (16 Bytes), nicht 32 Bytes
  </Accordion>
  
  <Accordion title="Attestation-Verifikation fehlgeschlagen">
    - Prüfen, dass das Modell E2EE unterstützt (`supportsE2EE: true`)
    - API-Schlüssel-Gültigkeit und Zugriff auf das angefragte Modell verifizieren
    - Netzwerkverbindung zur Venice-API prüfen
  </Accordion>
  
  <Accordion title="Entschlüsselung fehlgeschlagen">
    - Sicherstellen, dass derselbe Private Key verwendet wird, der den in den Headern gesendeten Public Key erzeugt hat
    - Prüfen, dass der Antwortinhalt tatsächlich hex-kodiert ist (E2EE aktiv)
    - Sicherstellen, dass der für die Verschlüsselung verwendete Modell-Public-Key übereinstimmt
  </Accordion>
  
  <Accordion title="Encrypted field is not valid hex">
    - Alle `user`- und `system`-Rollen-Nachrichten müssen verschlüsselt werden, wenn E2EE-Header gesetzt sind
    - Verifizieren, dass Ihr verschlüsselter Inhalt die `isValidEncrypted()`-Validierung besteht (mindestens 186 Hex-Zeichen)
    - Prüfen, dass die Verschlüsselungsausgabe in Kleinbuchstaben-Hex ohne Präfixe vorliegt
  </Accordion>
  
  <Accordion title="'Invalid public key'-Fehler">
    - Der Client-Public-Key muss exakt **130 Hex-Zeichen** lang sein und mit `04` beginnen
    - Die Hilfsfunktion `validateClientPubkey()` zur Format-Prüfung vor dem Versand verwenden
    - Sicherstellen, dass das Uncompressed-Public-Key-Format verwendet wird (65 Bytes = 130 Hex-Zeichen)
  </Accordion>
  
  <Accordion title="Modell nicht gefunden">
    - Modell-ID auf Korrektheit prüfen und sicherstellen, dass das Modell E2EE unterstützt
    - Über den `/models`-Endpoint die verfügbaren E2EE-Modelle prüfen
  </Accordion>
</AccordionGroup>
