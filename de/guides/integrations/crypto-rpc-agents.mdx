Venice gibt Ihrem Agenten sowohl Inference (230+ Modelle) als auch Blockchain-Zugriff (10 EVM-Chains plus Starknet) über eine einzige Credential. Ihr Agent kann denken, signieren und Transaktionen senden, ohne separate Accounts für Inference- und RPC-Anbieter zu jonglieren.

## Warum Venice für On-Chain-Agenten?

| Fähigkeit | Was Ihr Agent erhält |
|---|---|
| **Inference** | 230+ Text-, Bild-, Video-, Audio- und Embedding-Modelle über einen OpenAI-kompatiblen Endpoint |
| **Crypto-RPC** | JSON-RPC-2.0-Proxy zu 10 EVM-Chains plus Starknet (Mainnet und Testnets) |
| **Authentifizierung** | Standard-API-Schlüssel oder x402-Wallet-Auth (kein Venice-Konto erforderlich) |
| **Finanzierung** | Autonom: VVV-Staking für tägliches DIEM. Browser: USD- oder Krypto-Top-ups über das Dashboard |
| **Batching** | Bis zu 100 JSON-RPC-Aufrufe pro Anfrage, Multi-Chain parallel |
| **Idempotenz** | Sichere Retries mit dem `Idempotency-Key`-Header |

## Authentifizierung

Wählen Sie die Auth-Methode, die zu Ihrem Agenten-Setup passt.

| Methode | Am besten für | Funktionsweise |
|---|---|---|
| **API-Schlüssel** | Serverseitige Agenten, feste Deployments | `Authorization: Bearer <key>`-Header. Schlüssel über [venice.ai/settings/api](https://venice.ai/settings/api) holen. |
| **x402-Wallet** | Autonome, krypto-native oder kurzlebige Agenten | Wallet signiert eine Sign-In-With-X-Nachricht und zahlt pro Anfrage in USDC auf Base oder Solana. Kein Venice-Konto nötig. Siehe den [x402-Leitfaden](/de/guides/integrations/x402-venice-api). |

Beide Methoden teilen sich dieselben Rate Limits und Abrechnung in Venice-Credits.

<Tip>
Wirklich autonome Agenten können ihren eigenen API-Schlüssel erzeugen, indem sie VVV auf Base staken. Siehe [API-Schlüssel-Erzeugung für autonome Agenten](/de/guides/getting-started/generating-api-key-agent).
</Tip>

## Crypto-RPC-Schnellstart

Senden Sie jede JSON-RPC-2.0-Methode an `POST /crypto/rpc/{network}`.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

Antwort:

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Response-Header enthalten `X-Venice-RPC-Credits` (berechnete Credits), `X-Venice-RPC-Cost-USD` (Dollar-Kosten) und `X-Request-ID` (Korrelations-ID).

### Unterstützte Netzwerke

| Familie | Mainnet | Testnets |
|---|---|---|
| Ethereum | `ethereum-mainnet` | `ethereum-sepolia`, `ethereum-holesky` |
| Base | `base-mainnet` | `base-sepolia` |
| Arbitrum | `arbitrum-mainnet` | `arbitrum-sepolia` |
| Optimism | `optimism-mainnet` | `optimism-sepolia` |
| Polygon | `polygon-mainnet` | `polygon-amoy` |
| Linea | `linea-mainnet` | `linea-sepolia` |
| Avalanche C-Chain | `avalanche-mainnet` | `avalanche-fuji` |
| BNB Smart Chain | `bsc-mainnet` | `bsc-testnet` |
| Blast | `blast-mainnet` | `blast-sepolia` |
| zkSync Era | `zksync-mainnet` | `zksync-sepolia` |
| Starknet | `starknet-mainnet` | `starknet-sepolia` |

Verwenden Sie [`GET /crypto/rpc/networks`](/de/api-reference/endpoint/crypto/networks) für die maßgebliche Live-Liste.

### Methoden-Stufen

Methoden sind in drei Credit-Stufen gruppiert. Gesamtkosten = `baseCredits[chain] × methodTier`.

| Stufe | Multiplikator | Beispiele |
|---|---|---|
| **Standard** | 1× | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas` |
| **Advanced** | 2× | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction` |
| **Large** | 4× | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content` |

Vollständige Liste und Preisdetails in der [Crypto-RPC-API-Referenz](/de/api-reference/endpoint/crypto/rpc).

## Agenten-Rezepte

Gängige Muster für KI-Agenten, die On-Chain lesen und schreiben müssen.

### Native Balance einer Wallet lesen

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getBalance",
    "params": ["0xYourWalletAddress", "latest"],
    "id": 1
  }'
```

### ERC-20-Token-Balance lesen

Rufen Sie den `balanceOf(address)`-Selektor mit `eth_call` auf. Das `data`-Feld ist der 4-Byte-Selektor (`0x70a08231`), gefolgt von der links auf 32 Bytes aufgefüllten Wallet-Adresse. Am einfachsten ist es, eine Bibliothek die Kodierung übernehmen zu lassen:

```typescript
import { encodeFunctionData, parseAbi } from 'viem'

const data = encodeFunctionData({
  abi: parseAbi(['function balanceOf(address) view returns (uint256)']),
  args: ['0xWalletAddress'],
})

const response = await fetch('https://api.venice.ai/api/v1/crypto/rpc/base-mainnet', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.VENICE_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    jsonrpc: '2.0',
    method: 'eth_call',
    params: [{ to: '0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf', data }, 'latest'],
    id: 1,
  }),
})
```

Die obige Vertragsadresse ist VVV auf Base. Tauschen Sie sie gegen einen beliebigen ERC-20-Vertrag aus.

### Eine signierte Transaktion senden (vollständiger Lebenszyklus)

Venice hält Ihre privaten Schlüssel nie. Der Agent sammelt tx-Parameter per RPC-Lese-Aufrufen, signiert lokal mit einer Bibliothek wie [viem](https://viem.sh) oder [ethers](https://docs.ethers.org) und leitet die rohen Hex-Daten durch Venice.

<Steps>
  <Step title="Die nächste Nonce holen">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionCount","params":["0xAgentWallet","pending"],"id":1}'
    ```
    Verwenden Sie `"pending"`, damit aufeinanderfolgende Sends nicht kollidieren.
  </Step>
  <Step title="Gas-Preis abrufen">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_gasPrice","params":[],"id":1}'
    ```
    Bei EIP-1559-Chains bevorzugen Sie `eth_feeHistory`, um `maxFeePerGas` und `maxPriorityFeePerGas` zu berechnen.
  </Step>
  <Step title="Gas abschätzen">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"from":"0xAgentWallet","to":"0xRecipient","value":"0x0","data":"0x..."}],"id":1}'
    ```
  </Step>
  <Step title="Lokal signieren">
    ```typescript
    import { privateKeyToAccount } from 'viem/accounts'
    import { base } from 'viem/chains'

    const account = privateKeyToAccount(process.env.AGENT_PRIVATE_KEY)

    const signed = await account.signTransaction({
      chainId: base.id,
      nonce,                  // aus Schritt 1
      gas,                    // aus Schritt 3
      maxFeePerGas,           // aus Schritt 2 (fee history)
      maxPriorityFeePerGas,   // aus Schritt 2 (fee history)
      to: '0xRecipient',
      value: 0n,
      data: '0x...',
    })
    ```
  </Step>
  <Step title="Über Venice einreichen">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Idempotency-Key: agent-tx-<id>" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_sendRawTransaction","params":["0xSignedHex"],"id":1}'
    ```
    Setzen Sie bei Relays immer `Idempotency-Key`, damit ein Netzwerkaussetzer keinen doppelten Broadcast verursachen kann.
  </Step>
  <Step title="Receipt pollen">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xTxHash"],"id":1}'
    ```
    Pollen Sie alle paar Sekunden, bis `result` nicht-null ist. Prüfen Sie `result.status` (`"0x1"` = Erfolg).
  </Step>
</Steps>

<Note>
Jeder `eth_sendRawTransaction`-Aufruf wird serverseitig mit Tx-Hash, Netzwerk, Request-ID und aufrufender User-ID protokolliert. Die signierte Nutzlast selbst wird nicht aufbewahrt. Dieses Audit-Trail existiert, damit kompromittierte, für illegale Relays missbrauchte Schlüssel auf das verantwortliche Konto zurückgeführt werden können.
</Note>

### Mehrere Aufrufe batchen (Multi-Chain-Portfolio-Check)

Senden Sie bis zu 100 JSON-RPC-Objekte in einer Anfrage. Jedes wird unabhängig validiert und abgerechnet.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[
    { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 1 },
    { "jsonrpc": "2.0", "method": "eth_getBalance", "params": ["0xWallet", "latest"], "id": 2 },
    { "jsonrpc": "2.0", "method": "eth_gasPrice", "params": [], "id": 3 }
  ]'
```

Für Multi-Chain-Reads (ein Aufruf pro Chain) stellen Sie parallele Requests an verschiedene `{network}`-Endpoints.

### Sichere Retries mit Idempotenz

Setzen Sie den `Idempotency-Key`-Header auf einen beliebigen String, der dem Muster `[A-Za-z0-9_-]{1,255}` entspricht. Venice cacht die Antwort 24 Stunden lang anhand `(user, key)`. Replays geben das gecachte Ergebnis mit `Idempotent-Replayed: true` zurück und berechnen nichts.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Idempotency-Key: agent-tx-2026-04-21-001" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_sendRawTransaction",
    "params": ["0xSignedRawTxHex"],
    "id": 1
  }'
```

Das ist entscheidend für Transaktions-Relays, bei denen ein Netzwerkaussetzer sonst dazu führen könnte, dass Ihr Agent dieselbe Tx zweimal sendet.

## Den API-Schlüssel des Agenten finanzieren

Sobald der Agent einen Venice-API-Schlüssel hat, benötigt er ein einlösbares Guthaben am zugrunde liegenden Konto, bevor kostenpflichtige Endpoints den Schlüssel akzeptieren. Es gibt zwei Wege, dort Guthaben anzulegen:

| Weg | Autonom? | Funktionsweise |
|---|---|---|
| **DIEM aus VVV-Staking** | Ja | VVV im [Venice-Staking-Smart-Contract](https://basescan.org/address/0x321b7ff75154472B18EDb199033fF4D116F340Ff#code) auf Base staken. Die tägliche DIEM-Zuteilung der Wallet ist proportional zu ihrem Anteil am Staking-Pool. Das Konto benötigt mindestens 0,1 angesammeltes DIEM, bevor überhaupt DIEM ausgegeben werden kann. DIEM wird um 00:00 UTC erneuert. Um die täglichen Ausgaben zu erhöhen, mehr VVV staken. |
| **USD- oder Krypto-Top-up via Dashboard** | Nein (Browser) | Melden Sie sich auf [venice.ai](https://venice.ai) mit derselben Wallet an (Sign-In-With-Ethereum) und ergänzen Sie in den Einstellungen unter API Credits. Sowohl Stripe (Karte) als auch Coinbase (Krypto) liegen hinter dieser Seite und erfordern einen Browser. Credits verfallen nie. |

Für einen unbeaufsichtigt laufenden Agenten ist **DIEM via VVV-Staking heute der einzige vollständig headless Finanzierungsweg für einen erzeugten API-Schlüssel**. Überschreiten die täglichen Ausgaben des Agenten seine DIEM-Zuteilung, sind die realistischen Optionen: mehr VVV staken oder einen Operator anmelden und in USD oder Krypto aufladen lassen.

### Autonomes VVV-Staking und Schlüsselerzeugung

Ein wirklich autonomer Agent kann seine eigene VVV-Wallet auf Base verwalten, sie staken und seinen eigenen Venice-API-Schlüssel ohne menschliches Eingreifen erzeugen. Der vollständige Ablauf:

<Steps>
  <Step title="VVV und ETH für Gas beschaffen">
    Senden Sie VVV an die Wallet des Agenten (oder lassen Sie den Agenten auf [Aerodrome](https://aerodrome.finance) oder [Uniswap](https://app.uniswap.org) tauschen) plus etwas ETH auf Base für die beiden Staking-Transaktionen.
  </Step>
  <Step title="VVV staken">
    `approve` des Staking-Contracts am VVV-Token, dann `stake(amount)` an `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. Der sVVV-Saldo der Wallet wird atomar mit dem Stake aktualisiert.
  </Step>
  <Step title="Einen API-Schlüssel erzeugen">
    `GET /api/v1/api_keys/generate_web3_key` gibt ein JWT zurück, das 15 Minuten nach Ausstellung abläuft. Signieren Sie den rohen Token mit der Staking-Wallet und senden Sie anschließend Adresse, Signatur und Token per `POST` zurück. Venice gibt einen API-Schlüssel zurück, der an das aus dieser Wallet abgeleitete Nutzerkonto gebunden ist.
  </Step>
</Steps>

Das Erzeugen erfordert lediglich einen sVVV-Saldo ungleich Null — 1 gestaktes VVV reicht aus, um einen Schlüssel zu erhalten. **Das Ausgeben** mit dem Schlüssel ist eine separate Frage, geregelt durch die Finanzierungstabelle oben.

Den vollständigen Durchlauf mit Code und der vollen Fehlerreferenz finden Sie unter [API-Schlüssel-Erzeugung für autonome Agenten](/de/guides/getting-started/generating-api-key-agent).

## x402-Wallet-Auth in 30 Sekunden

Hat Ihr Agent bereits eine Base- oder Solana-Wallet, können Sie den API-Schlüssel komplett überspringen. Das SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) übernimmt Sign-In-With-X-Signierung, Top-ups und Guthabenverfolgung.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // überspringen, wenn die Wallet bereits Guthaben hat

const response = await venice.chat({
  model: 'kimi-k2-6',
  messages: [{ role: 'user', content: 'What is the latest block on Base?' }]
})
```

Dieselbe Wallet-Auth funktioniert auch gegen `/crypto/rpc/{network}` für Blockchain-Reads und -Writes. Vollständige Protokolldetails im [x402-Leitfaden](/de/guides/integrations/x402-venice-api).

## Preise

Crypto-RPC wird in Venice-Credits abgerechnet. Jede Antwort enthält `X-Venice-RPC-Credits` (berechnete Credits) und `X-Venice-RPC-Cost-USD` (Dollar-Kosten), sodass Ihr Agent die Ausgaben pro Anfrage verfolgen kann.

### Basis-Credits pro Chain

| Basis-Credits | Chains |
|---|---|
| **20** | Ethereum, Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast, Starknet |
| **30** | zkSync Era |

### Kostenbeispiele

Beobachtete Preise für Standard-, Advanced- und Large-Methodenstufen:

| Aufruf | Credits | USD-Kosten |
|---|---|---|
| `eth_call` auf Ethereum (20 × 1×) | 20 | ~0,0000140 $ |
| `trace_transaction` auf Ethereum (20 × 2×) | 40 | ~0,0000280 $ |
| `trace_replayTransaction` auf Ethereum (20 × 4×) | 80 | ~0,0000560 $ |
| `eth_call` auf zkSync (30 × 1×) | 30 | ~0,0000210 $ |

Vertrauen Sie stets dem Header `X-Venice-RPC-Cost-USD` für die maßgeblichen Kosten. Fehlerhafte Elemente in Batch-Anfragen werden mit pauschal 5 Credits pro Stück berechnet.

### Rate Limits

| Tier | Requests pro Minute |
|---|---|
| Standard | 100 |
| Staff | 1.000 |

Bei Überschreitung gibt der Endpoint `429` mit den Standard-`X-RateLimit-*`-Response-Headern zurück.

## Fehlerbehandlung

Übliche HTTP-Antworten, die Ihr Agent behandeln sollte:

| Status | Bedeutung | Was zu tun ist |
|---|---|---|
| `400` | Nicht unterstützte oder nicht zugeordnete JSON-RPC-Methode oder fehlerhafter Batch | Methode gegen die [Allowlist](/de/api-reference/endpoint/crypto/rpc) prüfen. Der Fehler-Body nennt die problematische Methode. |
| `400` | Replay eines `Idempotency-Key` mit anderem Body | Verwenden Sie einen frischen Schlüssel für unterschiedliche Anfragen. |
| `402` | Gar kein Auth-Header (Antwort-Body enthält `authOptions` mit beiden unterstützten Auth-Wegen) oder Credits aufgebraucht trotz gültigem Auth-Header | Ohne Auth: `Authorization: Bearer ...` oder den x402-`X-Sign-In-With-X`-Header anhängen. Ohne Credits: bei Bearer-Schlüssel Konto auffüllen (DIEM, USD oder Dashboard-Top-up); bei x402-Auth direkt `POST /api/v1/x402/top-up` aufrufen. |
| `429` | Rate Limit erreicht (100 req/min Standard, 1.000 req/min Staff) | `X-RateLimit-Reset` beachten und zurücknehmen. Batchen Sie bis zu 100 Aufrufe pro Anfrage, um das Limit zu amortisieren. |
| `5xx` | Upstream-RPC-Knoten-Aussetzer | Mit demselben `Idempotency-Key` erneut senden, um doppelte Belastungen zu vermeiden. |

Pro-Element-Batch-Fehler (z. B. ungültige Params an einem von N Aufrufen) kommen in einer `200 OK`-Antwort mit einem JSON-RPC-`error`-Feld am betroffenen Element zurück. Diese Elemente werden mit pauschal 5 Credits pro Stück berechnet.

## Nicht unterstützt

Folgende Methoden-Kategorien werden bewusst abgelehnt:

- **Nur-WebSocket** (`eth_subscribe`, `eth_unsubscribe`): Der Proxy ist HTTP-only. Stattdessen pollen.
- **Zustandsbehaftete Filter** (`eth_newFilter`, `eth_getFilterChanges` usw.): Filterzustand ist an ein einzelnes Backend gebunden und bricht auf einem lastverteilten Proxy. Verwenden Sie stattdessen `eth_getLogs`.
- **Schlüssel-haltende Methoden** (`eth_sign`, `eth_accounts`, `eth_mining`): Gehostete Anbieter halten keine Nutzerschlüssel. Client-seitig signieren und über `eth_sendRawTransaction` einreichen.
- **Nicht zugeordnete Methoden**: Alles, was nicht in der Allowlist steht, liefert `400`. Wenden Sie sich an den Support, um Ergänzungen anzufragen.
