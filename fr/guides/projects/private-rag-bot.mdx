Le retrieval-augmented generation, ou RAG, est l'un des schémas les plus utiles pour construire des applications IA qui doivent répondre à partir de vos propres documents. Au lieu de demander à un modèle de se reposer uniquement sur sa mémoire, vous récupérez d'abord du matériel source pertinent, envoyez ce contexte au modèle et lui demandez de répondre avec des citations.

Dans ce tutoriel, nous allons construire un bot RAG privé en utilisant Python, Venice pour les embeddings et chat completions, Qdrant pour la recherche vectorielle et FastEmbed pour le re-ranking local. À la fin, vous aurez les composants principaux d'un assistant de documents local qui peut ingérer vos fichiers, récupérer les chunks pertinents, les re-classer et répondre avec des citations.

## Comment fonctionne un bot RAG moderne

Un bon pipeline RAG est plus que "mettre des documents dans une base de données vectorielle". Le flux de base ressemble à ceci :

| Étape | Ce qui se passe |
| --- | --- |
| Load | Lire des fichiers Markdown, texte ou reStructuredText locaux |
| Chunk | Découper les longs documents en sections chevauchantes |
| Embed | Utiliser les embeddings Venice pour transformer les chunks en vecteurs |
| Store | Enregistrer les vecteurs et les métadonnées de source dans Qdrant |
| Retrieve | Embedder la question de l'utilisateur et lancer la recherche vectorielle |
| Re-rank | Utiliser un cross-encoder pour re-noter les meilleurs candidats |
| Answer | Envoyer le meilleur contexte à un modèle de chat Venice avec des instructions de citation |

L'étape de re-ranking est l'amélioration qui rend ceci plus utile qu'une démo RAG basique. La recherche vectorielle est rapide et performante pour trouver des chunks sémantiquement similaires, mais elle peut encore renvoyer des passages adjacents au sujet plutôt que directement utiles. Un cross-encoder lit la question et chaque chunk candidat ensemble, puis note dans quelle mesure ce chunk répond réellement à la question.

## Choisir les modèles

Créez un fichier nommé `rag_bot.py`, puis commencez par ajouter les imports, les structures de données, l'URL de l'API et les noms de modèles :

```python
import os
import textwrap
import uuid
from dataclasses import dataclass
from pathlib import Path

from fastembed.rerank.cross_encoder import TextCrossEncoder
from openai import OpenAI
from qdrant_client import QdrantClient, models

VENICE_BASE_URL = "https://api.venice.ai/api/v1"
CHAT_MODEL = "kimi-k2-6"
EMBEDDING_MODEL = "text-embedding-bge-m3"
RERANKER_MODEL = "Xenova/ms-marco-MiniLM-L-6-v2"
COLLECTION_NAME = "private_rag_bot"


@dataclass
class SourceDocument:
    content: str
    metadata: dict


@dataclass
class RankedChunk:
    content: str
    metadata: dict
    vector_score: float
    rerank_score: float
```

Le nom du modèle d'embedding est intentionnellement compatible OpenAI. Venice mappe les noms de modèles d'embedding compatibles aux modèles d'embedding hébergés par Venice, donc le code existant utilisant le SDK OpenAI peut généralement migrer en changeant la `base_url` et la clé API.

Vous pouvez lister les modèles Venice disponibles avec :

```bash
curl "https://api.venice.ai/api/v1/models?type=embedding" \
  -H "Authorization: Bearer $VENICE_API_KEY"
```

Pour les modèles de chat :

```bash
curl "https://api.venice.ai/api/v1/models?type=text" \
  -H "Authorization: Bearer $VENICE_API_KEY"
```

## Créer les clients Venice et Qdrant

Créez un seul client Venice compatible OpenAI pour à la fois les embeddings et les chat completions :

```python
venice = OpenAI(
    api_key=os.environ["VENICE_API_KEY"],
    base_url=VENICE_BASE_URL,
)
```

Pour Qdrant, vous avez trois modes utiles :

| Mode | Quand l'utiliser |
| --- | --- |
| `QdrantClient(":memory:")` | Démos et tests locaux rapides |
| `QdrantClient(path="./qdrant_data")` | Stockage local persistant |
| `QdrantClient(url=..., api_key=...)` | Un cluster Qdrant distant ou managé |

Pour un bot local privé, commencez avec un chemin Qdrant local sur disque :

```python
qdrant = QdrantClient(path="./qdrant_data")
```

Il y a différentes façons de gérer le déploiement en production. Cependant, si vous utilisez un déploiement Qdrant distant, rappelez-vous que vos chunks de documents et métadonnées y seront stockés. Venice peut garder la couche d'inférence privée, mais vous devez quand même choisir le bon déploiement Qdrant pour vos données.

## Charger et chunker les documents

Pour ce tutoriel, nous laissons le bot ingérer des fichiers ou dossiers locaux. Commencez par les fichiers `.md`, `.rst` et `.txt` :

```python
TEXT_EXTENSIONS = {".md", ".rst", ".txt"}

def expand_paths(paths: list[Path]) -> list[Path]:
    files = []
    for path in paths:
        if path.is_dir():
            files.extend(
                sorted(
                    file_path
                    for file_path in path.rglob("*")
                    if file_path.is_file()
                    and file_path.suffix.lower() in TEXT_EXTENSIONS
                )
            )
        elif path.is_file():
            files.append(path)
        else:
            raise FileNotFoundError(f"Document path does not exist: {path}")
    return files
```

Une fois les fichiers chargés, nous devons découper le texte en le "chunkant" — le séparer en chunks de données. Une stratégie naïve pourrait diviser les chunks de manière égale. Cependant, dans la plupart des cas, cela peut perdre des informations à des frontières sémantiques données, ce qui peut réduire l'efficacité de votre système RAG.

La stratégie de chunking que nous utiliserons préfère les frontières de paragraphe ou de phrase afin que le modèle obtienne un contexte cohérent :

```python
def chunk_text(text: str, chunk_size: int, chunk_overlap: int) -> list[str]:
    clean_text = textwrap.dedent(text).strip()
    if not clean_text:
        return []
    if len(clean_text) <= chunk_size:
        return [clean_text]

    chunks = []
    start = 0
    while start < len(clean_text):
        end = min(start + chunk_size, len(clean_text))

        if end < len(clean_text):
            paragraph_break = clean_text.rfind("\n\n", start, end)
            sentence_break = clean_text.rfind(". ", start, end)
            split_at = max(paragraph_break, sentence_break)
            if split_at > start + chunk_size // 2:
                end = split_at + 1

        chunk = clean_text[start:end].strip()
        if chunk:
            chunks.append(chunk)

        if end >= len(clean_text):
            break

        start = max(end - chunk_overlap, start + 1)

    return chunks
```

Une taille de chunk de départ de `1000` caractères avec `150` caractères de chevauchement est une bonne valeur par défaut pour des documents mixtes Markdown et texte. Des chunks plus petits peuvent améliorer la précision. Des chunks plus grands peuvent préserver plus de contexte. Le bon réglage dépend souvent des types de documents que vous stockez.

## Embedder les documents avec Venice

Une fois que nous avons les chunks, nous les embeddons en lots :

```python
def embed(texts: list[str]) -> list[list[float]]:
    embeddings = []
    for start in range(0, len(texts), 32):
        batch = texts[start : start + 32]
        response = venice.embeddings.create(
            model="text-embedding-bge-m3",
            input=batch,
        )
        embeddings.extend(
            item.embedding
            for item in sorted(response.data, key=lambda item: item.index)
        )
    return embeddings
```

Le batching importe. Embedder un chunk à la fois est simple, mais cela ajoute une latence évitable. Gardez la taille du lot configurable afin que vous puissiez ajuster le débit en fonction de votre charge de travail.

## Stocker les vecteurs dans Qdrant

Avant d'insérer des points, créez une collection Qdrant avec la bonne taille de vecteur. La façon la plus facile de connaître la taille du vecteur est d'embedder le premier lot, puis d'utiliser `len(embeddings[0])`.

```python
qdrant.create_collection(
    collection_name=COLLECTION_NAME,
    vectors_config=models.VectorParams(
        size=len(embeddings[0]),
        distance=models.Distance.COSINE,
    ),
)
```

Chaque point stocke le vecteur plus les métadonnées de payload. Le payload inclut le texte original et un chemin source afin que la réponse puisse citer d'où provient le contexte :

```python
points.append(
    models.PointStruct(
        id=chunk_id,
        vector=embedding,
        payload={
            "text": chunk.content,
            "source": source,
            "chunk_index": chunk_index,
        },
    )
)

qdrant.upsert(collection_name=COLLECTION_NAME, points=points)
```

Utilisez des UUIDs déterministes dérivés de `source`, `chunk_index` et du contenu. Cela rend l'ingestion répétée idempotente pour les chunks non modifiés.

## Récupérer les chunks candidats

Au moment de la question, le bot embedde la question de l'utilisateur et demande à Qdrant les meilleures correspondances vectorielles :

```python
query_vector = embed([question])[0]
hits = qdrant.query_points(
    collection_name=COLLECTION_NAME,
    query=query_vector,
    with_payload=True,
    limit=8,
).points
```

Le `limit` ici est le nombre de candidats. Il devrait généralement être supérieur au nombre de chunks que vous prévoyez d'envoyer au modèle parce que l'étape suivante les re-classera. Une bonne valeur par défaut est de récupérer `8` candidats et d'envoyer les `4` meilleurs au modèle de chat.

## Re-ranking avec FastEmbed

Maintenant nous ajoutons la partie qui rend la récupération beaucoup plus intelligente.

```python
from fastembed.rerank.cross_encoder import TextCrossEncoder

reranker = TextCrossEncoder(model_name="Xenova/ms-marco-MiniLM-L-6-v2")

candidate_texts = [str((hit.payload or {}).get("text", "")) for hit in hits]
rerank_scores = list(reranker.rerank(question, candidate_texts))
reranked = sorted(
    zip(hits, rerank_scores),
    key=lambda hit_and_score: hit_and_score[1],
    reverse=True,
)
```

La différence importante entre la recherche par embedding et le re-ranking par cross-encoder est la façon dont le scoring se produit.

La recherche par embedding compare un vecteur pour la question contre un vecteur pour chaque chunk. C'est rapide et scalable. Un cross-encoder évalue la question et le chunk ensemble. C'est plus lent, mais cela peut juger la pertinence plus directement.

C'est pourquoi le schéma habituel est :

1. Récupérer un ensemble de candidats plus large avec la recherche vectorielle.
2. Re-classer uniquement ces candidats localement.
3. Envoyer les quelques meilleurs chunks au modèle de langage.

Un bon point de départ est `candidate_k=8` et `top_k=4`. Augmentez `candidate_k` si la bonne source est souvent à proximité mais ne parvient pas dans le contexte final.

## Répondre avec les chat completions Venice

Une fois le contexte sélectionné, formatez-le avec des numéros de source :

```python
def format_context(chunks: list[RankedChunk]) -> str:
    if not chunks:
        return "No relevant context was retrieved."

    context_parts = []
    for index, chunk in enumerate(chunks, start=1):
        source = chunk.metadata.get("source", "unknown")
        context_parts.append(
            f"[{index}] Source: {source} | "
            f"Vector score: {chunk.vector_score:.4f} | "
            f"Rerank score: {chunk.rerank_score:.4f}\n"
            f"{chunk.content}"
        )
    return "\n\n---\n\n".join(context_parts)
```

Puis envoyez le contexte à un modèle de chat Venice :

```python
response = venice.chat.completions.create(
    model="kimi-k2-6",
    temperature=0.2,
    messages=[
        {
            "role": "system",
            "content": (
                "You are a helpful RAG assistant. Answer using only the supplied "
                "context. If the context does not answer the question, say that "
                "you do not have enough information."
            ),
        },
        {
            "role": "user",
            "content": (
                f"Retrieved context:\n{context}\n\n"
                f"Question: {question}\n\n"
                "Answer with citations like [1] when the context supports the answer:"
            ),
        },
    ],
)
```

Notez le system prompt : on dit au bot de répondre uniquement à partir du contexte fourni. C'est un garde-fou simple mais important. Un assistant RAG ne doit pas répondre avec confiance à partir des connaissances générales du modèle lorsque les documents récupérés ne soutiennent pas la réponse.

## Exécuter le bot

Une fois que vous assemblez les pièces dans un script, enregistrez-le sous `rag_bot.py`. Une première exécution simple peut utiliser quelques documents d'exemple intégrés afin que vous puissiez vérifier le pipeline avant d'ingérer vos propres fichiers :

```bash
python rag_bot.py \
  --question "What does reranking improve in a RAG pipeline?"
```

Pour ingérer vos propres documents :

```bash
python rag_bot.py \
  --docs ./docs \
  --question "What does this project do?"
```

Pour garder une collection Qdrant locale sur disque et démarrer un chat interactif :

```bash
python rag_bot.py \
  --docs ./docs \
  --qdrant-path ./qdrant_data \
  --chat
```

Le script affiche la réponse, puis affiche les sources avec à la fois les scores vectoriels et de re-ranking :

```text
Answer
============================================================
Reranking improves retrieval quality by rescoring the top
vector-search candidates with a cross-encoder model [1].

Sources
============================================================
1. sample-docs (vector=0.8123, rerank=0.7342)
```

Si vous voulez inspecter le texte réel passé au modèle, ajoutez :

```bash
--show-context
```

## Options CLI utiles

Exposez les principaux paramètres de récupération comme options CLI afin que vous puissiez ajuster le bot sans modifier le code :

| Option | Défaut | Ce qu'elle contrôle |
| --- | --- | --- |
| `--candidate-k` | `8` | Nombre de résultats de recherche vectorielle à re-classer |
| `--top-k` | `4` | Nombre de chunks re-classés envoyés au modèle de chat |
| `--chunk-size` | `1000` | Taille maximale de chunk avant chevauchement |
| `--chunk-overlap` | `150` | Caractères répétés entre chunks voisins |
| `--embedding-batch-size` | `32` | Nombre de chunks par requête d'embeddings Venice |
| `--qdrant-path` | non défini | Chemin de stockage local persistant Qdrant |
| `--qdrant-url` | non défini | URL Qdrant distante |
| `--skip-ingest` | `false` | Interroger une collection existante sans recharger les docs |
| `--recreate-collection` | `false` | Supprimer et reconstruire la collection Qdrant |

Pour un développement local répété, un flux courant est :

```bash
python rag_bot.py \
  --docs ./docs \
  --qdrant-path ./qdrant_data \
  --recreate-collection \
  --question "Summarize the most important setup steps."
```

Puis posez des questions de suivi sans réingérer :

```bash
python rag_bot.py \
  --qdrant-path ./qdrant_data \
  --skip-ingest \
  --question "Which file explains deployment?"
```

## Notes sur la confidentialité

Pour une configuration RAG privée, réfléchissez à chaque couche séparément :

| Couche | Considération de confidentialité |
| --- | --- |
| Embeddings Venice | Les chunks de documents sont envoyés à Venice pour créer des vecteurs |
| Chat Venice | Le contexte récupéré est envoyé à Venice pour répondre à la question |
| Qdrant local | Les vecteurs et payloads restent sur votre machine |
| Qdrant distant | Les vecteurs et payloads sont stockés là où s'exécute votre serveur Qdrant |
| Re-ranker FastEmbed | Le re-ranking s'exécute localement une fois le modèle disponible |

La valeur par défaut la plus privée pour ce tutoriel est Venice pour l'inférence, Qdrant local sur disque et re-ranking FastEmbed local. Cela vous donne un bot RAG pratique sans envoyer les payloads de votre base de données vectorielle à un magasin de vecteurs tiers.

## Erreurs courantes à gérer dès le départ

| Symptôme | Ce que cela signifie habituellement | Que faire |
| --- | --- | --- |
| `Set VENICE_API_KEY before running this example.` | La variable d'environnement est manquante | Exportez `VENICE_API_KEY` avant d'exécuter le script |
| `Document path does not exist` | Un chemin passé à `--docs` est erroné | Vérifiez le chemin du fichier ou du dossier |
| Résultats de récupération vides | Rien n'a été ingéré, ou la mauvaise collection est interrogée | Retirez `--skip-ingest` ou confirmez `--collection` et `--qdrant-path` |
| Erreur de taille de vecteur Qdrant | La collection a été créée avec un modèle d'embedding différent | Recréez la collection après avoir changé de modèles d'embedding |
| Premier re-rank lent | FastEmbed peut être en train de télécharger ou d'initialiser le cross-encoder | Laissez la première exécution se terminer, puis les exécutions suivantes devraient être plus rapides |

Si vous changez de modèles d'embedding, recréez la collection Qdrant. Différents modèles d'embedding peuvent produire des vecteurs de dimensions différentes, et les collections Qdrant attendent une taille de vecteur fixe.

## Pour aller plus loin

Une fois que vous avez la base de référence en marche, les améliorations à plus fort impact sont généralement :

- Ajouter des chargeurs spécifiques au document pour les PDF, HTML, tickets ou pages wiki internes.
- Stocker des métadonnées plus riches telles que les titres, en-têtes, dates, propriétaires et URLs.
- Ajuster `candidate_k`, `top_k`, la taille du chunk et le chevauchement sur de vraies questions.
- Ajouter des questions d'évaluation afin de pouvoir mesurer la qualité de la récupération avant et après les modifications.
- Streamer la chat completion Venice finale pour une meilleure expérience de chat interactif.

Les systèmes RAG sont faciles à démontrer et étonnamment faciles à rendre médiocres. Le schéma recherche vectorielle plus re-ranking est une fondation solide parce qu'il garde la récupération rapide tout en donnant au bot une meilleure chance d'envoyer au modèle de langage le bon contexte.
