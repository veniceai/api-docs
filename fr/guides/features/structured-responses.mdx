Venice inclut désormais les sorties structurées via "response\_format" comme champ disponible dans l'API. Ce champ vous permet de générer des réponses à vos prompts qui suivent un format prédéfini spécifique. Avec cette nouvelle méthode, les modèles sont moins susceptibles d'halluciner des clés ou des valeurs incorrectes dans la réponse, ce qui était plus fréquent en tentant la manipulation via system prompt ou via function calling.

Le champ "response\_format" pour les sorties structurées utilise le format de l'API OpenAI, et est décrit plus en détail dans le guide OpenAI [ici](https://platform.openai.com/docs/guides/structured-outputs). OpenAI a également publié un article d'introduction sur l'utilisation des sorties structurées dans l'API [ici](https://openai.com/index/introducing-structured-outputs-in-the-api/). Comme il s'agit d'une fonctionnalité avancée, il y a quelques "gotchas" en bas de cette page qu'il convient de suivre.

Cette fonctionnalité n'est pas disponible nativement pour tous les modèles. Référez-vous à la section modèles [ici](https://docs.venice.ai/api-reference/endpoint/models/list?playground=open), et recherchez "supportsResponseSchema" pour les modèles applicables.

```json
    {
      "id": "venice-uncensored",
      "type": "text",
      "object": "model",
      "created": 1726869022,
      "owned_by": "venice.ai",
      "model_spec": {
        "availableContextTokens": 32768,
        "capabilities": {
          "supportsFunctionCalling": true,
          "supportsResponseSchema": true,
          "supportsWebSearch": true
        },
```

### Comment utiliser les réponses structurées

Pour utiliser correctement "response\_format", vous pouvez définir votre schéma avec diverses "properties", représentant des catégories de sorties, chacune avec des types de données configurés individuellement. Ces objets peuvent être imbriqués pour créer des structures de sortie plus avancées.

Voici un exemple d'appel d'API utilisant response\_format pour expliquer le processus étape par étape de résolution d'une équation mathématique.

Vous pouvez voir que les properties ont été configurées pour exiger à la fois "steps" et "final\_answer" dans la réponse. Dans l'imbrication, la catégorie steps consiste en une "explanation" et une "output", chacune en chaîne de caractères.

```json
curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header 'Authorization: Bearer <api-key>' \
  --header 'Content-Type: application/json' \
  --data '{
  "model": "venice-uncensored",
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful math tutor."
    },
    {
      "role": "user",
      "content": "solve 8x + 31 = 2"
    }
  ],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "math_response",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "steps": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "explanation": {
                  "type": "string"
                },
                "output": {
                  "type": "string"
                }
              },
              "required": ["explanation", "output"],
              "additionalProperties": false
            }
          },
          "final_answer": {
            "type": "string"
          }
        },
        "required": ["steps", "final_answer"],
        "additionalProperties": false
      }
    }
  }
}

```

Voici la réponse reçue du modèle. Vous pouvez voir que la structure a suivi les exigences en fournissant d'abord les "steps" avec l'"explanation" et l'"output" de chaque étape, puis la "final answer".

```json
{
  "steps": [
    {
      "explanation": "Subtract 31 from both sides to isolate the term with x.",
      "output": "8x + 31 - 31 = 2 - 31"
    },
    {
      "explanation": "This simplifies to 8x = -29.",
      "output": "8x = -29"
    },
    {
      "explanation": "Divide both sides by 8 to solve for x.",
      "output": "x = -29 / 8"
    }
  ],
  "final_answer": "x = -29 / 8"
}

```

Bien que ceci soit un exemple simple, cela peut être extrapolé à des cas d'usage plus avancés comme : l'extraction de données, les exercices de chaîne de pensée, la génération d'interface, la catégorisation de données et bien d'autres.

### Gotchas

Voici quelques exigences clés à garder à l'esprit lors de l'utilisation des sorties structurées via response\_format :

* Les requêtes initiales utilisant response\_format peuvent prendre plus de temps à générer une réponse. Les requêtes suivantes ne connaîtront pas la même latence que la requête initiale.

* Pour les requêtes plus volumineuses, le modèle peut échouer à se terminer si `max_tokens` ou le timeout du modèle sont atteints, ou si des limites de débit sont violées

* Un format de schéma incorrect entraînera des erreurs à la complétion, généralement dues au timeout

* Bien que response\_format garantisse que le modèle produira une sortie d'une manière particulière, cela ne garantit pas que le modèle a fourni les bonnes informations à l'intérieur. Le contenu est piloté par le prompt et la performance du modèle.

* Les sorties structurées via response\_format ne sont pas compatibles avec les appels de fonction parallèles

* Important : tous les champs ou paramètres doivent inclure une balise `required`. Pour rendre un champ optionnel, vous devez ajouter une option `null` dans le `type` du champ, comme ceci `"type": ["string", "null"]`&#x20;

* Il est possible de rendre les champs optionnels en donnant des options `null` dans le champ required pour permettre une réponse vide.

* Important : `additionalProperties` doit être défini sur false pour que response\_format fonctionne correctement

* Important : `strict` doit être défini sur true pour que response\_format fonctionne correctement
