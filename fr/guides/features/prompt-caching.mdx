Le prompt caching stocke les tokens d'entrée traités afin que les requêtes suivantes avec des préfixes identiques puissent les réutiliser au lieu de les retraiter. Cela réduit la latence (jusqu'à 80 % pour les longs prompts) et les coûts (jusqu'à 90 % de réduction sur les tokens en cache).

Venice gère le caching automatiquement pour les modèles pris en charge, mais comprendre comment chaque fournisseur implémente le caching vous aide à maximiser les taux de hit du cache et à minimiser les coûts.

## Comment fonctionne le caching

Le caching opère par **correspondance de préfixe** : le système stocke les tokens traités et les réutilise lorsque les requêtes suivantes commencent par le même contenu.

Considérez un chatbot avec un system prompt de 2 000 tokens :

<Steps>
  <Step title="Requête 1">
    System prompt (2 000 tokens) + message utilisateur (50 tokens)
    
    **Traités** : 2 050 tokens · **Depuis le cache** : 0 tokens
    
    Préfixe écrit dans le cache.
  </Step>
  <Step title="Requête 2">
    System prompt (2 000 tokens) + message utilisateur (80 tokens)
    
    **Traités** : 80 tokens · **Depuis le cache** : 2 000 tokens
  </Step>
  <Step title="Requête 3">
    System prompt (2 000 tokens) + message utilisateur (120 tokens)
    
    **Traités** : 120 tokens · **Depuis le cache** : 2 000 tokens
  </Step>
</Steps>

**Total sans caching** : 2 050 + 2 080 + 2 120 = 6 250 tokens au prix plein

**Total avec caching** : 2 050 + 80 + 120 = 2 250 tokens au prix plein, 4 000 tokens au tarif réduit

<Warning>
Le caching ne fonctionne que sur le **préfixe**. Tout changement au début de votre prompt invalide le cache pour tout ce qui suit. Placez toujours le contenu statique (system prompt, documents, exemples) avant le contenu dynamique (messages utilisateur).
</Warning>

## Comportement spécifique au fournisseur

Envoyez simplement vos requêtes et vérifiez la réponse pour les statistiques de cache. **Claude** nécessite des marqueurs de cache explicites au niveau du protocole, mais Venice les ajoute automatiquement pour les system prompts et l'historique de conversation.

### Claude Opus 4.5 (Anthropic)

Claude nécessite des points de coupure de cache explicites au niveau du protocole. Venice gère cela automatiquement :

- Les **system prompts** sont mis en cache automatiquement
- L'**historique de conversation** est mis en cache en plaçant un point de coupure sur l'avant-dernier message utilisateur

Cela signifie que votre historique de conversation est lu depuis le cache, et seul le dernier tour est traité comme une nouvelle entrée :

| Tour | Tokens de prompt | Lecture cache | Écriture cache | Économies |
|------|---------------|------------|-------------|---------|
| 1 | 10 979 | 0 | 10 938 | Première écriture |
| 2 | 11 031 | 10 938 | 31 | 99,7 % en cache |
| 3 | 11 062 | 10 969 | 52 | 99,5 % en cache |

**Détails supplémentaires :**

- **Jusqu'à 4 points de coupure par requête** : le système utilise le plus long préfixe correspondant
- **La clé de cache est exacte au byte près** : les changements d'espace, les encodages d'image différents ou les outils réordonnés cassent les hits de cache
- **Limites de débit conscientes du cache** : les tokens en cache ne comptent pas dans votre limite ITPM, permettant un débit effectif plus élevé
- **Prime d'écriture de 25 %** : la première requête coûte plus cher, mais 90 % d'économies sur les lectures suivantes

#### Contrôle manuel du cache

Pour des cas particuliers comme mettre en cache un grand document au premier tour, vous pouvez ajouter des points de coupure explicites :

```json
{
  "messages": [
    {
      "role": "system",
      "content": [{
        "type": "text",
        "text": "You are a legal assistant...",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    {
      "role": "user", 
      "content": [{
        "type": "text",
        "text": "[Long contract document...]",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    { "role": "assistant", "content": "I've reviewed the contract." },
    { "role": "user", "content": "What are the termination clauses?" }
  ]
}
```

Cela garantit que le system prompt et le document sont mis en cache dès la première requête. Pour les conversations typiques, vous n'avez pas besoin de marqueurs manuels.

### Tous les autres modèles

Le caching est **automatique**. Aucun paramètre spécial n'est nécessaire. Assurez-vous simplement que vos prompts dépassent ~1 024 tokens et utilisez `prompt_cache_key` pour un routage cohérent.

## Paramètres de requête

| Paramètre | Type | Modèles | Description |
|-----------|------|--------|-------------|
| `prompt_cache_key` | string | Tous | Indice de routage pour l'affinité de cache. Les requêtes avec la même clé sont plus susceptibles d'atteindre le même serveur avec un cache chaud. |
| `cache_control` | object | Claude | Marque les blocs de contenu pour le caching. Voir la section Claude Opus 4.5. |

### prompt_cache_key

Pour les conversations ou workflows agentiques, utilisez une `prompt_cache_key` cohérente pour améliorer les taux de hit du cache :

```json
{
  "model": "claude-opus-4-5",
  "prompt_cache_key": "session-abc-123",
  "messages": [...]
}
```

Cela route les requêtes vers des serveurs susceptibles d'avoir déjà votre contexte en cache. Utilisez un ID de session, ID de conversation ou ID d'utilisateur comme clé.

## Champs de réponse

L'objet `usage` de la réponse inclut des statistiques de cache :

```json
{
  "usage": {
    "prompt_tokens": 5500,
    "completion_tokens": 200,
    "total_tokens": 5700,
    "prompt_tokens_details": {
      "cached_tokens": 5000,
      "cache_creation_input_tokens": 0
    }
  }
}
```

| Champ | Description |
|-------|-------------|
| `prompt_tokens` | Total des tokens d'entrée dans la requête |
| `prompt_tokens_details.cached_tokens` | Tokens servis depuis le cache (facturés au tarif réduit) |
| `prompt_tokens_details.cache_creation_input_tokens` | Tokens écrits dans le cache (peut entraîner une prime sur Claude) |

**Détail de facturation** (en utilisant Claude Opus 4.5 comme exemple) :
- 5 000 tokens en cache × \$0.60/1M = \$0.003
- 500 tokens hors cache × \$6.00/1M = \$0.003
- Total : \$0.006 (vs \$0.033 sans caching, 82 % d'économies)

## Bonnes pratiques

### Structurez les prompts pour le caching

Placez le contenu statique au début, le contenu dynamique à la fin.

**Bonne structure**

| Position | Contenu | En cache ? |
|----------|---------|---------|
| 1 | Instructions système | Oui |
| 2 | Documents de référence | Oui |
| 3 | Exemples few-shot | Oui |
| 4 | Requête utilisateur | Non |

**Mauvaise structure**

| Position | Contenu | En cache ? |
|----------|---------|---------|
| 1 | Timestamp actuel | Non (invalide tout ce qui suit) |
| 2 | Instructions système | Non |
| 3 | Requête utilisateur | Non |

### Gardez les préfixes identiques au byte près

Les clés de cache sont calculées à partir de séquences de bytes exactes. Même des différences triviales cassent les hits de cache :

- Différences d'espaces ou de nouvelles lignes
- Timestamps ou IDs de requête dans les prompts
- Ordonnancement randomisé d'exemples few-shot
- Formatage différent du même contenu

### Atteignez les seuils minimums de tokens

Si vos prompts sont en dessous du minimum (typiquement 1 024 tokens), le caching ne s'activera pas. Pour les petits prompts, envisagez :

- Ajouter plus de contexte ou d'exemples pour atteindre le seuil
- Grouper plusieurs petites requêtes en prompts batchés
- Accepter que le caching ne s'applique pas pour les requêtes simples

### Utilisez prompt_cache_key pour les conversations

Pour les conversations en cours, définissez une `prompt_cache_key` cohérente :

```json
// Turn 1
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turn 2
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turn 3
{ "prompt_cache_key": "conv-xyz", "messages": [...] }
```

Cela augmente la probabilité que tous les tours atteignent le même serveur avec un cache chaud.

### Surveillez la performance du cache

Suivez ces métriques :
- **Taux de hit du cache** : `cached_tokens / prompt_tokens`
- **Économies de coût** : comparez le coût réel vs le coût sans cache
- **Réduction de latence** : time-to-first-token avec vs sans hits de cache

Si `cached_tokens` est constamment 0 :
1. Les prompts peuvent être en dessous du seuil minimum de tokens
2. Les prompts peuvent changer entre requêtes
3. Les requêtes peuvent atteindre différents serveurs (utilisez `prompt_cache_key`)
4. Le cache peut avoir expiré (requêtes trop peu fréquentes)

### Considérez l'économie du cache

**Prime d'écriture de cache Claude Opus 4.5** : la première requête coûte 25 % de plus, mais 90 % d'économies sur les lectures suivantes.

| Scénario | La prime d'écriture vaut-elle la peine ? |
|----------|-------------------------------|
| 1 requête avec ce prompt | Non (paye 25 % de plus, aucun bénéfice) |
| 2+ requêtes avec le même préfixe | Oui (équilibre à la 2e requête) |
| Prompts changeant rapidement | Non (coûts d'écriture constants) |
| System prompt stable, nombreuses requêtes | Oui (amorti sur de nombreuses lectures) |

## Durée du cache

Les caches expirent après une période d'inactivité (typiquement 5-10 minutes). Cela signifie :

| Schéma de trafic | Bénéfice du caching |
|-----------------|-----------------|
| Requêtes continues (< 5 min d'écart) | Élevé : le cache reste chaud |
| Trafic en rafales (écarts > 10 min) | Limité : le cache expire entre les rafales |
| Requêtes sporadiques (à des heures d'intervalle) | Aucun : cache toujours froid |

## Caching avec outils et fonctions

Les définitions de fonctions peuvent être mises en cache avec les system prompts :

```json
{
  "model": "claude-opus-4-5",
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_database",
        "description": "Search the product database",
        "parameters": { ... }
      }
    }
  ],
  "messages": [
    {
      "role": "system",
      "content": [
        {
          "type": "text",
          "text": "You are a shopping assistant...",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    ...
  ]
}
```

Les définitions d'outils deviennent partie du préfixe en cache. Si vous avez de nombreux outils, cela peut réduire significativement les coûts par requête.

## Caching avec images et documents

Pour les modèles de vision, les images peuvent être incluses dans le contenu en cache :

```json
{
  "model": "claude-opus-4-5",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": { "url": "data:image/png;base64,..." }
        },
        {
          "type": "text",
          "text": "This is the floor plan. I'll ask several questions about it.",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    {
      "role": "assistant",
      "content": "I can see the floor plan. What would you like to know?"
    },
    {
      "role": "user",
      "content": "How many bedrooms are there?"
    }
  ]
}
```

L'image et le contexte initial sont mis en cache, donc les questions de suivi sur la même image ne la retraitent pas.

## Dépannage

<Accordion title="cached_tokens est toujours 0">

| Cause | Solution |
|-------|----------|
| Prompt trop court | Vérifiez que le prompt dépasse ~1 024 tokens (4 000 pour Claude) |
| Préfixe modifié | Vérifiez le contenu dynamique au début de votre prompt |
| Première requête | Attendu : la première requête écrit dans le cache, les suivantes lisent |
| Cache expiré | Réduisez le temps entre les requêtes à moins de 5 minutes |
| Serveurs différents | Ajoutez `prompt_cache_key` pour router les requêtes de manière cohérente |

</Accordion>

<Accordion title="cache_creation_input_tokens à chaque requête">

| Cause | Solution |
|-------|----------|
| Prompt qui change | Retirez timestamps, IDs de requête ou autre contenu dynamique du préfixe |
| cache_control manquant | Pour Claude, vérifiez que le marqueur `cache_control` est présent sur les blocs de contenu |
| En dessous du seuil | Les prompts en dessous du nombre minimum de tokens ne déclenchent pas le caching |
| Message utilisateur unique | Attendu pour le premier tour. Le cache grandit avec l'historique de conversation. |

</Accordion>

<Accordion title="Coûts plus élevés que prévu">

| Cause | Solution |
|-------|----------|
| Prime d'écriture de cache | Claude facture 25 % de plus pour les écritures. Ne vaut le coup que si vous réutilisez le prompt. |
| Faible réutilisation | Si chaque prompt est unique, vous payez les coûts d'écriture sans bénéfice de lecture |
| Mauvaise structure de prompt | Déplacez le contenu dynamique à la fin afin que le préfixe reste stable |

</Accordion>
