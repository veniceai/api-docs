X402 vous permet d'utiliser les routes payantes de l'API Venice en vous authentifiant avec un portefeuille et en maintenant un solde USDC prépayé. Pas de clé API ni de compte requis. Signez un message, rechargez sur Base ou Solana, et appelez n'importe quelle route prise en charge.

## Qu'est-ce que X402 ?

[X402](https://www.x402.org/) est un standard de paiement ouvert qui permet aux applications et aux agents de payer des services de manière programmatique en utilisant des crypto-monnaies. Venice implémente X402 pour que les portefeuilles puissent s'authentifier et payer l'inférence directement avec de l'USDC sur Base ou Solana.

## Prérequis

- Un portefeuille sur Base ou Solana
- Un token natif pour le gas sur la chaîne sélectionnée, tel que ETH sur Base ou SOL sur Solana
- De l'USDC sur la chaîne sélectionnée (ou un solde existant adossé à DIEM depuis un compte Venice lié)

<Tip>
  Envisagez d'utiliser un portefeuille dédié pour l'automatisation plutôt que votre portefeuille principal.
</Tip>

## Démarrage rapide

Le SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) fournit des helpers pour l'auth de portefeuille, les rechargements et le suivi du solde.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // skip if the wallet already has spendable balance

const response = await venice.chat({
  model: 'kimi-k2-5',
  messages: [{ role: 'user', content: 'Hello!' }]
})
```

Le client génère un en-tête `X-Sign-In-With-X` neuf pour chaque requête et suit automatiquement le solde à partir des en-têtes de réponse `X-Balance-Remaining`.

### Avec des outils compatibles OpenAI

Si vous utilisez un outil qui accepte un `fetch` personnalisé, utilisez `createAuthFetch` pour ajouter l'auth de portefeuille à n'importe quelle requête :

```typescript
import { createAuthFetch } from 'venice-x402-client'

const authFetch = createAuthFetch(process.env.WALLET_KEY)
```

### Helpers disponibles

Le SDK inclut des helpers de première classe pour les routes Venice x402 les plus courantes. Pour tout ce qui n'est pas couvert, utilisez `request()` ou `createAuthFetch()` directement.

| Catégorie | Méthodes |
| --- | --- |
| Chat | `chat()`, `chatStream()` |
| Responses | `responses.create()`, `responses.stream()` |
| Images | `images.generate()`, `images.generations()`, `images.upscale()`, `images.edit()`, `images.multiEdit()`, `images.backgroundRemove()` |
| Audio | `audio.speech()`, `audio.transcribe()`, `audio.queue()`, `audio.retrieve()`, `audio.complete()` |
| Vidéo | `video.queue()`, `video.retrieve()`, `video.generate()`, `video.complete()`, `video.transcribe()` |
| Embeddings | `embeddings()` |
| Models | `models()` |
| Portefeuille | `getBalance()`, `getTransactions()`, `topUp()` |

## Flux manuel

Si vous n'utilisez pas le SDK ou avez besoin de comprendre le protocole sous-jacent, voici le flux étape par étape. Pour un nouveau portefeuille, supposez que vous devez recharger d'abord, sauf s'il dispose déjà d'un solde DIEM dépensable.

### Étape 1 : créer l'en-tête X-Sign-In-With-X

Venice attend une charge utile JSON encodée en Base64 contenant un message Sign-In-With-X signé. Les portefeuilles EVM signent un message SIWE EIP-4361 sur Base. Les portefeuilles Solana signent le message SIWX Solana avec Ed25519. Générez un nonce et un timestamp neufs pour chaque flux de requête.

Pour Solana, définissez `chainId` sur `solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp`, incluez `type: "ed25519"` dans la charge utile JSON encodée et fournissez la signature Ed25519 en base58 ou base64. Le message signé commence par `<domain> wants you to sign in with your Solana account:`, suivi de l'adresse du portefeuille et des champs standard `URI`, `Version`, `Chain ID`, `Nonce`, `Issued At` et `Expiration Time` optionnel.

```typescript
import { Wallet } from 'ethers'
import { SiweMessage, generateNonce } from 'siwe'

const wallet = new Wallet(process.env.EVM_PRIVATE_KEY)
const now = new Date()
const resourceUrl = 'https://api.venice.ai/api/v1/chat/completions'

const siwe = new SiweMessage({
  domain: 'api.venice.ai',
  address: wallet.address,
  statement: 'Sign in to Venice AI',
  uri: resourceUrl,
  version: '1',
  chainId: 8453,
  nonce: generateNonce(),
  issuedAt: now.toISOString(),
  expirationTime: new Date(now.getTime() + 5 * 60 * 1000).toISOString(),
})

const message = siwe.prepareMessage()
const signature = await wallet.signMessage(message)

const xSignInWithX = Buffer.from(
  JSON.stringify({
    address: wallet.address,
    message,
    signature,
    timestamp: now.getTime(),
    chainId: 8453,
  }),
  'utf8',
).toString('base64')

console.log(xSignInWithX)
```

### Étape 2 : vérifier le solde

Avant d'effectuer une requête payante, vérifiez que le portefeuille a un solde dépensable :

```bash
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS_OR_SOLANA_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

La réponse inclut :

- `canConsume` : indique si le portefeuille peut effectuer des requêtes payantes
- `balanceUsd` : solde dépensable actuel
- `minimumTopUpUsd` et `suggestedTopUpUsd` : conseils pour les rechargements
- `diemBalanceUsd` : solde adossé à DIEM, le cas échéant

Le paramètre de chemin `walletAddress` accepte soit une adresse EVM (`0x...`), soit une adresse Solana base58.

### Étape 3 : recharger

Rechargez avec de l'USDC sur Base ou Solana :

```bash
curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up
```

Le premier appel renvoie `402 Payment Required` avec un en-tête `PAYMENT-REQUIRED` contenant un tableau `accepts`. Chaque entrée décrit une option de paiement acceptée, incluant `network`, `asset`, `payTo` et `amount`. Choisissez l'option Base ou Solana avec laquelle vous voulez payer, utilisez ces détails exacts pour construire un en-tête `X-402-Payment`, et réessayez la même route.

#### Construire l'en-tête X-402-Payment pour Base

Le script suivant crée un paiement x402 signé sur Base et envoie la requête de rechargement. Nécessite les packages npm `x402` et `viem`.

```bash
export EVM_PRIVATE_KEY=0xYOUR_PRIVATE_KEY
```

```bash
export X402_PAYMENT="$(
node --input-type=module <<'EOF'
import { createPaymentHeader } from "x402/client";
import { privateKeyToAccount } from "viem/accounts";

const signer = privateKeyToAccount(process.env.EVM_PRIVATE_KEY);
const amountUsd = 5;
const amountBaseUnits = String(Math.round(amountUsd * 1e6));

const header = await createPaymentHeader(signer, 2, {
  scheme: "exact",
  network: "base",
  maxAmountRequired: amountBaseUnits,
  resource: "https://api.venice.ai/api/v1/x402/top-up",
  description: "Venice x402 top-up",
  mimeType: "application/json",
  payTo: "0x2670B922ef37C7Df47158725C0CC407b5382293F",
  maxTimeoutSeconds: 300,
  asset: "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
  extra: { name: "USD Coin", version: "2" },
});

process.stdout.write(header);
EOF
)"

curl -X POST "https://api.venice.ai/api/v1/x402/top-up" \
  -H "X-402-Payment: $X402_PAYMENT"
```

<Note>
  Utilisez la dernière réponse `PAYMENT-REQUIRED` / `accepts` comme source de vérité en production plutôt que de coder en dur ces valeurs. Pour les rechargements Solana, construisez le paiement à partir de l'entrée Solana renvoyée dans `accepts`. Les entrées Solana utilisent `network: "solana"`, le mint USDC comme `asset`, et peuvent inclure des métadonnées spécifiques au réseau telles que `extra.feePayer`.
</Note>

### Étape 4 : effectuer une requête

Une fois que le portefeuille a un solde dépensable, appelez n'importe quel endpoint pris en charge avec l'en-tête `X-Sign-In-With-X` :

```bash
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header "Content-Type: application/json" \
  --header "X-Sign-In-With-X: $X402_AUTH" \
  --data '{
    "model": "kimi-k2-5",
    "messages": [
      {
        "role": "user",
        "content": "Hello from an x402-authenticated wallet."
      }
    ]
  }'
```

Les réponses réussies peuvent inclure un en-tête `X-Balance-Remaining`.

### Étape 5 : inspecter les transactions (optionnel)

Consultez l'historique des transactions du portefeuille :

```bash
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS_OR_SOLANA_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/transactions/$WALLET_ADDRESS?limit=10&offset=0" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

Le grand livre inclut des entrées telles que `TOP_UP`, `CHARGE` et `REFUND`.

Le paramètre de chemin `walletAddress` accepte soit une adresse EVM (`0x...`), soit une adresse Solana base58.

## Routes prises en charge

### Routes d'inférence payantes

Les routes publiques payantes Venice suivantes prennent actuellement en charge l'authentification par portefeuille x402.

| Catégorie | Endpoints |
| --- | --- |
| Chat | `POST /api/v1/chat/completions`, `POST /api/v1/responses` |
| Image | `POST /api/v1/image/generate`, `POST /api/v1/images/generations`, `POST /api/v1/image/upscale`, `POST /api/v1/image/edit`, `POST /api/v1/image/multi-edit`, `POST /api/v1/image/background-remove` |
| Embeddings | `POST /api/v1/embeddings` |
| Audio | `POST /api/v1/audio/speech`, `POST /api/v1/audio/transcriptions`, `POST /api/v1/audio/complete`, `POST /api/v1/audio/queue`, `POST /api/v1/audio/retrieve` |
| Vidéo | `POST /api/v1/video/complete`, `POST /api/v1/video/queue`, `POST /api/v1/video/retrieve`, `POST /api/v1/video/transcriptions` |

### Route de rechargement

| Endpoint | Auth | Objectif |
| --- | --- | --- |
| `POST /api/v1/x402/top-up` | Requête initiale : aucune. Retry : `X-402-Payment` | Ajouter des crédits USDC au solde dépensable du portefeuille en utilisant une option de paiement Base ou Solana acceptée |

### Routes réservées au portefeuille

Ces routes utilisent `X-Sign-In-With-X` pour l'identité mais ne facturent pas le solde.

| Endpoint | Objectif |
| --- | --- |
| `GET /api/v1/x402/balance/{walletAddress}` | Vérifier le solde dépensable pour une adresse de portefeuille EVM ou Solana |
| `GET /api/v1/x402/transactions/{walletAddress}` | Consulter l'historique des transactions pour une adresse de portefeuille EVM ou Solana |

## Erreurs

| Statut | Cause probable | Que faire |
| --- | --- | --- |
| `401` | Charge utile Sign-In-With-X malformée ou expirée | Reconstruisez `X-Sign-In-With-X` avec un nonce et un timestamp neufs |
| `402` sur une route payante | Solde dépensable insuffisant | Rechargez et réessayez |
| `402` sur `/x402/top-up` | Attendu. C'est le flux d'initiation de paiement. | Utilisez les détails de paiement renvoyés pour construire `X-402-Payment` et réessayez |
| `403` sur balance ou transactions | Décalage de portefeuille | Assurez-vous que le portefeuille authentifié correspond au paramètre de chemin `walletAddress` |
| `400` sur top-up | En-tête de paiement malformé | Reconstruisez à partir de la dernière réponse `402` |

---

## Pour les agents

Le flux est le même. Stockez les clés privées dans des variables d'environnement ou un gestionnaire de secrets, et vérifiez le solde avant les requêtes pour éviter des allers-retours `402` inutiles.
