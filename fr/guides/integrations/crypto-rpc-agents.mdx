Venice donne à votre agent à la fois l'inférence (230+ modèles) et l'accès blockchain (10 chaînes EVM plus Starknet) via un seul justificatif. Votre agent peut réfléchir, signer et envoyer des transactions sans jongler avec des comptes séparés pour les fournisseurs d'inférence et de RPC.

## Pourquoi Venice pour les agents on-chain ?

| Capacité | Ce que votre agent obtient |
|---|---|
| **Inférence** | 230+ modèles texte, image, vidéo, audio et embedding via un seul endpoint compatible OpenAI |
| **RPC crypto** | Proxy JSON-RPC 2.0 vers 10 chaînes EVM plus Starknet (mainnet et testnets) |
| **Authentification** | Clé API standard ou auth de portefeuille x402 (aucun compte Venice requis) |
| **Financement** | Autonome : staking VVV pour du DIEM quotidien. Navigateur : rechargements USD ou crypto via le tableau de bord |
| **Batching** | Jusqu'à 100 appels JSON-RPC par requête, multi-chaînes en parallèle |
| **Idempotence** | Nouveaux essais sûrs avec en-tête `Idempotency-Key` |

## Authentification

Choisissez la méthode d'auth qui correspond à la façon dont votre agent s'exécute.

| Méthode | Idéal pour | Comment ça marche |
|---|---|---|
| **Clé API** | Agents côté serveur, déploiements fixes | En-tête `Authorization: Bearer <key>`. Obtenez une clé sur [venice.ai/settings/api](https://venice.ai/settings/api). |
| **Portefeuille x402** | Agents autonomes, crypto-natifs ou éphémères | Le portefeuille signe un message Sign-In-With-X, paie à la requête en USDC sur Base ou Solana. Aucun compte Venice nécessaire. Consultez le [guide x402](/fr/guides/integrations/x402-venice-api). |

Les deux méthodes partagent les mêmes limites de débit et la même facturation en crédits Venice.

<Tip>
Les agents véritablement autonomes peuvent émettre leur propre clé API en stakant VVV sur Base. Voir [Création autonome de clé API par un agent](/fr/guides/getting-started/generating-api-key-agent).
</Tip>

## Démarrage rapide RPC crypto

Envoyez n'importe quelle méthode JSON-RPC 2.0 à `POST /crypto/rpc/{network}`.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

Réponse :

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Les en-têtes de réponse incluent `X-Venice-RPC-Credits` (crédits facturés), `X-Venice-RPC-Cost-USD` (coût en dollars) et `X-Request-ID` (ID de corrélation).

### Réseaux pris en charge

| Famille | Mainnet | Testnets |
|---|---|---|
| Ethereum | `ethereum-mainnet` | `ethereum-sepolia`, `ethereum-holesky` |
| Base | `base-mainnet` | `base-sepolia` |
| Arbitrum | `arbitrum-mainnet` | `arbitrum-sepolia` |
| Optimism | `optimism-mainnet` | `optimism-sepolia` |
| Polygon | `polygon-mainnet` | `polygon-amoy` |
| Linea | `linea-mainnet` | `linea-sepolia` |
| Avalanche C-Chain | `avalanche-mainnet` | `avalanche-fuji` |
| BNB Smart Chain | `bsc-mainnet` | `bsc-testnet` |
| Blast | `blast-mainnet` | `blast-sepolia` |
| zkSync Era | `zksync-mainnet` | `zksync-sepolia` |
| Starknet | `starknet-mainnet` | `starknet-sepolia` |

Utilisez [`GET /crypto/rpc/networks`](/fr/api-reference/endpoint/crypto/networks) pour la liste vivante et faisant autorité.

### Paliers de méthodes

Les méthodes sont regroupées en trois paliers de crédits. Coût total = `baseCredits[chain] × methodTier`.

| Palier | Multiplicateur | Exemples |
|---|---|---|
| **Standard** | 1x | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas` |
| **Advanced** | 2x | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction` |
| **Large** | 4x | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content` |

Liste complète et détail tarifaire dans la [référence API Crypto RPC](/fr/api-reference/endpoint/crypto/rpc).

## Recettes pour agents

Schémas courants pour les agents IA qui doivent lire et écrire on-chain.

### Lire le solde natif d'un portefeuille

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getBalance",
    "params": ["0xYourWalletAddress", "latest"],
    "id": 1
  }'
```

### Lire le solde d'un token ERC-20

Appelez le sélecteur `balanceOf(address)` avec `eth_call`. Le champ `data` est le sélecteur de 4 octets (`0x70a08231`) suivi de l'adresse du portefeuille padée à gauche sur 32 octets. Le plus simple est de laisser une bibliothèque l'encoder :

```typescript
import { encodeFunctionData, parseAbi } from 'viem'

const data = encodeFunctionData({
  abi: parseAbi(['function balanceOf(address) view returns (uint256)']),
  args: ['0xWalletAddress'],
})

const response = await fetch('https://api.venice.ai/api/v1/crypto/rpc/base-mainnet', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.VENICE_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    jsonrpc: '2.0',
    method: 'eth_call',
    params: [{ to: '0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf', data }, 'latest'],
    id: 1,
  }),
})
```

L'adresse de contrat ci-dessus est VVV sur Base. Remplacez-la par n'importe quel contrat ERC-20.

### Envoyer une transaction signée (cycle complet)

Venice ne détient jamais vos clés privées. L'agent recueille les paramètres de transaction via des lectures RPC, signe localement avec une bibliothèque comme [viem](https://viem.sh) ou [ethers](https://docs.ethers.org), puis relaie le hex brut via Venice.

<Steps>
  <Step title="Obtenir le prochain nonce">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionCount","params":["0xAgentWallet","pending"],"id":1}'
    ```
    Utilisez `"pending"` afin que des envois successifs n'entrent pas en collision.
  </Step>
  <Step title="Obtenir le gas price">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_gasPrice","params":[],"id":1}'
    ```
    Pour les chaînes EIP-1559, préférez `eth_feeHistory` pour calculer `maxFeePerGas` et `maxPriorityFeePerGas`.
  </Step>
  <Step title="Estimer le gas">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"from":"0xAgentWallet","to":"0xRecipient","value":"0x0","data":"0x..."}],"id":1}'
    ```
  </Step>
  <Step title="Signer localement">
    ```typescript
    import { privateKeyToAccount } from 'viem/accounts'
    import { base } from 'viem/chains'

    const account = privateKeyToAccount(process.env.AGENT_PRIVATE_KEY)

    const signed = await account.signTransaction({
      chainId: base.id,
      nonce,                  // from step 1
      gas,                    // from step 3
      maxFeePerGas,           // from step 2 (fee history)
      maxPriorityFeePerGas,   // from step 2 (fee history)
      to: '0xRecipient',
      value: 0n,
      data: '0x...',
    })
    ```
  </Step>
  <Step title="Soumettre via Venice">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Idempotency-Key: agent-tx-<id>" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_sendRawTransaction","params":["0xSignedHex"],"id":1}'
    ```
    Définissez toujours `Idempotency-Key` sur les relais pour qu'une perturbation réseau ne puisse pas double-diffuser.
  </Step>
  <Step title="Sonder le reçu">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xTxHash"],"id":1}'
    ```
    Sondez toutes les quelques secondes jusqu'à ce que `result` ne soit pas null. Vérifiez `result.status` (`"0x1"` = succès).
  </Step>
</Steps>

<Note>
Chaque appel à `eth_sendRawTransaction` est journalisé côté serveur avec le hash de la transaction, le réseau, l'identifiant de requête et l'identifiant utilisateur appelant. La charge utile signée elle-même n'est pas conservée. Cette piste d'audit existe pour que les clés compromises utilisées pour des relais illicites puissent être tracées jusqu'au compte responsable.
</Note>

### Batcher plusieurs appels (vérification de portefeuille multi-chaînes)

Envoyez jusqu'à 100 objets JSON-RPC en une seule requête. Chacun est validé et facturé indépendamment.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[
    { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 1 },
    { "jsonrpc": "2.0", "method": "eth_getBalance", "params": ["0xWallet", "latest"], "id": 2 },
    { "jsonrpc": "2.0", "method": "eth_gasPrice", "params": [], "id": 3 }
  ]'
```

Pour des lectures multi-chaînes (un appel par chaîne), émettez des requêtes parallèles vers différents endpoints `{network}`.

### Nouveaux essais sûrs avec idempotence

Définissez l'en-tête `Idempotency-Key` sur n'importe quelle chaîne correspondant à `[A-Za-z0-9_-]{1,255}`. Venice met la réponse en cache pendant 24 heures, indexée sur `(user, key)`. Les rejeux renvoient le résultat mis en cache avec `Idempotent-Replayed: true` et ne facturent rien.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Idempotency-Key: agent-tx-2026-04-21-001" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_sendRawTransaction",
    "params": ["0xSignedRawTxHex"],
    "id": 1
  }'
```

C'est critique pour les relais de transactions où une perturbation réseau pourrait sinon amener votre agent à diffuser la même transaction deux fois.

## Financer la clé API de l'agent

Une fois que l'agent dispose d'une clé API Venice, il a besoin d'un solde dépensable sur le compte sous-jacent avant que les endpoints payants n'acceptent la clé. Il y a deux façons d'y mettre du solde :

| Chemin | Autonome ? | Comment ça marche |
|---|---|---|
| **DIEM depuis le staking VVV** | Oui | Stakez VVV dans le [Smart Contract de staking Venice](https://basescan.org/address/0x321b7ff75154472B18EDb199033fF4D116F340Ff#code) sur Base. L'allocation DIEM quotidienne du portefeuille est proportionnelle à sa part dans le pool de staking. Le compte a besoin d'au moins 0,1 DIEM accumulé avant que du DIEM soit dépensable. Le DIEM se recharge à 00:00 UTC. Pour augmenter la dépense quotidienne, stakez plus de VVV. |
| **Rechargement USD ou crypto via le tableau de bord** | Non (navigateur) | Connectez-vous à [venice.ai](https://venice.ai) avec le même portefeuille (Sign-In-With-Ethereum), puis ajoutez des crédits dans Settings, API. Stripe (carte) et Coinbase (crypto) sont tous deux derrière cette page et nécessitent un navigateur. Les crédits n'expirent jamais. |

Pour un agent qui s'exécute sans surveillance, **le DIEM via staking VVV est aujourd'hui le seul chemin de financement entièrement headless pour une clé API émise**. Si la dépense quotidienne de l'agent dépasse son allocation DIEM, les options réalistes sont : staker plus de VVV ou faire en sorte qu'un opérateur se connecte et recharge en USD ou en crypto.

### Staking VVV autonome et génération de clé

Un agent véritablement autonome peut gérer son propre portefeuille VVV sur Base, le staker et émettre sa propre clé API Venice sans intervention humaine. Le flux complet :

<Steps>
  <Step title="Acquérir VVV et ETH pour le gas">
    Envoyez du VVV au portefeuille de l'agent (ou faites en sorte que l'agent fasse un swap sur [Aerodrome](https://aerodrome.finance) ou [Uniswap](https://app.uniswap.org)), plus une petite quantité d'ETH sur Base pour les deux transactions de staking.
  </Step>
  <Step title="Staker VVV">
    `approve` le contrat de staking sur le token VVV, puis `stake(amount)` sur `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. Le solde sVVV du portefeuille se met à jour atomiquement avec le stake.
  </Step>
  <Step title="Émettre une clé API">
    `GET /api/v1/api_keys/generate_web3_key` renvoie un JWT qui expire 15 minutes après son émission. Signez le token brut avec le portefeuille de staking, puis `POST` l'adresse, la signature et le token. Venice renvoie une clé API liée au compte utilisateur dérivé de ce portefeuille.
  </Step>
</Steps>

L'émission nécessite seulement un solde sVVV non nul, donc 1 VVV staké suffit pour recevoir une clé. **Dépenser** avec la clé est une question séparée, régie par le tableau de financement ci-dessus.

Consultez [Création autonome de clé API par un agent](/fr/guides/getting-started/generating-api-key-agent) pour le walkthrough complet avec code et la référence d'erreurs complète.

## Auth de portefeuille x402 en 30 secondes

Si votre agent dispose déjà d'un portefeuille Base ou Solana, sautez entièrement la clé API. Le SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) gère la signature Sign-In-With-X, les rechargements et le suivi du solde.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // skip if the wallet already has balance

const response = await venice.chat({
  model: 'kimi-k2-6',
  messages: [{ role: 'user', content: 'What is the latest block on Base?' }]
})
```

La même auth de portefeuille fonctionne contre `/crypto/rpc/{network}` pour les lectures et écritures blockchain. Détails complets du protocole dans le [guide x402](/fr/guides/integrations/x402-venice-api).

## Tarification

Le RPC crypto est facturé en crédits Venice. Chaque réponse inclut `X-Venice-RPC-Credits` (crédits facturés) et `X-Venice-RPC-Cost-USD` (coût en dollars) afin que votre agent puisse suivre la dépense par requête.

### Crédits de base par chaîne

| Crédits de base | Chaînes |
|---|---|
| **20** | Ethereum, Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast, Starknet |
| **30** | zkSync Era |

### Exemples de coût

Tarification observée pour les paliers de méthodes standard, advanced et large :

| Appel | Crédits | Coût USD |
|---|---|---|
| `eth_call` sur Ethereum (20 × 1x) | 20 | ~$0.0000140 |
| `trace_transaction` sur Ethereum (20 × 2x) | 40 | ~$0.0000280 |
| `trace_replayTransaction` sur Ethereum (20 × 4x) | 80 | ~$0.0000560 |
| `eth_call` sur zkSync (30 × 1x) | 30 | ~$0.0000210 |

Faites toujours confiance à l'en-tête de réponse `X-Venice-RPC-Cost-USD` pour le coût faisant autorité. Les éléments en erreur dans les requêtes en lot sont facturés à 5 crédits forfaitaires chacun.

### Limites de débit

| Palier | Requêtes par minute |
|---|---|
| Standard | 100 |
| Staff | 1 000 |

Une fois dépassée, l'endpoint renvoie `429` avec les en-têtes de réponse standards `X-RateLimit-*`.

## Gestion des erreurs

Réponses HTTP courantes que votre agent doit gérer :

| Statut | Signification | Que faire |
|---|---|---|
| `400` | Méthode JSON-RPC non prise en charge ou non mappée, ou lot malformé | Vérifiez la méthode contre l'[allowlist](/fr/api-reference/endpoint/crypto/rpc). Le corps d'erreur nomme la méthode incriminée. |
| `400` | Rejeu d'une `Idempotency-Key` avec un corps différent | Utilisez une nouvelle clé pour des requêtes distinctes. |
| `402` | Aucun en-tête d'auth du tout (le corps de la réponse inclut `authOptions` listant les deux chemins d'auth pris en charge), ou plus de crédits avec un en-tête d'auth valide | Si pas d'auth : attachez `Authorization: Bearer ...` ou l'en-tête x402 `X-Sign-In-With-X`. Plus de crédits : avec une clé Bearer, financez le compte (DIEM, USD ou recharge via tableau de bord) ; avec auth x402, appelez `POST /api/v1/x402/top-up` directement. |
| `429` | Limite de débit atteinte (100 req/min standard, 1 000 req/min staff) | Respectez `X-RateLimit-Reset` et faites du backoff. Batchez jusqu'à 100 appels par requête pour amortir la limite. |
| `5xx` | Hoquet du nœud RPC en amont | Réessayez avec la même `Idempotency-Key` pour éviter le double-facturation. |

Les erreurs par élément en lot (par ex. paramètres invalides sur l'un des N appels) reviennent dans une réponse `200 OK` avec un champ JSON-RPC `error` sur l'élément incriminé. Ces éléments sont facturés à 5 crédits forfaitaires chacun.

## Non pris en charge

Ces catégories de méthodes sont intentionnellement rejetées :

- **WebSocket uniquement** (`eth_subscribe`, `eth_unsubscribe`) : le proxy est HTTP uniquement. Pratiquez le polling.
- **Filtres avec état** (`eth_newFilter`, `eth_getFilterChanges`, etc.) : l'état des filtres est lié à un seul backend et se casse sur un proxy à répartition de charge. Utilisez `eth_getLogs` à la place.
- **Méthodes détenant des clés** (`eth_sign`, `eth_accounts`, `eth_mining`) : les fournisseurs hébergés ne détiennent pas les clés utilisateur. Signez côté client et soumettez via `eth_sendRawTransaction`.
- **Méthodes non mappées** : tout ce qui n'est pas dans l'allowlist renvoie `400`. Contactez le support pour demander des ajouts.
