Venice의 핵심 원칙 중 하나는 사용자 프라이버시입니다. 플랫폼의 아키텍처는 이 철학적 원칙에서 흘러나오며, 모든 구성 요소는 이 목표를 염두에 두고 설계되었습니다.

> 합리적인 사용자 프라이버시를 달성하는 유일한 방법은 이러한 정보를 애초에 수집하지 않는 것입니다. 엔지니어링 관점에서는 더 어렵지만, 우리는 이것이 올바른 접근 방식이라고 믿습니다.

Venice API는 Venice 플랫폼과 동일한 백엔드 프라이버시 아키텍처를 복제합니다. 요청은 암호화된 연결을 통해 Venice 프록시를 통과하며, Venice는 일반 추론에 대해 프롬프트 및 응답 콘텐츠를 저장하거나 기록하지 않습니다. 그리고 선택된 각 모델은 런타임 계층에서 네 가지 프라이버시 모드 중 하나(Anonymous, Private, TEE, E2EE)를 추가합니다.

![Venice AI 프라이버시 아키텍처](/images/privacy-architecture.png "Venice AI 프라이버시 아키텍처")

## 프라이버시 아키텍처

Venice 프록시는 모든 프라이버시 모드를 위한 공통 기반입니다. 요청은 HTTPS/TLS를 통해 Venice를 통과하며 Venice가 프롬프트나 응답 콘텐츠를 저장하지 않고 중계됩니다. 선택된 모델의 프라이버시 모드는 제공자 또는 모델 런타임 계층에서 다음에 무엇이 일어나는지 결정합니다.

Venice는 모델 프라이버시를 네 가지 모드로 제공합니다. 이들은 동일한 프록시 기반 위에서 구축되며, 제공자로부터 신원을 숨기는 것부터 프롬프트를 검증된 enclave로 종단간 암호화하는 것까지 점진적으로 더 강력한 보호를 추가합니다.

`/models` 엔드포인트는 각 모델의 프라이버시 수준을 알려줍니다. `anonymized`로 표시된 모델은 Anonymous 모델이고, `private`으로 표시된 모델은 Private 모델입니다. TEE와 E2EE는 `supportsTeeAttestation` 및 `supportsE2EE`와 같이 모델의 capabilities에 별도로 표시됩니다.

## TEE 및 E2EE

TEE 및 E2EE 모델은 Venice의 기본 무콘텐츠 보존 접근 방식 위에 암호화 및 하드웨어 지원 제어를 추가합니다.

<CardGroup cols={2}>
  <Card title="TEE를 사용해야 할 때" icon="shield-halved">
    모델이 인증된 하드웨어 enclave 내에서 실행되기를 원하지만, 클라이언트가 일반 API 요청을 통해 평문 프롬프트를 보낼 수 있는 경우.
  </Card>
  <Card title="E2EE를 사용해야 할 때" icon="lock">
    프롬프트가 클라이언트를 떠나기 전에 암호화되고 검증된 TEE 내부에서만 복호화되기를 원하는 경우.
  </Card>
</CardGroup>

E2EE 흐름은 E2EE 가능 모델과 함께 `/chat/completions`를 사용합니다. 클라이언트는 attestation을 가져오고, nonce 및 enclave 증거를 검증하고, `user` 및 `system` 메시지를 암호화하고, `X-Venice-TEE-*` 헤더를 보내고, 응답을 스트리밍하며, 응답 콘텐츠를 검증/복호화해야 합니다.

E2EE는 또한 웹 검색, 메모리, 요약, 일부 tool 흐름 및 기타 서버 측 처리와 같이 enclave 외부에 평문이 필요한 기능을 비활성화합니다.

## 모델 선택

요청을 보내기 전에 각 모델이 어떤 프라이버시 보호를 지원하는지 확인하려면 `/models`를 사용하세요.

각 모델에는 관련된 두 필드가 있습니다.

- `model_spec.privacy`는 모델의 기본 프라이버시 모드를 알려줍니다.
  - `anonymized`: Venice는 제공자로부터 신원을 숨기지만, 제공자는 여전히 프롬프트를 볼 수 있습니다.
  - `private`: Venice는 zero data retention 인프라를 통해 요청을 라우팅합니다.
- `model_spec.capabilities`는 모델이 더 강력한 보호를 지원하는지 알려줍니다.
  - `supportsTeeAttestation`: 모델은 검증 가능한 Trusted Execution Environment 내에서 실행될 수 있습니다.
  - `supportsE2EE`: 모델은 TEE 내부에서만 복호화되는 클라이언트 암호화 프롬프트를 받을 수 있습니다.

E2EE는 클라이언트 주도 흐름입니다. 애플리케이션은 요청을 암호화하고, attestation을 검증하고, 응답을 검증/복호화해야 합니다. [TEE & E2EE 모델 가이드](/ko/guides/features/tee-e2ee-models)를 참고하세요.

<CodeGroup>
```bash cURL
curl https://api.venice.ai/api/v1/models \
  -H "Authorization: Bearer $API_KEY_VENICE" | \
  jq '.data[] | {
    id,
    privacy: .model_spec.privacy,
    tee: .model_spec.capabilities.supportsTeeAttestation,
    e2ee: .model_spec.capabilities.supportsE2EE
  }'
```
</CodeGroup>

간단한 가이드라인: zero data retention을 위해서는 `private`을, 하드웨어 지원 격리를 위해서는 `tee: true`를, 프롬프트가 클라이언트를 떠나기 전에 암호화되어야 할 때는 `e2ee: true`를 선택하세요.

## 운영 메타데이터

Venice는 인증, 청구, 남용 방지, 안정성, 분석 및 지원에 필요한 메타데이터를 처리할 수 있습니다. 제품 사용 방식에 따라 계정 또는 지갑 식별자, API 키 식별자, 요청 타임스탬프, 선택된 모델, 토큰 수, 청구 금액, 레이트 리밋 상태, 요청 ID, IP 주소, 브라우저 또는 디바이스 정보, 제품 이벤트 로그가 포함될 수 있습니다.

이 메타데이터는 API를 운영하는 데 사용되며 프롬프트 및 응답 콘텐츠와는 별개입니다. 청구 및 사용량 기록은 모델, 엔드포인트, 토큰 수, 타임스탬프 및 계정 식별자와 같은 세부 정보를 추적하며, 프롬프트나 completion을 저장할 필요가 없습니다.
