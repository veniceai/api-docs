Venice supports wallet-native access to paid API routes using Sign-In With X (`X-Sign-In-With-X`) plus an x402-compatible top-up flow (`X-402-Payment`). This guide covers the current Venice flow for both humans and agents.

## TL;DR

- If you already have a Venice account and want the simplest integration, use an API key.
- If you want wallet-native, agent-friendly, or pay-as-you-go access, use x402.
- Venice's discovery source for agents is `https://api.venice.ai/openapi.json`.
- Paid Venice routes can return `402 Payment Required` plus a `PAYMENT-REQUIRED` header.
- The top-up endpoint is `POST /api/v1/x402/top-up`.
- Wallet identity routes are:
  - `GET /api/v1/x402/balance/{walletAddress}`
  - `GET /api/v1/x402/transactions/{walletAddress}`
- Venice runs on Base for x402:
  - Chain ID: `8453`
  - Network: `eip155:8453`
  - Token for top-ups: USDC on Base
- Minimum top-up is currently `$5`.

## When to use x402 vs API keys

| Use case | Recommended auth |
| --- | --- |
| A standard backend or app that already has a Venice account | API key |
| An autonomous agent with its own wallet | x402 |
| A wallet-native integration where you do not want to mint or manage API keys | x402 |
| A human developer experimenting from scripts or Cursor/AgentCash | x402 or API key |

If you are building an agent-first workflow, x402 is usually the better fit. If you are building a conventional app server tied to a Venice account, API keys are often simpler.

## How Venice's x402 flow works

```text  theme={"system"}
Wallet / agent
    |
    | 1) Create a short-lived SIWE message for api.venice.ai on Base
    |    and send it as X-Sign-In-With-X
    v
Venice validates the wallet identity
    |
    | 2) If the wallet has spendable balance, the request continues
    |    and Venice charges linked DIEM-backed or x402-backed balance
    |
    | 3) If balance is missing or too low, Venice returns HTTP 402
    |    with PAYMENT-REQUIRED and machine-readable payment details
    v
Client / agent tops up balance using X-402-Payment
    |
    | 4) Retry the original request
    v
Request succeeds and Venice returns the response
```

<Info>
  Venice uses linked DIEM-backed balance first when available. If there is no spendable DIEM-backed balance, top up with USDC on Base and continue using the same wallet.
</Info>

## What you need

- An EVM wallet on Base
- A small amount of ETH for Base gas
- Either:
  - linked DIEM-backed balance, or
  - USDC on Base for top-ups
- A safe way to manage the wallet's private key
  - environment variables
  - a secret manager
  - an agent wallet managed by AgentCash

<Warning>
  Use a limited-purpose wallet for automation. Do not hand an agent your main treasury wallet.
</Warning>

## Supported Venice routes

Venice currently exposes x402 discovery metadata for the following routes.

### Paid routes with wallet auth

These routes accept either a Bearer API key or `X-Sign-In-With-X`:

- Chat
  - `POST /api/v1/chat/completions`
  - `POST /api/v1/responses`
- Image
  - `POST /api/v1/image/generate`
  - `POST /api/v1/images/generations`
  - `POST /api/v1/image/upscale`
  - `POST /api/v1/image/edit`
  - `POST /api/v1/image/multi-edit`
  - `POST /api/v1/image/background-remove`
- Embeddings
  - `POST /api/v1/embeddings`
- Audio
  - `POST /api/v1/audio/speech`
  - `POST /api/v1/audio/transcriptions`
  - `POST /api/v1/audio/complete`
  - `POST /api/v1/audio/queue`
  - `POST /api/v1/audio/retrieve`
- Video
  - `POST /api/v1/video/complete`
  - `POST /api/v1/video/queue`
  - `POST /api/v1/video/retrieve`
  - `POST /api/v1/video/transcriptions`
### SIWX-only routes

These routes require wallet identity but are not paid routes:

- `GET /api/v1/x402/balance/{walletAddress}`
- `GET /api/v1/x402/transactions/{walletAddress}`

### Top-up route

Top-up is its own x402 payment flow:

- `POST /api/v1/x402/top-up`

This route does not use `X-Sign-In-With-X`. Instead, you call it once to receive payment requirements, then retry with `X-402-Payment`.

## Manual flow

If you are building your own wallet-native client, Venice's x402 flow has three main parts:

1. Generate `X-Sign-In-With-X`
2. Check whether the wallet can currently consume paid routes
3. Top up with `X-402-Payment` when needed

### 1. Create `X-Sign-In-With-X`

Venice expects a Base64-encoded JSON payload containing:

- `address`
- `message` (the signed SIWE message)
- `signature`
- `timestamp`
- `chainId`

Important details for Venice:

- `domain` should be `api.venice.ai`
- `chainId` should be `8453`
- the payload is effectively short-lived; plan on about 5 minutes
- the nonce is single-use
- the payload `timestamp` should match the SIWE `issuedAt` closely
- use a fresh payload for each short request flow

Here is a minimal script that creates a compatible header:

```js  theme={"system"}
import { Wallet } from 'ethers'
import { SiweMessage, generateNonce } from 'siwe'

const wallet = new Wallet(process.env.EVM_PRIVATE_KEY)
const issuedAt = new Date()
const resourceUrl = 'https://api.venice.ai/api/v1/chat/completions'

const siwe = new SiweMessage({
  domain: 'api.venice.ai',
  address: wallet.address,
  statement: 'Sign in to Venice AI',
  uri: resourceUrl,
  version: '1',
  chainId: 8453,
  nonce: generateNonce(),
  issuedAt: issuedAt.toISOString(),
  expirationTime: new Date(issuedAt.getTime() + 5 * 60 * 1000).toISOString(),
})

const message = siwe.prepareMessage()
const signature = await wallet.signMessage(message)

const xSignInWithX = Buffer.from(
  JSON.stringify({
    address: wallet.address,
    message,
    signature,
    timestamp: issuedAt.getTime(),
    chainId: 8453,
  }),
  'utf8',
).toString('base64')

console.log(xSignInWithX)
```

<Info>
  Venice validates the SIWE domain and other auth fields. Using the exact route URL as the SIWE `uri` is the safest and most future-proof approach.
</Info>

### 2. Check balance first

The balance endpoint is the quickest way to see whether a wallet can currently use paid Venice routes.

```bash  theme={"system"}
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

Example response:

```json  theme={"system"}
{
  "success": true,
  "data": {
    "walletAddress": "0xyour_wallet_address",
    "balanceUsd": 12.5,
    "canConsume": true,
    "minimumTopUpUsd": 5,
    "suggestedTopUpUsd": 10,
    "diemBalanceUsd": 5.25
  }
}
```

Useful fields:

- `canConsume`: whether the wallet can currently use x402-backed Venice routes
- `balanceUsd`: current x402-backed prepaid balance
- `minimumTopUpUsd`: current minimum top-up
- `suggestedTopUpUsd`: a convenient next top-up amount
- `diemBalanceUsd`: linked DIEM-backed balance, when available

### 3. Call a paid route

Once the wallet can consume, send the same header to a paid endpoint:

```bash  theme={"system"}
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header "Content-Type: application/json" \
  --header "X-Sign-In-With-X: $X402_AUTH" \
  --data '{
    "model": "kimi-k2-5",
    "messages": [
      {
        "role": "user",
        "content": "Hello from an x402-authenticated wallet."
      }
    ]
  }'
```

Successful x402-backed responses can include `X-Balance-Remaining`, which reports the remaining wallet-backed balance after the request.

### 4. Top up with `X-402-Payment`

If the wallet cannot currently consume, use the top-up endpoint.

#### Ask Venice for payment requirements

Send the top-up request without `X-402-Payment`:

```bash  theme={"system"}
curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up
```

Venice will respond with:

- `402 Payment Required`
- a `PAYMENT-REQUIRED` header
- a machine-readable JSON body describing the payment requirement

For x402-compatible clients, the header is the canonical machine-readable signal. The body is intended to be useful for both people and agents.

#### Build the payment header

Use the `accepts` data from the response to create the payment authorization. The easiest options are:

- AgentCash
- your own x402-compatible payment client
- a dedicated Venice wallet client, if you already have one integrated

Do not hardcode payment parameters when you can avoid it. Use the top-up response as the source of truth.

#### Retry the same request with `X-402-Payment`

```bash  theme={"system"}
export X402_PAYMENT=YOUR_SIGNED_PAYMENT_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up \
  --header "X-402-Payment: $X402_PAYMENT"
```

Example success response:

```json  theme={"system"}
{
  "success": true,
  "data": {
    "walletAddress": "0xyour_wallet_address",
    "amountCredited": 10,
    "newBalance": 22.5,
    "paymentId": "payment_01HZY8M4W4Y6QZ8B6Q4P0V3J2K"
  }
}
```

After a successful top-up, call the balance endpoint again or retry the original paid request.

### 5. Inspect transaction history

You can inspect the x402 wallet ledger with:

```bash  theme={"system"}
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/transactions/$WALLET_ADDRESS?limit=10&offset=0" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

The ledger includes entries such as `TOP_UP`, `CHARGE`, and `REFUND`.

## Discovery and agent interoperability

Venice publishes an x402-friendly discovery contract at:

```text  theme={"system"}
https://api.venice.ai/openapi.json
```

That discovery contract is intended for:

- AgentCash
- x402scan
- coding agents that need invocable route metadata
- custom clients that want machine-readable pricing and schema information

If you are building an agent that discovers APIs dynamically, use this OpenAPI document instead of hardcoding the route inventory.

## Common failures and how to fix them

| Symptom | Likely cause | What to do |
| --- | --- | --- |
| `401` with an x402 sign-in error code | malformed SIWE payload, expired payload, wrong domain, invalid signature, reused nonce | rebuild `X-Sign-In-With-X` from scratch with a fresh nonce and timestamp |
| `403` on balance or transactions | the authenticated wallet does not match the `walletAddress` path parameter | sign with the same wallet you are querying |
| `402` on a paid route | wallet auth worked, but the wallet does not have enough spendable balance | call the balance endpoint, then top up and retry |
| `402` on the first call to `/api/v1/x402/top-up` | expected | build `X-402-Payment` from the returned requirement and retry |
| `400` / invalid payment header | malformed or mismatched `X-402-Payment` | regenerate the payment header from the current payment requirement |
| `429` | concurrency or rate limit pressure | retry with backoff |

<Warning>
  The most common manual-auth mistake is reusing an old SIWE payload. Generate a fresh payload with a new nonce and timestamp instead.
</Warning>

## Security checklist

- use a dedicated wallet for automation
- store private keys in environment variables or a secret manager
- generate fresh SIWE payloads frequently
- never paste private keys directly into prompts
- avoid hardcoding payment requirements when the top-up endpoint can supply them dynamically
