Venice supports wallet-native access to paid API routes using Sign-In With X (`X-Sign-In-With-X`) plus an x402-compatible top-up flow (`X-402-Payment`). This guide covers the current Venice flow for both humans and agents.

<CardGroup cols={3}>
  <Card title="Wallet Auth" icon="wallet">
    Authenticate with a short-lived SIWE payload in `X-Sign-In-With-X`.
  </Card>

  <Card title="Prepaid Balance" icon="coins">
    Venice charges linked DIEM-backed balance first, then x402-backed USDC credits.
  </Card>

  <Card title="Agent Friendly" icon="robot">
    AgentCash can discover Venice automatically and handle SIWX plus payment retries for you.
  </Card>
</CardGroup>

If you already have a Venice account and just want the simplest integration, use an API key. If you want wallet-native access, autonomous-agent workflows, or a cleaner pay-as-you-go story, use x402.

Venice's machine-readable discovery source is `https://api.venice.ai/openapi.json`. Paid routes can return `402 Payment Required` with a `PAYMENT-REQUIRED` header, and Venice currently runs the x402 flow on Base using USDC for top-ups. The current minimum top-up is `$5`.

## The Venice x402 flow

<Steps>
  <Step title="Start with spendable balance">
    For most new wallets, the first real step is topping up. The only exception is a wallet that is already linked to a Venice account with spendable DIEM-backed balance. In that case, you can skip straight to auth and inference.

    Venice always prefers spendable DIEM-backed balance first. If there is no spendable balance, top up with USDC on Base and continue using the same wallet.
  </Step>

  <Step title="Generate a fresh wallet auth header">
    Venice uses `X-Sign-In-With-X` for wallet identity. Under the hood, this is a short-lived SIWE payload wrapped in JSON and Base64-encoded.

    Venice validates the SIWE domain, signature, nonce, chain, and timestamp, so treat the header as short-lived and generate it fresh for each short request flow.
  </Step>

  <Step title="Call the paid Venice route">
    Send the signed wallet header with the request. If the wallet has enough spendable balance, Venice processes the request and charges the linked balance automatically.
  </Step>

  <Step title="If Venice returns 402, top up and retry">
    When a wallet does not have enough spendable balance, Venice returns `402 Payment Required` plus a `PAYMENT-REQUIRED` header and machine-readable payment details. Use those details to create `X-402-Payment`, top up the wallet, and retry the original request.
  </Step>

  <Step title="Check balance and transaction history when needed">
    Venice also exposes wallet-only routes for balance and transaction history. These are useful when an app or agent wants to check whether it can spend before making a paid request.
  </Step>
</Steps>

## What you need

You need an EVM wallet on Base, a small amount of ETH for Base gas, and either spendable DIEM-backed balance or USDC on Base for top-ups. For automation, keep the private key in environment variables, a secret manager, or an agent wallet managed by AgentCash.

<Warning>
  Use a limited-purpose wallet for automation. Do not hand an agent your main treasury wallet.
</Warning>

## Supported Venice routes

Venice currently exposes x402 discovery metadata for the following routes.

| Group | Routes | Notes |
| --- | --- | --- |
| Chat | `POST /api/v1/chat/completions`, `POST /api/v1/responses` | Paid routes with wallet auth or API key |
| Image | `POST /api/v1/image/generate`, `POST /api/v1/images/generations`, `POST /api/v1/image/upscale`, `POST /api/v1/image/edit`, `POST /api/v1/image/multi-edit`, `POST /api/v1/image/background-remove` | Paid routes with wallet auth or API key |
| Embeddings | `POST /api/v1/embeddings` | Paid route with wallet auth or API key |
| Audio | `POST /api/v1/audio/speech`, `POST /api/v1/audio/transcriptions`, `POST /api/v1/audio/complete`, `POST /api/v1/audio/queue`, `POST /api/v1/audio/retrieve` | Paid routes with wallet auth or API key |
| Video | `POST /api/v1/video/complete`, `POST /api/v1/video/queue`, `POST /api/v1/video/retrieve`, `POST /api/v1/video/transcriptions` | Paid routes with wallet auth or API key |
| Top-up | `POST /api/v1/x402/top-up` | Starts with `402`, then retry with `X-402-Payment` |
| SIWX-only | `GET /api/v1/x402/balance/{walletAddress}`, `GET /api/v1/x402/transactions/{walletAddress}` | Wallet identity only, not paid routes |

## Manual flow

If you are building your own wallet-native client, this is the cleanest manual path.

<Steps>
  <Step title="If the wallet is new, top up first">
    If the wallet is not already linked to a Venice account with spendable DIEM-backed balance, assume you will need to top up before the first successful paid request.

    The top-up endpoint is:

    ```text  theme={"system"}
    POST https://api.venice.ai/api/v1/x402/top-up
    ```

    Call it once without `X-402-Payment` to receive the payment requirement, then retry with a signed `X-402-Payment` header.
  </Step>

  <Step title="Create X-Sign-In-With-X">
    Venice expects a Base64-encoded JSON payload containing `address`, `message`, `signature`, `timestamp`, and `chainId`.

    For Venice specifically:

    | Field | Expected value |
    | --- | --- |
    | `domain` | `api.venice.ai` |
    | `chainId` | `8453` |
    | lifetime | treat it as short-lived; about 5 minutes is a safe assumption |
    | `nonce` | single-use |
    | `timestamp` | should closely match the SIWE `issuedAt` |

    Here is a minimal script that creates a compatible header:

```js  theme={"system"}
import { Wallet } from 'ethers'
import { SiweMessage, generateNonce } from 'siwe'

const wallet = new Wallet(process.env.EVM_PRIVATE_KEY)
const issuedAt = new Date()
const resourceUrl = 'https://api.venice.ai/api/v1/chat/completions'

const siwe = new SiweMessage({
  domain: 'api.venice.ai',
  address: wallet.address,
  statement: 'Sign in to Venice AI',
  uri: resourceUrl,
  version: '1',
  chainId: 8453,
  nonce: generateNonce(),
  issuedAt: issuedAt.toISOString(),
  expirationTime: new Date(issuedAt.getTime() + 5 * 60 * 1000).toISOString(),
})

const message = siwe.prepareMessage()
const signature = await wallet.signMessage(message)

const xSignInWithX = Buffer.from(
  JSON.stringify({
    address: wallet.address,
    message,
    signature,
    timestamp: issuedAt.getTime(),
    chainId: 8453,
  }),
  'utf8',
).toString('base64')

console.log(xSignInWithX)
```

    <Info>
      Venice validates the SIWE domain and other auth fields. Using the exact route URL as the SIWE `uri` is the safest and most future-proof approach.
    </Info>
  </Step>

  <Step title="Check whether the wallet can spend">
    The balance endpoint is the quickest way to see whether a wallet can currently use paid Venice routes.

```bash  theme={"system"}
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

Example response:

```json  theme={"system"}
{
  "success": true,
  "data": {
    "walletAddress": "0xyour_wallet_address",
    "balanceUsd": 12.5,
    "canConsume": true,
    "minimumTopUpUsd": 5,
    "suggestedTopUpUsd": 10,
    "diemBalanceUsd": 5.25
  }
}
```

Useful fields:

- `canConsume`: whether the wallet can currently use x402-backed Venice routes
- `balanceUsd`: current x402-backed prepaid balance
- `minimumTopUpUsd`: current minimum top-up
- `suggestedTopUpUsd`: a convenient next top-up amount
- `diemBalanceUsd`: linked DIEM-backed balance, when available
  </Step>

  <Step title="Call a paid route with the wallet header">
    Once the wallet can consume, send the same header to a paid endpoint:

```bash  theme={"system"}
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header "Content-Type: application/json" \
  --header "X-Sign-In-With-X: $X402_AUTH" \
  --data '{
    "model": "kimi-k2-5",
    "messages": [
      {
        "role": "user",
        "content": "Hello from an x402-authenticated wallet."
      }
    ]
  }'
```

Successful x402-backed responses can include `X-Balance-Remaining`, which reports the remaining wallet-backed balance after the request.
  </Step>

  <Step title="If Venice returns 402, top up and retry">
    If the wallet cannot currently consume, or if Venice returns `402`, use the top-up endpoint.

    Send the top-up request without `X-402-Payment`:

```bash  theme={"system"}
curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up
```

    Venice will respond with:

    - `402 Payment Required`
    - a `PAYMENT-REQUIRED` header
    - a machine-readable JSON body describing the payment requirement

    For x402-compatible clients, the header is the canonical machine-readable signal. The body is intended to be useful for both people and agents.

    Use the `accepts` data from the response to create the payment authorization. The easiest options are AgentCash, your own x402-compatible payment client, or a dedicated Venice wallet client if you already have one.

    Do not hardcode payment parameters when you can avoid it. Use the top-up response as the source of truth.

    Retry the same request with `X-402-Payment`:

```bash  theme={"system"}
export X402_PAYMENT=YOUR_SIGNED_PAYMENT_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up \
  --header "X-402-Payment: $X402_PAYMENT"
```

Example success response:

```json  theme={"system"}
{
  "success": true,
  "data": {
    "walletAddress": "0xyour_wallet_address",
    "amountCredited": 10,
    "newBalance": 22.5,
    "paymentId": "payment_01HZY8M4W4Y6QZ8B6Q4P0V3J2K"
  }
}
```

    After a successful top-up, call the balance endpoint again or retry the original paid request.
  </Step>

  <Step title="Inspect transaction history when you need to debug or monitor spend">
    You can inspect the x402 wallet ledger with:

```bash  theme={"system"}
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/transactions/$WALLET_ADDRESS?limit=10&offset=0" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

The ledger includes entries such as `TOP_UP`, `CHARGE`, and `REFUND`.
  </Step>
</Steps>

## Discovery and agent interoperability

Venice publishes an x402-friendly discovery contract at:

```text  theme={"system"}
https://api.venice.ai/openapi.json
```

That discovery contract is intended for:

- AgentCash
- x402scan
- coding agents that need invocable route metadata
- custom clients that want machine-readable pricing and schema information

If you are building an agent that discovers APIs dynamically, use this OpenAPI document instead of hardcoding the route inventory.

## Common failures and how to fix them

| Symptom | Likely cause | What to do |
| --- | --- | --- |
| `401` with an x402 sign-in error code | malformed SIWE payload, expired payload, wrong domain, invalid signature, reused nonce | rebuild `X-Sign-In-With-X` from scratch with a fresh nonce and timestamp |
| `403` on balance or transactions | the authenticated wallet does not match the `walletAddress` path parameter | sign with the same wallet you are querying |
| `402` on a paid route | wallet auth worked, but the wallet does not have enough spendable balance | call the balance endpoint, then top up and retry |
| `402` on the first call to `/api/v1/x402/top-up` | expected | build `X-402-Payment` from the returned requirement and retry |
| `400` / invalid payment header | malformed or mismatched `X-402-Payment` | regenerate the payment header from the current payment requirement |
| `429` | concurrency or rate limit pressure | retry with backoff |

<Warning>
  The most common manual-auth mistake is reusing an old SIWE payload. Generate a fresh payload with a new nonce and timestamp instead.
</Warning>

## Security checklist

- use a dedicated wallet for automation
- store private keys in environment variables or a secret manager
- generate fresh SIWE payloads frequently
- never paste private keys directly into prompts
- avoid hardcoding payment requirements when the top-up endpoint can supply them dynamically
