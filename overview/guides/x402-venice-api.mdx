This guide is about **what agents can do with X402 on Venice**.

With this flow, an agent can:

- authenticate to Venice with a wallet instead of a standard API key
- spend **DIEM-backed balance** from that wallet automatically
- call Venice endpoints like `chat/completions` directly
- check available balance before starting longer agent tasks
- operate with a compact auth contract that can be handed to Claw-style agents

<Info>
This page intentionally stays focused on **agent usage on Venice**, not general SIWE or wallet implementation patterns.

The example request below uses `e2ee-glm-5` because that is the model in the source prompt. If you need Venice's full encrypted transport flow for E2EE models, also see the [TEE & E2EE Models guide](/overview/guides/tee-e2ee-models).
</Info>

## What X402 Lets Agents Do on Venice

For agents, X402 on Venice is mainly useful for these workflows:

### 1. Pay for inference directly from a wallet

An agent can authenticate with wallet-based X402 credentials and then call Venice directly. This is useful when you want the agent to operate from a funded wallet instead of a traditional API key.

### 2. Check balance before running a task

Before a long reasoning loop or workflow, an agent can check:

```bash
GET https://api.venice.ai/api/v1/x402/balance/<address>
```

That lets the agent decide whether it has enough spendable balance before making inference calls.

### 3. Call `chat/completions`

An agent can use the same wallet-auth header to call:

```bash
POST https://api.venice.ai/api/v1/chat/completions
```

This is the most important Venice endpoint for agent behavior, because it powers:

- conversational agents
- task-planning agents
- tool-using agents that route through chat completions
- Claw-style coding or execution agents that rely on a chat endpoint

### 4. Reuse one short-lived auth window

The signed auth material expires after 10 minutes, which is usually enough for an agent to:

- check balance
- make one or more calls
- refresh auth again when needed

## The Agent Contract

Agents do not need a long explanation of X402 internals. They mostly need this contract:

1. Build a SIWE message
2. Sign it with your wallet using **EIP-191**
3. Wrap the SIWE message and signature in JSON
4. Base64-encode that JSON
5. Send it as the `X-Sign-In-With-X` header

Use these exact values:

| Field | Required Value |
|---|---|
| Domain | `outerface.venice.ai` |
| URI | `https://outerface.venice.ai` |
| Chain ID | `8453` |
| Expiration | 10 minutes after creation |

The header payload should look like this before base64 encoding:

```json
{
  "address": "0xYOUR_ADDRESS",
  "message": "<SIWE message text>",
  "signature": "0x...",
  "timestamp": 1712345678901,
  "chainId": 8453
}
```

## Venice Endpoints Agents Should Care About

### 1. `POST /chat/completions`

```bash
POST https://api.venice.ai/api/v1/chat/completions
```

Headers:

- `Content-Type: application/json`
- `X-Sign-In-With-X: <base64-encoded-json>`

Example body:

```json
{
  "model": "e2ee-glm-5",
  "messages": [
    {
      "role": "user",
      "content": "Hello"
    }
  ]
}
```

This is the main endpoint agents will use for:

- chat
- planning
- code generation
- task execution
- structured prompting
- model-driven tool orchestration

### 2. `GET /x402/balance/<address>`

```bash
GET https://api.venice.ai/api/v1/x402/balance/<address>
```

Use the same `X-Sign-In-With-X` header here.

This endpoint is useful when agents need to:

- verify spendable balance before starting
- fail early instead of halfway through a workflow
- report billing state back to a user or orchestrator

## How Payment Works for Agents

Venice uses the wallet's available payment source automatically:

1. **DIEM staked on the wallet** is used first
2. If no DIEM is available, fund the wallet with **USDC on Base**

<Warning>
If the wallet has no spendable DIEM-backed balance, top it up with **USDC on Base**. The guidance provided with this flow says the minimum top-up is **$5**.
</Warning>

## Typical Agent Flow on Venice

For most agents, the flow is:

1. Generate a short-lived wallet-auth header
2. Optionally call `/x402/balance/<address>`
3. Call `/chat/completions`
4. Refresh the header when the 10-minute window expires

That is usually enough for:

- a Claw agent making a single task request
- a supervisor agent checking whether work can be funded
- an automation loop that periodically refreshes wallet auth

## Prompt You Can Hand to a Claw Agent

If you want a Claw agent to use X402 on Venice, this is the shortest useful prompt:

```text
Use X402 with Venice.

Authenticate with wallet auth by building a SIWE message with:
- domain: outerface.venice.ai
- uri: https://outerface.venice.ai
- chainId: 8453
- expiration: 10 minutes

Sign it with EIP-191, then base64-encode this JSON into X-Sign-In-With-X:
{"address":"0xYOUR_ADDRESS","message":"<SIWE message text>","signature":"0x...","timestamp":<unix ms>,"chainId":8453}

Use that header to:
1. optionally GET https://api.venice.ai/api/v1/x402/balance/<address>
2. POST https://api.venice.ai/api/v1/chat/completions

For chat, send:
{"model":"e2ee-glm-5","messages":[{"role":"user","content":"Hello"}]}

DIEM staked on the wallet should be used automatically for payment.
If there is no DIEM-backed balance, use USDC on Base.
```

<Warning>
For safety, give the agent the private key via environment variables or a secret manager, not by pasting it directly into the prompt.
</Warning>

## What to Tell Agents Explicitly

When an agent is using X402 with Venice, the most useful instructions are:

- use `X-Sign-In-With-X`, not `Authorization: Bearer`
- use `chainId: 8453`
- use `outerface.venice.ai`
- use `https://outerface.venice.ai`
- treat auth as short-lived and refresh after 10 minutes
- check `/x402/balance/<address>` before expensive loops
- use `chat/completions` as the primary inference endpoint

That gives the agent enough to operate without turning the guide into a generic implementation manual.

## Minimal Request Shape

Agents generally only need to know the request shape, not a full SDK walkthrough.

### Auth header payload before base64 encoding

```json
{
  "address": "0xYOUR_ADDRESS",
  "message": "<SIWE message text>",
  "signature": "0x...",
  "timestamp": 1712345678901,
  "chainId": 8453
}
```

### Chat request

```http
POST https://api.venice.ai/api/v1/chat/completions
Content-Type: application/json
X-Sign-In-With-X: <base64>
```

```json
{
  "model": "e2ee-glm-5",
  "messages": [
    {
      "role": "user",
      "content": "Hello"
    }
  ]
}
```

### Balance request

```http
GET https://api.venice.ai/api/v1/x402/balance/<address>
X-Sign-In-With-X: <base64>
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="My agent gets auth failures">
    Make sure the agent is using the Venice-specific SIWE values:

    - domain is `outerface.venice.ai`
    - URI is `https://outerface.venice.ai`
    - chain ID is `8453`
    - the message has not expired
    - the `address` matches the signing wallet
    - the JSON is base64-encoded and sent in `X-Sign-In-With-X`
  </Accordion>

  <Accordion title="My agent has no spendable balance">
    DIEM staked on the wallet is supposed to be used automatically. If none is available, fund the wallet with USDC on Base and retry. The guidance for this flow says to use at least $5.
  </Accordion>

  <Accordion title="What should my agent call first">
    If the task may involve multiple model calls or a longer loop, start with `/x402/balance/<address>`. If it is a one-shot task, the agent can go directly to `/chat/completions`.
  </Accordion>

  <Accordion title="I am using e2ee-glm-5 and need the full E2EE transport flow">
    This page covers wallet auth only. For the additional TEE attestation, encryption, and streaming details used by Venice E2EE models, see [TEE & E2EE Models](/overview/guides/tee-e2ee-models).
  </Accordion>
</AccordionGroup>
