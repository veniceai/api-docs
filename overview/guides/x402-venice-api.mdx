Venice supports wallet-based access to paid API routes using `X-Sign-In-With-X` and an x402-compatible top-up flow using `X-402-Payment`.

<CardGroup cols={3}>
  <Card title="Wallet Auth" icon="wallet">

  </Card>

  <Card title="Top Up on Base" icon="coins">
    If the wallet does not already have spendable balance, top up with USDC on Base.
  </Card>

  <Card title="DIEM First" icon="sparkles">

  </Card>
</CardGroup>

## Before you start

To use Venice with x402 you need:

- an EVM wallet on Base
- a small amount of ETH on Base for gas
- either spendable DIEM-backed balance or USDC on Base for top-ups

<Warning>
  Use a dedicated wallet for automation. Do not use your main treasury wallet.
</Warning>

## The Venice flow

<Steps>
  <Step title="Start with spendable balance">
    A new wallet will usually need a top-up before the first paid request. The exception is a wallet that is already linked to a Venice account with spendable DIEM-backed balance.

    Venice always spends linked DIEM-backed balance first. If there is no spendable balance, top up with USDC on Base and continue using the same wallet.
  </Step>

  <Step title="Create a fresh X-Sign-In-With-X header">
    Venice expects a short-lived SIWE payload wrapped in Base64 JSON. Use `api.venice.ai` as the SIWE domain, `8453` as the chain ID, and generate a fresh nonce and timestamp for each short request flow.
  </Step>

  <Step title="Call the paid route">
    Send the signed wallet header with the request. If the wallet can spend, Venice processes the request and charges the linked balance automatically.
  </Step>

  <Step title="If Venice returns 402, top up and retry">
    When the wallet does not have enough spendable balance, Venice returns `402 Payment Required` with a `PAYMENT-REQUIRED` header and machine-readable payment details. Use those details to create `X-402-Payment`, top up the wallet, and retry the original request.
  </Step>

  <Step title="Use balance and transactions when needed">
    Venice also exposes wallet-only routes for balance and transaction history. Use them before a paid request or when you want to inspect spend.
  </Step>
</Steps>

## Supported routes

Venice currently exposes x402 metadata for these route groups:

- Chat: `POST /api/v1/chat/completions`, `POST /api/v1/responses`
- Image: `POST /api/v1/image/generate`, `POST /api/v1/images/generations`, `POST /api/v1/image/upscale`, `POST /api/v1/image/edit`, `POST /api/v1/image/multi-edit`, `POST /api/v1/image/background-remove`
- Embeddings: `POST /api/v1/embeddings`
- Audio: `POST /api/v1/audio/speech`, `POST /api/v1/audio/transcriptions`, `POST /api/v1/audio/complete`, `POST /api/v1/audio/queue`, `POST /api/v1/audio/retrieve`
- Video: `POST /api/v1/video/complete`, `POST /api/v1/video/queue`, `POST /api/v1/video/retrieve`, `POST /api/v1/video/transcriptions`
- Top-up: `POST /api/v1/x402/top-up`
- Wallet-only: `GET /api/v1/x402/balance/{walletAddress}`, `GET /api/v1/x402/transactions/{walletAddress}`

## Manual flow

<Steps>
  <Step title="Top up first if the wallet is new">
    If the wallet is not already linked to a Venice account with spendable DIEM-backed balance, start here:

    ```bash  theme={"system"}
    curl --request POST \
      --url https://api.venice.ai/api/v1/x402/top-up
    ```

    The first call returns `402`, a `PAYMENT-REQUIRED` header, and an `accepts` array describing how to pay. Use that response as the source of truth for the top-up, then retry the same route with `X-402-Payment`.
  </Step>

  <Step title="Create X-Sign-In-With-X">
    Venice expects a Base64-encoded JSON payload with `address`, `message`, `signature`, `timestamp`, and `chainId`.

    For Venice specifically:

    - `domain`: `api.venice.ai`
    - `chainId`: `8453`
    - generate a fresh nonce and timestamp
    - do not reuse old auth headers

    Minimal example:

    ```js  theme={"system"}
    import { Wallet } from 'ethers'
    import { SiweMessage, generateNonce } from 'siwe'

    const wallet = new Wallet(process.env.EVM_PRIVATE_KEY)
    const now = new Date()
    const resourceUrl = 'https://api.venice.ai/api/v1/chat/completions'

    const siwe = new SiweMessage({
      domain: 'api.venice.ai',
      address: wallet.address,
      statement: 'Sign in to Venice AI',
      uri: resourceUrl,
      version: '1',
      chainId: 8453,
      nonce: generateNonce(),
      issuedAt: now.toISOString(),
      expirationTime: new Date(now.getTime() + 5 * 60 * 1000).toISOString(),
    })

    const message = siwe.prepareMessage()
    const signature = await wallet.signMessage(message)

    const xSignInWithX = Buffer.from(
      JSON.stringify({
        address: wallet.address,
        message,
        signature,
        timestamp: now.getTime(),
        chainId: 8453,
      }),
      'utf8',
    ).toString('base64')

    console.log(xSignInWithX)
    ```
  </Step>

  <Step title="Check whether the wallet can spend">
    This is the quickest way to see whether the wallet can currently use paid Venice routes:

    ```bash  theme={"system"}
    export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
    export X402_AUTH=YOUR_BASE64_SIWX_HEADER

    curl --request GET \
      --url "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
      --header "X-Sign-In-With-X: $X402_AUTH"
    ```

    The response tells you:

    - `canConsume`
    - `balanceUsd`
    - `minimumTopUpUsd`
    - `suggestedTopUpUsd`
    - optional `diemBalanceUsd`
  </Step>

  <Step title="Call a paid route">
    Once the wallet can spend, send the wallet header with the request:

    ```bash  theme={"system"}
    export X402_AUTH=YOUR_BASE64_SIWX_HEADER

    curl --request POST \
      --url https://api.venice.ai/api/v1/chat/completions \
      --header "Content-Type: application/json" \
      --header "X-Sign-In-With-X: $X402_AUTH" \
      --data '{
        "model": "kimi-k2-5",
        "messages": [
          {
            "role": "user",
            "content": "Hello from an x402-authenticated wallet."
          }
        ]
      }'
    ```

    Successful wallet-backed responses can include `X-Balance-Remaining`.
  </Step>

  <Step title="Inspect transactions when you need them">
    To review the wallet ledger:

    ```bash  theme={"system"}
    export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS
    export X402_AUTH=YOUR_BASE64_SIWX_HEADER

    curl --request GET \
      --url "https://api.venice.ai/api/v1/x402/transactions/$WALLET_ADDRESS?limit=10&offset=0" \
      --header "X-Sign-In-With-X: $X402_AUTH"
    ```

    The ledger includes entries such as `TOP_UP`, `CHARGE`, and `REFUND`.
  </Step>
</Steps>

## Common failures

| Symptom | Likely cause | What to do |
| --- | --- | --- |
| `401` on a wallet-auth request | malformed or expired SIWE payload | rebuild `X-Sign-In-With-X` with a fresh nonce and timestamp |
| `403` on balance or transactions | wallet mismatch | make sure the authenticated wallet matches the `walletAddress` path parameter |
| `402` on a paid route | not enough spendable balance | top up and retry |
| `402` on the first call to `/api/v1/x402/top-up` | expected | use the returned payment requirement to build `X-402-Payment` |
| `400` on top-up | malformed payment header | rebuild the top-up request from the latest 402 response |

## For agents

The Venice flow for agents is the same as for any other client:

- top up first unless the wallet already has spendable DIEM-backed balance
- generate a fresh `X-Sign-In-With-X` header for each short request flow
- use the top-up `402` response as the source of truth for payment details
- keep private keys in environment variables or a secret manager
