This guide covers the wallet-auth flow described by your colleague: **sign a SIWE message, base64-encode it into `X-Sign-In-With-X`, and call the Venice API directly with your wallet**.

In this flow:

- you do **not** send a standard Venice API key
- you authenticate with a Base wallet
- Venice uses **DIEM staked on that wallet** automatically for payment
- if you do not have DIEM available, you can top up with **USDC on Base** instead

<Info>
This page focuses on the **wallet auth header** used to call Venice directly.

The example request below targets `e2ee-glm-5` because that is the model in the source prompt. If you are implementing Venice's full E2EE transport flow, also see the [TEE & E2EE Models guide](/overview/guides/tee-e2ee-models).
</Info>

## What You Send

For every wallet-authenticated request:

1. Build a SIWE message
2. Sign it with your wallet using **EIP-191**
3. Wrap the SIWE message and signature in JSON
4. Base64-encode that JSON
5. Send it as the `X-Sign-In-With-X` header

Use these fixed SIWE values:

| Field | Required Value |
|---|---|
| Domain | `outerface.venice.ai` |
| URI | `https://outerface.venice.ai` |
| Chain ID | `8453` |
| Expiration | 10 minutes after creation |

The header payload should look like this before base64 encoding:

```json
{
  "address": "0xYOUR_ADDRESS",
  "message": "<SIWE message text>",
  "signature": "0x...",
  "timestamp": 1712345678901,
  "chainId": 8453
}
```

<Note>
The `message` field should contain the SIWE message text, and `signature` should contain the wallet signature for that message.
</Note>

## Venice Endpoints in This Flow

### Chat completions

```bash
POST https://api.venice.ai/api/v1/chat/completions
```

Headers:

- `Content-Type: application/json`
- `X-Sign-In-With-X: <base64-encoded-json>`

Body:

```json
{
  "model": "e2ee-glm-5",
  "messages": [
    {
      "role": "user",
      "content": "Hello"
    }
  ]
}
```

### Balance

```bash
GET https://api.venice.ai/api/v1/x402/balance/<address>
```

Use the **same** `X-Sign-In-With-X` header on the balance request.

---

## Payment Model

Venice uses the wallet's available payment source automatically:

1. **DIEM staked on the wallet** is used first
2. If no DIEM is available, fund the wallet with **USDC on Base**

<Warning>
If the wallet has no spendable DIEM-backed balance, top it up with **USDC on Base**. The guidance provided with this flow says the minimum top-up is **$5**.
</Warning>

## JavaScript Example

This example:

- builds the required SIWE message
- signs it with a Base wallet
- creates `X-Sign-In-With-X`
- calls `chat/completions`
- checks balance with the same auth header

### Install dependency

```bash
npm install viem
```

### Example script

```javascript
import { randomBytes } from 'node:crypto'
import { privateKeyToAccount } from 'viem/accounts'

const DOMAIN = 'outerface.venice.ai'
const URI = 'https://outerface.venice.ai'
const CHAIN_ID = 8453
const BASE_URL = 'https://api.venice.ai/api/v1'

function buildSiweMessage(address) {
  const issuedAt = new Date().toISOString()
  const expirationTime = new Date(Date.now() + 10 * 60 * 1000).toISOString()
  const nonce = randomBytes(16).toString('hex')

  return `${DOMAIN} wants you to sign in with your Ethereum account:
${address}

Use Venice API with your wallet.

URI: ${URI}
Version: 1
Chain ID: ${CHAIN_ID}
Nonce: ${nonce}
Issued At: ${issuedAt}
Expiration Time: ${expirationTime}`
}

async function buildWalletHeader(privateKey) {
  const account = privateKeyToAccount(privateKey)
  const message = buildSiweMessage(account.address)
  const signature = await account.signMessage({ message })

  const payload = {
    address: account.address,
    message,
    signature,
    timestamp: Date.now(),
    chainId: CHAIN_ID,
  }

  return {
    address: account.address,
    headerValue: Buffer.from(JSON.stringify(payload), 'utf8').toString('base64'),
  }
}

async function main() {
  const privateKey = process.env.WALLET_PRIVATE_KEY
  if (!privateKey) {
    throw new Error('Set WALLET_PRIVATE_KEY=0x...')
  }

  const { address, headerValue } = await buildWalletHeader(privateKey)

  const chatResponse = await fetch(`${BASE_URL}/chat/completions`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Sign-In-With-X': headerValue,
    },
    body: JSON.stringify({
      model: 'e2ee-glm-5',
      messages: [
        { role: 'user', content: 'Hello' },
      ],
    }),
  })

  const chatData = await chatResponse.json()
  console.log('Chat response:', chatData)

  const balanceResponse = await fetch(`${BASE_URL}/x402/balance/${address}`, {
    headers: {
      'X-Sign-In-With-X': headerValue,
    },
  })

  const balanceData = await balanceResponse.json()
  console.log('Balance:', balanceData)
}

main().catch(console.error)
```

## cURL Usage

Because the auth header must be generated from a signed SIWE message, the usual pattern is:

1. generate `X-Sign-In-With-X` in a short script
2. reuse that value in `curl`

Example chat request once you have the header:

```bash
curl https://api.venice.ai/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-Sign-In-With-X: $X_SIGN_IN_WITH_X" \
  -d '{
    "model": "e2ee-glm-5",
    "messages": [
      {
        "role": "user",
        "content": "Hello"
      }
    ]
  }'
```

Example balance request:

```bash
curl "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
  -H "X-Sign-In-With-X: $X_SIGN_IN_WITH_X"
```

## Prompt You Can Hand to a Claw Agent

If you want a Claw agent to make the request for you, this is the compact version to give it:

```text
Use the Venice API with wallet auth.

Build a SIWE message for:
- domain: outerface.venice.ai
- uri: https://outerface.venice.ai
- chainId: 8453
- expiration: 10 minutes

Sign the SIWE message with EIP-191 using my wallet private key.

Then base64-encode this JSON into the X-Sign-In-With-X header:
{"address":"0xYOUR_ADDRESS","message":"<SIWE message text>","signature":"0x...","timestamp":<unix ms>,"chainId":8453}

Call:
POST https://api.venice.ai/api/v1/chat/completions

Headers:
- Content-Type: application/json
- X-Sign-In-With-X: <base64>

Body:
{"model":"e2ee-glm-5","messages":[{"role":"user","content":"Hello"}]}

Also support:
GET https://api.venice.ai/api/v1/x402/balance/<address>
with the same X-Sign-In-With-X header.

DIEM staked on the wallet should be used automatically for payment.
If there is no DIEM-backed balance, use USDC on Base.
```

<Warning>
For safety, give the agent the private key via environment variables or a secret manager, not by pasting it directly into the prompt.
</Warning>

## SIWE Message Requirements

The SIWE message should be standard EIP-4361 text with the required Venice-specific values.

Example template:

```text
outerface.venice.ai wants you to sign in with your Ethereum account:
0xYOUR_ADDRESS

Use Venice API with your wallet.

URI: https://outerface.venice.ai
Version: 1
Chain ID: 8453
Nonce: RANDOM_NONCE
Issued At: 2026-04-02T12:00:00.000Z
Expiration Time: 2026-04-02T12:10:00.000Z
```

Keep the message fresh. The supplied flow expects it to expire after **10 minutes**.

## Best Practices

<AccordionGroup>
  <Accordion title="Create a fresh header for each request window">
    Because the SIWE message expires after 10 minutes, generate a fresh message and signature whenever your auth window is stale or you get an auth-related failure.
  </Accordion>

  <Accordion title="Use Base mainnet settings exactly">
    This flow depends on Base mainnet values: `chainId: 8453`, `outerface.venice.ai`, and `https://outerface.venice.ai`. Small mismatches can invalidate the auth header.
  </Accordion>

  <Accordion title="Keep the private key outside your source code">
    Store the wallet private key in environment variables, a secret manager, or your agent runtime's secret store. Never hardcode it in scripts or commit it to git.
  </Accordion>

  <Accordion title="Check balance before automation loops">
    If an agent is going to make repeated requests, hit `/x402/balance/<address>` first so you can fail early when the wallet has no DIEM-backed balance or USDC funding.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Authentication fails">
    Double-check all required SIWE fields:

    - domain is `outerface.venice.ai`
    - URI is `https://outerface.venice.ai`
    - chain ID is `8453`
    - the message has not expired
    - the `address` matches the signing wallet
    - the JSON is base64-encoded and sent in `X-Sign-In-With-X`
  </Accordion>

  <Accordion title="Balance is empty or payment does not go through">
    DIEM staked on the wallet is supposed to be used automatically. If none is available, fund the wallet with USDC on Base and retry. The guidance for this flow says to use at least $5.
  </Accordion>

  <Accordion title="I want to use this with a Claw agent">
    Give the agent the short prompt from the section above, and provide the wallet secret through the agent runtime rather than the prompt itself.
  </Accordion>

  <Accordion title="I am using e2ee-glm-5 and need full encrypted transport">
    This page covers wallet auth only. For the additional TEE attestation, encryption, and streaming details used by Venice E2EE models, see [TEE & E2EE Models](/overview/guides/tee-e2ee-models).
  </Accordion>
</AccordionGroup>
