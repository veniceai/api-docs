يستطيع وكيل ذكاء اصطناعي يتحكم بمحفظة على Base إنشاء مفتاح Venice API الخاص به دون أي تدخل بشري. يحصل الوكيل على VVV، ويرهنه، ويوقّع رمز تحقق قصير العمر يصدره Venice، ثم يُعيد إرسال الرمز الموقّع لاستلام مفتاح API جديد مرتبط بمحفظة الرهن.

يشرح هذا الدليل التدفق الكامل من البداية إلى النهاية ويغطي خيارات التمويل لدفع تكلفة الاستدلال فعلياً بعد إنشاء المفتاح.

## المتطلبات الأساسية

* محفظة EVM على Base يتحكم بها الوكيل (المفتاح الخاص في متغير بيئة أو مدير أسرار).
* كمية صغيرة من ETH على Base للغاز (الرهن عبارة عن معاملتين: `approve` ثم `stake`).
* أي مقدار غير صفري من VVV للرهن. تشترط نقطة نهاية الإنشاء فقط أن يكون لدى المحفظة رصيد sVVV غير صفري، لذا فإن 1 VVV يكفي لإنشاء مفتاح. راجع [دفع تكلفة الاستدلال](#paying-for-inference) لما تحتاج إليه فعلياً لاستدعاء نقاط النهاية المدفوعة.

<Tip>
  استخدم محفظة وكيل مخصصة بدلاً من محفظة الخزينة. المفتاح الخاص للمحفظة يوقّع كل طلب رمز Venice، لذا يجب أن يكون نطاق الضرر المحتمل صغيراً.
</Tip>

## الخطوات

<Steps>
  <Step title="احصل على VVV">
    أرسل VVV إلى محفظة الوكيل، أو اجعل الوكيل يبادل عبر DEX مثل [Aerodrome](https://aerodrome.finance/swap?from=eth\&to=0xacfe6019ed1a7dc6f7b508c02d1b04ec88cc21bf\&chain0=8453\&chain1=8453) أو [Uniswap](https://app.uniswap.org/swap?chain=base\&inputCurrency=NATIVE\&outputCurrency=0xacfe6019ed1a7dc6f7b508c02d1b04ec88cc21bf).

    عقد رمز VVV على Base: `0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf`
  </Step>

  <Step title="ارهن VVV لدى Venice">
    ارهن VVV في [عقد الرهن الذكي لـ Venice](https://basescan.org/address/0x321b7ff75154472b18edb199033ff4d116f340ff#code) على `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. هذا عبارة عن معاملتين:

    1. `approve(spender, amount)` على رمز VVV، حيث `spender` هو عقد الرهن.
    2. `stake(amount)` على عقد الرهن.

    <Frame as="div">
      <img src="/images/guides/SC-Stake.png" alt="رهن العقد الذكي" />
    </Frame>

    عند تأكيد المعاملة الثانية، ينخفض رصيد VVV في المحفظة ويزداد رصيد sVVV بالقيمة نفسها. تقرأ نقطة نهاية الإنشاء رصيد sVVV لتأكيد أن المحفظة قد رهنت.
  </Step>

  <Step title="اطلب رمز تحقق">
    استدعِ `GET /api/v1/api_keys/generate_web3_key` للحصول على رمز قصير العمر موقّع من Venice. هذه النقطة لا تتطلب مصادقة.

    ```bash
    curl --request GET \
      --url https://api.venice.ai/api/v1/api_keys/generate_web3_key
    ```

    تتضمن الاستجابة حقل `token`. ينتهي الرمز بعد 15 دقيقة من الإصدار، لذا قم بتوقيعه وإرساله قبل ذلك بكثير.
  </Step>

  <Step title="وقّع الرمز بمحفظة الرهن">
    وقّع سلسلة الرمز الخام بالمحفظة التي تحمل VVV المرهون. هذا توقيع `personal_sign` قياسي على بايتات الرمز. كل من `ethers.Wallet.signMessage(token)` و`account.signMessage({ message: token })` في `viem` ينتجان التوقيع الصحيح.
  </Step>

  <Step title="أنشئ مفتاح API">
    أرسل `POST` يحوي العنوان والتوقيع والرمز إلى نفس نقطة النهاية، مع نوع المفتاح الذي تريد.

    ```bash
    curl --request POST \
      --url https://api.venice.ai/api/v1/api_keys/generate_web3_key \
      --header 'Content-Type: application/json' \
      --data '{
        "address": "<wallet address>",
        "signature": "<signed token>",
        "token": "<unsigned token>",
        "apiKeyType": "INFERENCE",
        "description": "Agent key minted on <date>"
      }'
    ```

    الحقول المطلوبة: `address`، `signature`، `token`، `apiKeyType` (`INFERENCE` أو `ADMIN`).

    الحقول الاختيارية: `description`، `expiresAt`، `consumptionLimit` (يحدد إجمالي الإنفاق على هذا المفتاح، مقوَّماً بـ `usd` أو `vcu` أو `diem`).

    عند النجاح تتضمن الاستجابة سلسلة `apiKey` المُنشأة. خزّنها في مخزن أسرار الوكيل واستخدمها كرمز Bearer عادي (`Authorization: Bearer <key>`).
  </Step>
</Steps>

## مثال شامل من البداية إلى النهاية

يستخدم المثال أدناه محفظة حقيقية من متغير بيئة بدلاً من محفظة مولّدة عشوائياً. المحفظة العشوائية ليس لديها VVV مرهون وسيُرفض الإنشاء بخطأ `Wallet has no staked VVV on Base`.

```typescript
import { ethers } from "ethers"

const wallet = new ethers.Wallet(process.env.WALLET_PRIVATE_KEY!)
const address = wallet.address

const tokenResponse = await fetch("https://api.venice.ai/api/v1/api_keys/generate_web3_key")
const { data: { token } } = await tokenResponse.json()

const signature = await wallet.signMessage(token)

const mintResponse = await fetch("https://api.venice.ai/api/v1/api_keys/generate_web3_key", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    address,
    signature,
    token,
    apiKeyType: "INFERENCE",
    description: "Agent key",
  }),
})

const result = await mintResponse.json()
if (!mintResponse.ok) {
  throw new Error(`Mint failed: ${result.error}`)
}

console.log("Minted key:", result.data.apiKey)
```

## مرجع الأخطاء

تُعيد نقطة النهاية رسائل أخطاء محددة وقابلة للتنفيذ. قم بمعالجتها في الوكيل ليقرر ما إذا كان سيُعيد المحاولة أو يطلب رمزاً جديداً أو يتوقف.

| الحالة | تحتوي رسالة الخطأ | ماذا تعني | ما العمل |
| --- | --- | --- | --- |
| `400` | `Invalid wallet address` | حقل `address` ليس عنوان EVM صالحاً. | أصلح العنوان وأعد الإرسال. |
| `400` | `JWT has expired` | انتهت صلاحية رمز التحقق قبل أن توقّعه وترسله. | اطلب رمزاً جديداً، ووقّعه، وأرسله فوراً. |
| `400` | `JWT signature is invalid` | الرمز لم يوقَّع من Venice (على الأرجح تم العبث به أو تلفيقه). | استخدم دائماً رمزاً جديداً من نقطة نهاية `GET`. |
| `400` | `JWT claims are invalid` | مُصدر الرمز أو جمهوره لا يطابق ما تتوقعه Venice. | استخدم الرمز غير المعدّل المُعاد من نقطة نهاية `GET`. |
| `400` | `JWT is malformed` | `token` المُرسل ليس JWT. | تأكد من أنك ترسل سلسلة `token` تماماً كما أعادتها نقطة نهاية `GET`. |
| `400` | `Wallet signature does not match` | `signature` لا يطابق `address` للرمز المعطى. | وقّع بايتات الرمز الخام بالمحفظة التي تملك `address`. |
| `400` | `Could not verify wallet signature` | فشل استدعاء RPC للتحقق من التوقيع (عابر). | أعد المحاولة مع تراجع زمني. |
| `400` | `Wallet has no staked VVV on Base` | رصيد sVVV للمحفظة صفر. | ارهن VVV أولاً، ثم أعد المحاولة. |

## دفع تكلفة الاستدلال

إنشاء مفتاح والقدرة على استدعاء نقاط النهاية المدفوعة به أمران منفصلان. مفتاح جديد يصادَق به بشكل صحيح ولكن لا يمكنه استدعاء نقاط النهاية المدفوعة (مثل `/chat/completions`) حتى يكون لحساب المحفظة رصيد قابل للإنفاق.

يمكن للمفتاح المُنشأ الإنفاق من حساب المستخدم وفق ترتيب الأولوية التالي: DIEM، ثم الأرصدة المُجمَّعة، ثم USD.

| مصدر التمويل | ذاتي؟ | كيف |
| --- | --- | --- |
| **DIEM من رهن VVV** | نعم | تخصيص DIEM اليومي للمحفظة يتناسب مع حصتها من مجمّع الرهن. يحتاج الحساب إلى 0.1 DIEM مرهون على الأقل ليكون أي DIEM قابلاً للإنفاق. الحصص الأكبر تحصل على DIEM يومي أعلى بالتناسب، ويُحدَّث كل حقبة (00:00 UTC). |
| **USD عبر Stripe** | لا (عبر المتصفح) | سجّل الدخول إلى venice.ai بنفس المحفظة (Sign-In-With-Ethereum). تعثر لوحة التحكم على سجل المستخدم الموجود. أضف الأرصدة في الإعدادات، API. |
| **اشتراك Coinbase للعملات الرقمية** | لا (عبر المتصفح) | تسجيل الدخول بنفس المحفظة، ثم الاشتراك عبر لوحة التحكم. يُعيد التدفق التوجيه إلى Coinbase Commerce للدفع الفعلي، لذا لا يمكن تشغيله من سكربت. |
| **Coinbase onramp** | لا (عبر المتصفح) | تسجيل الدخول بنفس المحفظة، ثم استخدام أداة onramp في لوحة التحكم. مستضافة على واجهة Coinbase. |

إذا احتاج الوكيل إلى مسار تمويل بدون رأس وعملات رقمية بالكامل، فإن أنظف الخيارات هي:

1. **رهن المزيد من VVV** ليغطي تخصيص DIEM اليومي إنفاق الوكيل. سيلتقط المفتاح المُنشأ ذلك تلقائياً.
2. **استخدام [تدفق محفظة x402](/ar/guides/integrations/x402-venice-api) بدلاً من مفتاح API.** مع x402 يوقّع الوكيل رسالة Sign-In-With-X لكل طلب، ويعيد التعبئة مباشرة بـ USDC على Base أو Solana عبر `POST /api/v1/x402/top-up`، ويدفع لكل طلب. رصيد x402 من USDC مرتبط بالمحفظة، لا بالمستخدم، لذا لا يظهر كرصيد لمفتاح Bearer المُنشأ، لكنه يتيح للمحفظة نفسها دفع تكلفة الاستدلال برمجياً.
