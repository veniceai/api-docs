يقوم التخزين المؤقت للطلبات بتخزين رموز الإدخال المعالجة بحيث يمكن للطلبات اللاحقة ذات البادئات المتطابقة إعادة استخدامها بدلاً من إعادة معالجتها. وهذا يقلل من زمن الاستجابة (بما يصل إلى 80% للطلبات الطويلة) ومن التكاليف (بخصم يصل إلى 90% على الرموز المخزّنة).

تتولى Venice التخزين المؤقت تلقائياً للنماذج المدعومة، لكن فهم كيفية تنفيذ كل مزوّد للتخزين المؤقت يساعدك على تعظيم معدلات استخدام التخزين المؤقت وتقليل التكاليف.

## كيف يعمل التخزين المؤقت

يعمل التخزين المؤقت على **مطابقة البادئات**: يخزّن النظام الرموز المعالجة ويعيد استخدامها عندما تبدأ الطلبات اللاحقة بالمحتوى نفسه.

تخيّل روبوت محادثة بـ system prompt مكوّن من 2,000 رمز:

<Steps>
  <Step title="الطلب 1">
    system prompt (2,000 رمز) + رسالة المستخدم (50 رمزاً)
    
    **معالجة**: 2,050 رمزاً · **من التخزين المؤقت**: 0 رمز
    
    تتم كتابة البادئة إلى التخزين المؤقت.
  </Step>
  <Step title="الطلب 2">
    system prompt (2,000 رمز) + رسالة المستخدم (80 رمزاً)
    
    **معالجة**: 80 رمزاً · **من التخزين المؤقت**: 2,000 رمز
  </Step>
  <Step title="الطلب 3">
    system prompt (2,000 رمز) + رسالة المستخدم (120 رمزاً)
    
    **معالجة**: 120 رمزاً · **من التخزين المؤقت**: 2,000 رمز
  </Step>
</Steps>

**الإجمالي بدون تخزين مؤقت**: 2,050 + 2,080 + 2,120 = 6,250 رمزاً بالسعر الكامل

**الإجمالي مع التخزين المؤقت**: 2,050 + 80 + 120 = 2,250 رمزاً بالسعر الكامل، و4,000 رمز بالسعر المخفّض

<Warning>
يعمل التخزين المؤقت على **البادئة** فقط. أي تغيير في بداية طلبك يُبطل التخزين المؤقت لكل ما يليه. ضع دائماً المحتوى الثابت (system prompt، المستندات، الأمثلة) قبل المحتوى الديناميكي (رسائل المستخدم).
</Warning>

## السلوك الخاص بكل مزوّد

تعمل Venice على توحيد التخزين المؤقت عبر المزوّدين. بالنسبة لمعظم النماذج، يكون التخزين المؤقت تلقائياً. أرسل طلباتك فقط وافحص الاستجابة بحثاً عن إحصاءات التخزين المؤقت. تتطلب **Claude** علامات تخزين مؤقت صريحة على مستوى البروتوكول، لكن Venice تضيفها تلقائياً لـ system prompts وسجل المحادثة.

سلوك التخزين المؤقت يتحكم به في النهاية كل مزوّد وقد يتغير، لذا راجع وثائق المزوّد للاطلاع على أحدث التفاصيل.

### Claude Opus 4.5 (Anthropic)

يتطلب Claude نقاط فاصلة صريحة للتخزين المؤقت على مستوى البروتوكول. تتولى Venice ذلك تلقائياً:

- **System prompts** يتم تخزينها مؤقتاً تلقائياً
- **سجل المحادثة** يتم تخزينه مؤقتاً عبر وضع نقطة فاصلة على رسالة المستخدم قبل الأخيرة

هذا يعني أن سجل محادثتك يُقرأ من التخزين المؤقت، ولا تتم معالجة سوى آخر دور كإدخال جديد:

| الدور | رموز الطلب | قراءة التخزين المؤقت | كتابة التخزين المؤقت | التوفير |
|------|---------------|------------|-------------|---------|
| 1 | 10,979 | 0 | 10,938 | كتابة أولى |
| 2 | 11,031 | 10,938 | 31 | 99.7% مخزّن |
| 3 | 11,062 | 10,969 | 52 | 99.5% مخزّن |

**تفاصيل إضافية:**

- **حتى 4 نقاط فاصلة لكل طلب**: يستخدم النظام أطول بادئة مطابقة
- **مفتاح التخزين المؤقت دقيق بالبايت**: تغييرات المسافات البيضاء، أو ترميزات الصور المختلفة، أو إعادة ترتيب الأدوات تكسر إصابات التخزين المؤقت
- **حدود معدل تراعي التخزين المؤقت**: الرموز المخزّنة لا تُحتسب ضمن حد ITPM، مما يتيح إنتاجية فعّالة أعلى
- **علاوة 25% على الكتابة**: الطلب الأول يكلّف أكثر، لكن مع توفير 90% على القراءات اللاحقة

#### التحكم اليدوي بالتخزين المؤقت

للحالات الخاصة مثل تخزين مستند كبير مؤقتاً في الدور الأول، يمكنك إضافة نقاط فاصلة صريحة:

```json
{
  "messages": [
    {
      "role": "system",
      "content": [{
        "type": "text",
        "text": "You are a legal assistant...",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    {
      "role": "user", 
      "content": [{
        "type": "text",
        "text": "[Long contract document...]",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    { "role": "assistant", "content": "I've reviewed the contract." },
    { "role": "user", "content": "What are the termination clauses?" }
  ]
}
```

هذا يضمن أن كلاً من system prompt والمستند يتم تخزينهما مؤقتاً منذ الطلب الأول. بالنسبة للمحادثات النموذجية، لست بحاجة إلى علامات يدوية.

### جميع النماذج الأخرى

التخزين المؤقت **تلقائي**. لا حاجة لمعاملات خاصة. تأكد فقط من أن طلباتك تتجاوز ~1,024 رمزاً واستخدم `prompt_cache_key` للتوجيه المتسق.

## معاملات الطلب

| المعامل | النوع | النماذج | الوصف |
|-----------|------|--------|-------------|
| `prompt_cache_key` | string | الكل | تلميح توجيه لتقارب التخزين المؤقت. من المرجح أن تصل الطلبات التي تحمل المفتاح نفسه إلى الخادم نفسه مع تخزين مؤقت دافئ. |
| `cache_control` | object | Claude | يحدّد كتل المحتوى للتخزين المؤقت. راجع قسم Claude Opus 4.5. |

### prompt_cache_key

للمحادثات أو سير العمل الوكيلية (agentic)، استخدم `prompt_cache_key` متسقاً لتحسين معدلات إصابة التخزين المؤقت:

```json
{
  "model": "claude-opus-4-5",
  "prompt_cache_key": "session-abc-123",
  "messages": [...]
}
```

هذا يوجّه الطلبات إلى الخوادم التي من المحتمل أن يكون لديها سياقك مخزّناً مؤقتاً بالفعل. استخدم معرف جلسة أو معرف محادثة أو معرف مستخدم كمفتاح.

## حقول الاستجابة

يتضمن كائن `usage` في الاستجابة إحصاءات التخزين المؤقت:

```json
{
  "usage": {
    "prompt_tokens": 5500,
    "completion_tokens": 200,
    "total_tokens": 5700,
    "prompt_tokens_details": {
      "cached_tokens": 5000,
      "cache_creation_input_tokens": 0
    }
  }
}
```

| الحقل | الوصف |
|-------|-------------|
| `prompt_tokens` | إجمالي رموز الإدخال في الطلب |
| `prompt_tokens_details.cached_tokens` | الرموز المقدَّمة من التخزين المؤقت (تُفوتر بالسعر المخفّض) |
| `prompt_tokens_details.cache_creation_input_tokens` | الرموز المكتوبة إلى التخزين المؤقت (قد تتحمل رسوماً إضافية على Claude) |

**تفصيل الفوترة** (باستخدام Claude Opus 4.5 كمثال):
- 5000 رمز مخزّن × \$0.60/1M = \$0.003
- 500 رمز غير مخزّن × \$6.00/1M = \$0.003
- الإجمالي: \$0.006 (مقابل \$0.033 بدون تخزين مؤقت، توفير 82%)

## أفضل الممارسات

### نظِّم الطلبات للتخزين المؤقت

ضع المحتوى الثابت في البداية، والمحتوى الديناميكي في النهاية.

**هيكل جيد**

| الموضع | المحتوى | مخزّن؟ |
|----------|---------|---------|
| 1 | تعليمات النظام | نعم |
| 2 | المستندات المرجعية | نعم |
| 3 | أمثلة few-shot | نعم |
| 4 | استعلام المستخدم | لا |

**هيكل سيئ**

| الموضع | المحتوى | مخزّن؟ |
|----------|---------|---------|
| 1 | الطابع الزمني الحالي | لا (يُبطل كل ما بعده) |
| 2 | تعليمات النظام | لا |
| 3 | استعلام المستخدم | لا |

### حافظ على تطابق البادئات بالبايت

تُحسب مفاتيح التخزين المؤقت من تسلسلات البايت الدقيقة. حتى الفروق التافهة تكسر إصابات التخزين المؤقت:

- اختلافات في المسافات البيضاء أو الأسطر الجديدة
- الطوابع الزمنية أو معرفات الطلبات في الطلبات
- ترتيب عشوائي لأمثلة few-shot
- تنسيقات مختلفة للمحتوى نفسه

### الالتزام بحدود الرموز الدنيا

إذا كانت طلباتك أقل من الحد الأدنى (عادةً 1,024 رمزاً)، فلن يتم تفعيل التخزين المؤقت. للطلبات الصغيرة، فكّر في:

- إضافة المزيد من السياق أو الأمثلة للوصول إلى الحد
- تجميع عدة طلبات صغيرة في طلبات مُجمَّعة
- قبول أن التخزين المؤقت لن ينطبق على الاستعلامات البسيطة

### استخدم prompt_cache_key للمحادثات

للمحادثات المستمرة، اضبط `prompt_cache_key` متسقاً:

```json
// Turn 1
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turn 2
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turn 3
{ "prompt_cache_key": "conv-xyz", "messages": [...] }
```

هذا يحسّن احتمال وصول جميع الأدوار إلى الخادم نفسه مع تخزين مؤقت دافئ.

### راقب أداء التخزين المؤقت

تتبّع هذه المقاييس:
- **معدل إصابة التخزين المؤقت**: `cached_tokens / prompt_tokens`
- **توفير التكاليف**: قارن التكلفة الفعلية مقابل التكلفة بدون تخزين مؤقت
- **تقليل زمن الاستجابة**: زمن الوصول إلى أول رمز مع إصابات التخزين المؤقت وبدونها

إذا كانت `cached_tokens` تساوي 0 باستمرار:
1. قد تكون الطلبات أقل من الحد الأدنى للرموز
2. قد تتغير الطلبات بين الاستدعاءات
3. قد تصل الطلبات إلى خوادم مختلفة (استخدم `prompt_cache_key`)
4. قد ينتهي التخزين المؤقت (الطلبات غير متكررة بما يكفي)

### راعِ اقتصاديات التخزين المؤقت

**علاوة كتابة التخزين المؤقت على Claude Opus 4.5**: الطلب الأول يكلف 25% أكثر، لكنه يوفر 90% على القراءات اللاحقة.

| السيناريو | هل علاوة كتابة التخزين المؤقت تستحق؟ |
|----------|-------------------------------|
| طلب واحد بهذا الطلب | لا (تدفع 25% أكثر، دون فائدة) |
| طلبان أو أكثر بنفس البادئة | نعم (نقطة التعادل عند الطلب الثاني) |
| طلبات تتغير بسرعة | لا (تكاليف كتابة ثابتة) |
| system prompt مستقر، استعلامات كثيرة | نعم (موزّعة على قراءات كثيرة) |

## عمر التخزين المؤقت

ينتهي عمر التخزينات المؤقتة بعد فترة من عدم النشاط (عادةً 5-10 دقائق). هذا يعني:

| نمط حركة المرور | فائدة التخزين المؤقت |
|-----------------|-----------------|
| طلبات مستمرة (فجوات < 5 دقائق) | عالية: يبقى التخزين المؤقت دافئاً |
| حركة مرور متفجرة (فجوات > 10 دقائق) | محدودة: ينتهي التخزين المؤقت بين الدفعات |
| طلبات متفرقة (تفصل بينها ساعات) | لا شيء: التخزين المؤقت بارد دائماً |

## التخزين المؤقت مع الأدوات والوظائف

يمكن تخزين تعريفات الوظائف مؤقتاً مع system prompts:

```json
{
  "model": "claude-opus-4-5",
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_database",
        "description": "Search the product database",
        "parameters": { ... }
      }
    }
  ],
  "messages": [
    {
      "role": "system",
      "content": [
        {
          "type": "text",
          "text": "You are a shopping assistant...",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    ...
  ]
}
```

تصبح تعريفات الأدوات جزءاً من البادئة المخزّنة. إذا كان لديك أدوات كثيرة، يمكن أن يقلل ذلك من تكاليف الطلب الواحد بشكل ملحوظ.

## التخزين المؤقت مع الصور والمستندات

بالنسبة لنماذج الرؤية، يمكن تضمين الصور في المحتوى المخزّن:

```json
{
  "model": "claude-opus-4-5",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": { "url": "data:image/png;base64,..." }
        },
        {
          "type": "text",
          "text": "This is the floor plan. I'll ask several questions about it.",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    {
      "role": "assistant",
      "content": "I can see the floor plan. What would you like to know?"
    },
    {
      "role": "user",
      "content": "How many bedrooms are there?"
    }
  ]
}
```

تُخزَّن الصورة والسياق الأولي مؤقتاً، بحيث لا تُعالَج الأسئلة المتابعة عن الصورة نفسها من جديد.
