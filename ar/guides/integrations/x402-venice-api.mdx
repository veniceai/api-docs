يتيح X402 استخدام مسارات Venice API المدفوعة من خلال مصادقة محفظة والاحتفاظ برصيد USDC مدفوع مسبقًا. لا حاجة إلى مفتاح API أو حساب. وقّع رسالة، اشحن على Base أو Solana، ثم استدعِ أي مسار مدعوم.

## ما هو X402؟

[X402](https://www.x402.org/) معيار دفع مفتوح يتيح للتطبيقات والوكلاء الدفع مقابل الخدمات برمجيًا باستخدام العملات المشفّرة. يطبّق Venice X402 بحيث تستطيع المحافظ المصادقة والدفع مقابل الاستدلال مباشرة بـ USDC على Base أو Solana.

## المتطلبات

- محفظة على Base أو Solana
- عملة أصلية للغاز على السلسلة المختارة، مثل ETH على Base أو SOL على Solana
- USDC على السلسلة المختارة (أو رصيد قائم مدعوم بـ DIEM من حساب Venice مرتبط)

<Tip>
  فكّر في استخدام محفظة مخصّصة للأتمتة بدلًا من محفظة خزينتك الرئيسية.
</Tip>

## بداية سريعة

توفّر حزمة SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) مساعدين لمصادقة المحفظة والشحن وتتبّع الرصيد.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // تخطَّ هذا إذا كانت المحفظة لديها رصيد قابل للإنفاق

const response = await venice.chat({
  model: 'kimi-k2-5',
  messages: [{ role: 'user', content: 'Hello!' }]
})
```

يُنشئ العميل رأس `X-Sign-In-With-X` جديدًا لكل طلب ويتعقّب الرصيد تلقائيًا من رؤوس الاستجابة `X-Balance-Remaining`.

### مع الأدوات المتوافقة مع OpenAI

إذا كنت تستخدم أداة تقبل `fetch` مخصصًا، استخدم `createAuthFetch` لإضافة مصادقة المحفظة إلى أي طلب:

```typescript
import { createAuthFetch } from 'venice-x402-client'

const authFetch = createAuthFetch(process.env.WALLET_KEY)
```

### المساعدون المتاحون

تتضمّن الـ SDK مساعدين من الدرجة الأولى لمعظم مسارات Venice x402 الشائعة. لأي شيء آخر غير مغطّى، استخدم `request()` أو `createAuthFetch()` مباشرة.

| الفئة | الطرق |
| --- | --- |
| الدردشة | `chat()`, `chatStream()` |
| الاستجابات | `responses.create()`, `responses.stream()` |
| الصور | `images.generate()`, `images.generations()`, `images.upscale()`, `images.edit()`, `images.multiEdit()`, `images.backgroundRemove()` |
| الصوت | `audio.speech()`, `audio.transcribe()`, `audio.queue()`, `audio.retrieve()`, `audio.complete()` |
| الفيديو | `video.queue()`, `video.retrieve()`, `video.generate()`, `video.complete()`, `video.transcribe()` |
| Embeddings | `embeddings()` |
| النماذج | `models()` |
| المحفظة | `getBalance()`, `getTransactions()`, `topUp()` |

## التدفّق اليدوي

إذا لم تستخدم الـ SDK أو احتجت إلى فهم البروتوكول الأساسي، إليك التدفّق خطوة بخطوة. لمحفظة جديدة، افترض أنك بحاجة إلى الشحن أولًا ما لم يكن لديها رصيد DIEM قابل للإنفاق.

### الخطوة 1: إنشاء رأس X-Sign-In-With-X

يتوقّع Venice حمولة JSON مُرمَّزة Base64 تحتوي على رسالة Sign-In-With-X موقَّعة. تُوقّع محافظ EVM رسالة SIWE وفق EIP-4361 على Base. وتُوقّع محافظ Solana رسالة SIWX الخاصة بـ Solana بـ Ed25519. ولّد nonce وطابعًا زمنيًا جديدًا لكل تدفّق طلب.

لـ Solana، اضبط `chainId` على `solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp`، وأدرج `type: "ed25519"` في حمولة JSON المُرمَّزة، وقدّم توقيع Ed25519 بصيغة base58 أو base64. تبدأ الرسالة الموقَّعة بـ `<domain> wants you to sign in with your Solana account:` متبوعةً بعنوان المحفظة والحقول القياسية `URI` و`Version` و`Chain ID` و`Nonce` و`Issued At` و`Expiration Time` الاختياري.

```typescript
import { Wallet } from 'ethers'
import { SiweMessage, generateNonce } from 'siwe'

const wallet = new Wallet(process.env.EVM_PRIVATE_KEY)
const now = new Date()
const resourceUrl = 'https://api.venice.ai/api/v1/chat/completions'

const siwe = new SiweMessage({
  domain: 'api.venice.ai',
  address: wallet.address,
  statement: 'Sign in to Venice AI',
  uri: resourceUrl,
  version: '1',
  chainId: 8453,
  nonce: generateNonce(),
  issuedAt: now.toISOString(),
  expirationTime: new Date(now.getTime() + 5 * 60 * 1000).toISOString(),
})

const message = siwe.prepareMessage()
const signature = await wallet.signMessage(message)

const xSignInWithX = Buffer.from(
  JSON.stringify({
    address: wallet.address,
    message,
    signature,
    timestamp: now.getTime(),
    chainId: 8453,
  }),
  'utf8',
).toString('base64')

console.log(xSignInWithX)
```

### الخطوة 2: التحقق من الرصيد

قبل إجراء طلب مدفوع، تحقق من أن المحفظة لديها رصيد قابل للإنفاق:

```bash
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS_OR_SOLANA_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/balance/$WALLET_ADDRESS" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

تتضمّن الاستجابة:

- `canConsume`: ما إذا كانت المحفظة تستطيع إجراء طلبات مدفوعة
- `balanceUsd`: الرصيد القابل للإنفاق الحالي
- `minimumTopUpUsd` و`suggestedTopUpUsd`: إرشاد للشحن
- `diemBalanceUsd`: الرصيد المدعوم بـ DIEM، إن وُجد

يقبل معامل المسار `walletAddress` إمّا عنوان EVM (`0x...`) أو عنوان Solana بصيغة base58.

### الخطوة 3: الشحن

اشحن بـ USDC على Base أو Solana:

```bash
curl --request POST \
  --url https://api.venice.ai/api/v1/x402/top-up
```

يُعيد الاستدعاء الأول `402 Payment Required` مع رأس `PAYMENT-REQUIRED` يحتوي على مصفوفة `accepts`. يصف كل مدخل خيار دفع مقبول واحدًا، يتضمن `network` و`asset` و`payTo` و`amount`. اختر خيار Base أو Solana الذي تريد الدفع به، واستخدم تلك التفاصيل بالضبط لبناء رأس `X-402-Payment`، وأعِد محاولة نفس المسار.

#### بناء رأس X-402-Payment لـ Base

ينشئ السكربت التالي دفعة x402 موقَّعة على Base ويُرسل طلب الشحن. يتطلب حزمتي `x402` و`viem` من npm.

```bash
export EVM_PRIVATE_KEY=0xYOUR_PRIVATE_KEY
```

```bash
export X402_PAYMENT="$(
node --input-type=module <<'EOF'
import { createPaymentHeader } from "x402/client";
import { privateKeyToAccount } from "viem/accounts";

const signer = privateKeyToAccount(process.env.EVM_PRIVATE_KEY);
const amountUsd = 5;
const amountBaseUnits = String(Math.round(amountUsd * 1e6));

const header = await createPaymentHeader(signer, 2, {
  scheme: "exact",
  network: "base",
  maxAmountRequired: amountBaseUnits,
  resource: "https://api.venice.ai/api/v1/x402/top-up",
  description: "Venice x402 top-up",
  mimeType: "application/json",
  payTo: "0x2670B922ef37C7Df47158725C0CC407b5382293F",
  maxTimeoutSeconds: 300,
  asset: "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
  extra: { name: "USD Coin", version: "2" },
});

process.stdout.write(header);
EOF
)"

curl -X POST "https://api.venice.ai/api/v1/x402/top-up" \
  -H "X-402-Payment: $X402_PAYMENT"
```

<Note>
  استخدم أحدث استجابة `PAYMENT-REQUIRED` / `accepts` كمصدر للحقيقة في الإنتاج بدلًا من ترميز هذه القيم. لشحن Solana، ابنِ الدفعة من مدخل Solana المُعاد في `accepts`. تستخدم مدخلات Solana `network: "solana"`، وmint الـ USDC كـ `asset`، وقد تشمل بيانات وصفية خاصة بالشبكة مثل `extra.feePayer`.
</Note>

### الخطوة 4: إجراء طلب

بمجرد أن يصبح للمحفظة رصيد قابل للإنفاق، استدعِ أي نقطة نهاية مدعومة بالرأس `X-Sign-In-With-X`:

```bash
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header "Content-Type: application/json" \
  --header "X-Sign-In-With-X: $X402_AUTH" \
  --data '{
    "model": "kimi-k2-5",
    "messages": [
      {
        "role": "user",
        "content": "Hello from an x402-authenticated wallet."
      }
    ]
  }'
```

قد تتضمّن الاستجابات الناجحة رأس `X-Balance-Remaining`.

### الخطوة 5: فحص المعاملات (اختياري)

راجع سجل معاملات المحفظة:

```bash
export WALLET_ADDRESS=0xYOUR_WALLET_ADDRESS_OR_SOLANA_ADDRESS
export X402_AUTH=YOUR_BASE64_SIWX_HEADER

curl --request GET \
  --url "https://api.venice.ai/api/v1/x402/transactions/$WALLET_ADDRESS?limit=10&offset=0" \
  --header "X-Sign-In-With-X: $X402_AUTH"
```

يتضمّن السجل مدخلات مثل `TOP_UP` و`CHARGE` و`REFUND`.

يقبل معامل المسار `walletAddress` إمّا عنوان EVM (`0x...`) أو عنوان Solana بصيغة base58.

## المسارات المدعومة

### مسارات الاستدلال المدفوعة

تدعم مسارات Venice المدفوعة العامة التالية حاليًا مصادقة محفظة x402.

| الفئة | نقاط النهاية |
| --- | --- |
| الدردشة | `POST /api/v1/chat/completions`, `POST /api/v1/responses` |
| الصور | `POST /api/v1/image/generate`, `POST /api/v1/images/generations`, `POST /api/v1/image/upscale`, `POST /api/v1/image/edit`, `POST /api/v1/image/multi-edit`, `POST /api/v1/image/background-remove` |
| Embeddings | `POST /api/v1/embeddings` |
| الصوت | `POST /api/v1/audio/speech`, `POST /api/v1/audio/transcriptions`, `POST /api/v1/audio/complete`, `POST /api/v1/audio/queue`, `POST /api/v1/audio/retrieve` |
| الفيديو | `POST /api/v1/video/complete`, `POST /api/v1/video/queue`, `POST /api/v1/video/retrieve`, `POST /api/v1/video/transcriptions` |

### مسار الشحن

| نقطة النهاية | المصادقة | الغرض |
| --- | --- | --- |
| `POST /api/v1/x402/top-up` | الطلب الأول: لا شيء. إعادة المحاولة: `X-402-Payment` | إضافة أرصدة USDC إلى الرصيد القابل للإنفاق للمحفظة باستخدام خيار دفع مقبول من Base أو Solana |

### مسارات خاصة بالمحفظة فقط

تستخدم هذه المسارات `X-Sign-In-With-X` للهوية لكنها لا تخصم من الرصيد.

| نقطة النهاية | الغرض |
| --- | --- |
| `GET /api/v1/x402/balance/{walletAddress}` | التحقق من الرصيد القابل للإنفاق لعنوان محفظة EVM أو Solana |
| `GET /api/v1/x402/transactions/{walletAddress}` | عرض سجل المعاملات لعنوان محفظة EVM أو Solana |

## الأخطاء

| الحالة | السبب المحتمل | ما العمل |
| --- | --- | --- |
| `401` | حمولة Sign-In-With-X مُشوَّهة أو منتهية | أعِد بناء `X-Sign-In-With-X` بـ nonce وطابع زمني جديدين |
| `402` على مسار مدفوع | الرصيد القابل للإنفاق غير كافٍ | اشحن وأعِد المحاولة |
| `402` على `/x402/top-up` | متوقَّع. هذا تدفّق بدء الدفع. | استخدم تفاصيل الدفع المُعادة لبناء `X-402-Payment` وأعِد المحاولة |
| `403` على الرصيد أو المعاملات | عدم تطابق المحفظة | تأكد أن المحفظة المصادَق عليها تطابق معامل المسار `walletAddress` |
| `400` على الشحن | رأس دفع مُشوَّه | أعِد البناء من أحدث استجابة `402` |

## للوكلاء

التدفّق نفسه. خزّن المفاتيح الخاصة في متغيرات بيئة أو مدير أسرار، وتحقق من الرصيد قبل الطلبات لتفادي رحلات `402` غير الضرورية.
