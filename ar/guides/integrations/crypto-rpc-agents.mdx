## لماذا Venice للوكلاء السلسليين؟

| القدرة | ما يحصل عليه وكيلك |
|---|---|
| **الاستدلال** | أكثر من 230 نموذجاً للنص والصور والفيديو والصوت والتضمينات عبر نقطة نهاية واحدة متوافقة مع OpenAI |
| **Crypto RPC** | وكيل JSON-RPC 2.0 إلى 10 سلاسل EVM بالإضافة إلى Starknet (الرئيسية والتجريبية) |
| **المصادقة** | مفتاح API قياسي أو مصادقة محفظة x402 (دون الحاجة إلى حساب Venice) |
| **التمويل** | ذاتي: رهن VVV للحصول على DIEM يومياً. عبر المتصفح: عمليات شحن بالدولار أو العملات الرقمية عبر لوحة التحكم |
| **التجميع** | حتى 100 استدعاء JSON-RPC لكل طلب، متعدد السلاسل بالتوازي |
| **عدم التكرار (Idempotency)** | إعادة محاولات آمنة باستخدام ترويسة `Idempotency-Key` |

## المصادقة

اختر طريقة المصادقة التي تطابق طريقة تشغيل وكيلك.

| الطريقة | الأنسب لـ | كيف تعمل |
|---|---|---|
| **مفتاح API** | الوكلاء على الخادم، النشر الثابت | ترويسة `Authorization: Bearer <key>`. احصل على مفتاح من [venice.ai/settings/api](https://venice.ai/settings/api). |
| **محفظة x402** | الوكلاء المستقلون أو ذوو العمر القصير أو الأصليون بالعملات الرقمية | المحفظة توقّع رسالة Sign-In-With-X، وتدفع لكل طلب بـ USDC على Base أو Solana. لا حاجة لحساب Venice. راجع [دليل x402](/ar/guides/integrations/x402-venice-api). |

كلتا الطريقتين تشتركان في نفس حدود المعدل والفوترة بأرصدة Venice.

<Tip>
يمكن للوكلاء المستقلين فعلياً إنشاء مفتاح API الخاص بهم برهن VVV على Base. راجع [إنشاء مفتاح API لوكيل مستقل](/ar/guides/getting-started/generating-api-key-agent).
</Tip>

## بداية سريعة لـ Crypto RPC

أرسل أي طريقة JSON-RPC 2.0 إلى `POST /crypto/rpc/{network}`.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

الاستجابة:

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

تتضمن ترويسات الاستجابة `X-Venice-RPC-Credits` (الأرصدة المخصومة)، و`X-Venice-RPC-Cost-USD` (التكلفة بالدولار)، و`X-Request-ID` (معرف الترابط).

### الشبكات المدعومة

| العائلة | الشبكة الرئيسية | الشبكات التجريبية |
|---|---|---|
| Ethereum | `ethereum-mainnet` | `ethereum-sepolia`, `ethereum-holesky` |
| Base | `base-mainnet` | `base-sepolia` |
| Arbitrum | `arbitrum-mainnet` | `arbitrum-sepolia` |
| Optimism | `optimism-mainnet` | `optimism-sepolia` |
| Polygon | `polygon-mainnet` | `polygon-amoy` |
| Linea | `linea-mainnet` | `linea-sepolia` |
| Avalanche C-Chain | `avalanche-mainnet` | `avalanche-fuji` |
| BNB Smart Chain | `bsc-mainnet` | `bsc-testnet` |
| Blast | `blast-mainnet` | `blast-sepolia` |
| zkSync Era | `zksync-mainnet` | `zksync-sepolia` |
| Starknet | `starknet-mainnet` | `starknet-sepolia` |

استخدم [`GET /crypto/rpc/networks`](/ar/api-reference/endpoint/crypto/networks) للحصول على القائمة الحية الموثوقة.

### مستويات الطرق

تُجمَّع الطرق في ثلاثة مستويات أرصدة. التكلفة الإجمالية = `baseCredits[chain] × methodTier`.

| المستوى | المضاعف | أمثلة |
|---|---|---|
| **قياسي** | 1x | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas` |
| **متقدم** | 2x | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction` |
| **كبير** | 4x | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content` |

القائمة الكاملة وتفاصيل التسعير في [مرجع Crypto RPC API](/ar/api-reference/endpoint/crypto/rpc).

## وصفات الوكلاء

أنماط شائعة لوكلاء الذكاء الاصطناعي الذين يحتاجون إلى القراءة والكتابة على السلسلة.

### قراءة الرصيد الأصلي لمحفظة

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getBalance",
    "params": ["0xYourWalletAddress", "latest"],
    "id": 1
  }'
```

### قراءة رصيد رمز ERC-20

استدعِ منتقي `balanceOf(address)` باستخدام `eth_call`. حقل `data` هو المنتقي ذو 4 بايت (`0x70a08231`) متبوعاً بعنوان المحفظة المُحشى يساراً إلى 32 بايت. الأسهل ترك مكتبة تقوم بترميزها:

```typescript
import { encodeFunctionData, parseAbi } from 'viem'

const data = encodeFunctionData({
  abi: parseAbi(['function balanceOf(address) view returns (uint256)']),
  args: ['0xWalletAddress'],
})

const response = await fetch('https://api.venice.ai/api/v1/crypto/rpc/base-mainnet', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.VENICE_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    jsonrpc: '2.0',
    method: 'eth_call',
    params: [{ to: '0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf', data }, 'latest'],
    id: 1,
  }),
})
```

عنوان العقد أعلاه هو VVV على Base. استبدله بأي عقد ERC-20.

### إرسال معاملة موقّعة (دورة حياة كاملة)

لا تحتفظ Venice أبداً بمفاتيحك الخاصة. يجمع الوكيل معاملات tx عبر قراءات RPC، ويوقّع محلياً باستخدام مكتبة مثل [viem](https://viem.sh) أو [ethers](https://docs.ethers.org)، ثم يوصل الـ hex الخام عبر Venice.

<Steps>
  <Step title="احصل على الـ nonce التالي">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionCount","params":["0xAgentWallet","pending"],"id":1}'
    ```
    استخدم `"pending"` حتى لا تتصادم الإرسالات المتتابعة.
  </Step>
  <Step title="احصل على سعر الغاز">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_gasPrice","params":[],"id":1}'
    ```
    لسلاسل EIP-1559، فضّل `eth_feeHistory` لحساب `maxFeePerGas` و`maxPriorityFeePerGas`.
  </Step>
  <Step title="قدّر الغاز">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"from":"0xAgentWallet","to":"0xRecipient","value":"0x0","data":"0x..."}],"id":1}'
    ```
  </Step>
  <Step title="وقّع محلياً">
    ```typescript
    import { privateKeyToAccount } from 'viem/accounts'
    import { base } from 'viem/chains'

    const account = privateKeyToAccount(process.env.AGENT_PRIVATE_KEY)

    const signed = await account.signTransaction({
      chainId: base.id,
      nonce,                  // from step 1
      gas,                    // from step 3
      maxFeePerGas,           // from step 2 (fee history)
      maxPriorityFeePerGas,   // from step 2 (fee history)
      to: '0xRecipient',
      value: 0n,
      data: '0x...',
    })
    ```
  </Step>
  <Step title="أرسل عبر Venice">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Idempotency-Key: agent-tx-<id>" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_sendRawTransaction","params":["0xSignedHex"],"id":1}'
    ```
    اضبط دائماً `Idempotency-Key` على عمليات الإرسال حتى لا يتسبب انقطاع في الشبكة في بث مزدوج.
  </Step>
  <Step title="استطلع للحصول على الإيصال">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xTxHash"],"id":1}'
    ```
    استطلع كل بضع ثوانٍ حتى تكون `result` غير فارغة. تحقق من `result.status` (`"0x1"` = نجاح).
  </Step>
</Steps>

<Note>
يتم تسجيل كل استدعاء `eth_sendRawTransaction` من جانب الخادم مع تجزئة المعاملة والشبكة ومعرّف الطلب ومعرّف المستخدم المُستدعي. الحمولة الموقّعة نفسها لا يتم الاحتفاظ بها. هذا السجل التدقيقي موجود لكي يمكن تتبّع المفاتيح المخترقة المستخدمة في عمليات الإرسال غير المشروعة إلى الحساب المسؤول.
</Note>

### تجميع استدعاءات متعددة (فحص محفظة متعدد السلاسل)

أرسل حتى 100 كائن JSON-RPC في طلب واحد. كل كائن يُتحقق منه ويُفوتر بشكل مستقل.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[
    { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 1 },
    { "jsonrpc": "2.0", "method": "eth_getBalance", "params": ["0xWallet", "latest"], "id": 2 },
    { "jsonrpc": "2.0", "method": "eth_gasPrice", "params": [], "id": 3 }
  ]'
```

لقراءات متعددة السلاسل (استدعاء واحد لكل سلسلة)، أصدر طلبات متوازية إلى نقاط نهاية `{network}` مختلفة.

### إعادة محاولات آمنة مع عدم التكرار

اضبط ترويسة `Idempotency-Key` على أي سلسلة تطابق `[A-Za-z0-9_-]{1,255}`. تخزّن Venice الاستجابة مؤقتاً لمدة 24 ساعة مفتاحها `(user, key)`. تعيد إعادات التشغيل النتيجة المخزّنة مع `Idempotent-Replayed: true` ولا تكلّف شيئاً.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Idempotency-Key: agent-tx-2026-04-21-001" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_sendRawTransaction",
    "params": ["0xSignedRawTxHex"],
    "id": 1
  }'
```

هذا أمر حاسم لعمليات إرسال المعاملات حيث يمكن أن يتسبب انقطاع في الشبكة في بث وكيلك للمعاملة مرتين.

## تمويل مفتاح API الوكيل

بمجرد أن يحصل الوكيل على مفتاح Venice API، فإنه يحتاج إلى رصيد قابل للإنفاق على الحساب الأساسي قبل أن تقبل نقاط النهاية المدفوعة المفتاح. هناك طريقتان لوضع الرصيد هناك:

| المسار | ذاتي؟ | كيف تعمل |
|---|---|---|
| **DIEM من رهن VVV** | نعم | ارهن VVV في [عقد الرهن الذكي لـ Venice](https://basescan.org/address/0x321b7ff75154472B18EDb199033fF4D116F340Ff#code) على Base. تخصيص DIEM اليومي للمحفظة يتناسب مع حصتها من مجمّع الرهن. يحتاج الحساب إلى تراكم 0.1 DIEM على الأقل قبل أن يكون أي DIEM قابلاً للإنفاق. يُحدَّث DIEM في 00:00 UTC. لزيادة الإنفاق اليومي، ارهن المزيد من VVV. |
| **شحن بالدولار أو العملات الرقمية عبر لوحة التحكم** | لا (عبر المتصفح) | سجّل الدخول إلى [venice.ai](https://venice.ai) بنفس المحفظة (Sign-In-With-Ethereum)، ثم أضف الأرصدة في الإعدادات، API. كل من Stripe (بطاقة) وCoinbase (عملات رقمية) خلف تلك الصفحة ويتطلبان متصفحاً. الأرصدة لا تنتهي صلاحيتها أبداً. |

بالنسبة لوكيل يعمل دون مراقبة، **DIEM عبر رهن VVV هو مسار التمويل الوحيد بلا رأس بالكامل لمفتاح API مُنشأ اليوم**. إذا تجاوز الإنفاق اليومي للوكيل تخصيص DIEM، فإن الخيارات الواقعية هي: رهن المزيد من VVV، أو أن يقوم مشغّل بتسجيل الدخول والشحن بالدولار أو العملات الرقمية.

### رهن VVV مستقل وإنشاء المفاتيح

يمكن لوكيل مستقل حقيقي إدارة محفظة VVV الخاصة به على Base، ورهنها، وإنشاء مفتاح Venice API الخاص به دون أي تدخل بشري. التدفق الكامل:

<Steps>
  <Step title="احصل على VVV وETH للغاز">
    أرسل VVV إلى محفظة الوكيل (أو اجعل الوكيل يبادل على [Aerodrome](https://aerodrome.finance) أو [Uniswap](https://app.uniswap.org))، بالإضافة إلى كمية صغيرة من ETH على Base لمعاملتي الرهن.
  </Step>
  <Step title="ارهن VVV">
    `approve` لعقد الرهن على رمز VVV، ثم `stake(amount)` على `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. يتم تحديث رصيد sVVV للمحفظة بشكل ذري مع الرهن.
  </Step>
  <Step title="أنشئ مفتاح API">
    يُعيد `GET /api/v1/api_keys/generate_web3_key` رمز JWT تنتهي صلاحيته بعد 15 دقيقة من الإصدار. وقّع الرمز الخام بمحفظة الرهن، ثم أرسل `POST` بالعنوان والتوقيع والرمز مرة أخرى. تُعيد Venice مفتاح API مرتبطاً بحساب المستخدم المشتق من تلك المحفظة.
  </Step>
</Steps>

يتطلب الإنشاء فقط رصيد sVVV غير صفري، لذا فإن 1 VVV مرهون يكفي لاستلام مفتاح. **الإنفاق** بالمفتاح سؤال منفصل، يحكمه جدول التمويل أعلاه.

راجع [إنشاء مفتاح API لوكيل مستقل](/ar/guides/getting-started/generating-api-key-agent) للحصول على الشرح الكامل مع الشيفرة ومرجع الأخطاء الكامل.

## مصادقة محفظة x402 في 30 ثانية

إذا كان لدى وكيلك بالفعل محفظة Base أو Solana، فتجاوز مفتاح API تماماً. تتولى SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) توقيع Sign-In-With-X والشحن وتتبع الأرصدة.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // skip if the wallet already has balance

const response = await venice.chat({
  model: 'kimi-k2-6',
  messages: [{ role: 'user', content: 'What is the latest block on Base?' }]
})
```

تعمل نفس مصادقة المحفظة مقابل `/crypto/rpc/{network}` لقراءات وكتابات البلوكشين. تفاصيل البروتوكول الكاملة في [دليل x402](/ar/guides/integrations/x402-venice-api).

## التسعير

يتم فوترة Crypto RPC بأرصدة Venice. تتضمن كل استجابة `X-Venice-RPC-Credits` (الأرصدة المخصومة) و`X-Venice-RPC-Cost-USD` (التكلفة بالدولار) حتى يتمكن وكيلك من تتبع الإنفاق لكل طلب.

### الأرصدة الأساسية لكل سلسلة

| الأرصدة الأساسية | السلاسل |
|---|---|
| **20** | Ethereum, Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast, Starknet |
| **30** | zkSync Era |

### أمثلة على التكاليف

تسعير ملحوظ لمستويات الطرق القياسية والمتقدمة والكبيرة:

| الاستدعاء | الأرصدة | التكلفة بالدولار |
|---|---|---|
| `eth_call` على Ethereum (20 × 1x) | 20 | ~$0.0000140 |
| `trace_transaction` على Ethereum (20 × 2x) | 40 | ~$0.0000280 |
| `trace_replayTransaction` على Ethereum (20 × 4x) | 80 | ~$0.0000560 |
| `eth_call` على zkSync (30 × 1x) | 30 | ~$0.0000210 |

ثق دائماً بترويسة الاستجابة `X-Venice-RPC-Cost-USD` للحصول على التكلفة الموثوقة. يُفوتر العناصر الخاطئة في طلبات التجميع بـ 5 أرصدة ثابتة لكل عنصر.

### حدود المعدل

| المستوى | الطلبات في الدقيقة |
|---|---|
| قياسي | 100 |
| الموظفون | 1,000 |

عند التجاوز، تعيد نقطة النهاية `429` مع ترويسات استجابة قياسية `X-RateLimit-*`.

## معالجة الأخطاء

استجابات HTTP الشائعة التي ينبغي على وكيلك معالجتها:

| الحالة | المعنى | ما العمل |
|---|---|---|
| `400` | طريقة JSON-RPC غير مدعومة أو غير معيّنة، أو تجميع مشوّه | تحقق من الطريقة مقابل [قائمة السماح](/ar/api-reference/endpoint/crypto/rpc). يسمّي جسم الخطأ الطريقة المخالفة. |
| `400` | إعادة تشغيل `Idempotency-Key` بجسم مختلف | استخدم مفتاحاً جديداً للطلبات المتمايزة. |
| `402` | لا ترويسة مصادقة على الإطلاق (يتضمن جسم الاستجابة `authOptions` يدرج كلا مسارَي المصادقة المدعومين)، أو نفاد الأرصدة مع ترويسة مصادقة صالحة | إذا لم تكن هناك مصادقة: أرفق `Authorization: Bearer ...` أو ترويسة x402 `X-Sign-In-With-X`. إذا نفدت الأرصدة: مع مفتاح Bearer، موّل الحساب (DIEM أو USD أو شحن من لوحة التحكم)؛ مع مصادقة x402، استدعِ `POST /api/v1/x402/top-up` مباشرة. |
| `429` | تجاوز حد المعدل (100 طلب/دقيقة قياسي، 1,000 طلب/دقيقة للموظفين) | احترم `X-RateLimit-Reset` وتراجَع. اجمع حتى 100 استدعاء لكل طلب لتوزيع الحد. |
| `5xx` | عقبة في عقدة RPC العلوية | أعد المحاولة بنفس `Idempotency-Key` لتجنب الفوترة المزدوجة. |

تأتي أخطاء العناصر الفردية في التجميعات (مثل معاملات غير صالحة في أحد N استدعاءات) داخل استجابة `200 OK` مع حقل `error` بصيغة JSON-RPC على العنصر المخالف. تُفوتر تلك العناصر بـ 5 أرصدة ثابتة لكل عنصر.

## غير مدعوم

تُرفض هذه الفئات من الطرق عمداً:

- **مخصصة لـ WebSocket** (`eth_subscribe`, `eth_unsubscribe`): الوكيل HTTP فقط. استطلع بدلاً من ذلك.
- **مرشحات ذات حالة** (`eth_newFilter`, `eth_getFilterChanges`, إلخ): حالة المرشح مثبتة على خلفية واحدة وتنكسر على وكيل موزّن الحمل. استخدم `eth_getLogs` بدلاً من ذلك.
- **طرق تحمل المفاتيح** (`eth_sign`, `eth_accounts`, `eth_mining`): المزوّدون المستضافون لا يحتفظون بمفاتيح المستخدمين. وقّع من جانب العميل وأرسل عبر `eth_sendRawTransaction`.
- **طرق غير معيّنة**: أي شيء غير مدرج في قائمة السماح يعيد `400`. اتصل بالدعم لطلب إضافات.
