Reenvía una solicitud JSON-RPC 2.0 (individual o por lotes) a un nodo blockchain compatible. Admite tanto autenticación con API key como con cartera x402. La facturación es por crédito y se denomina en tu saldo de Venice: una sola credencial, una sola factura y todas las cadenas listadas abajo.

## Autenticación

Este endpoint admite dos métodos de autenticación:

- **API Key**: autenticación estándar con Bearer token mediante la cabecera `Authorization: Bearer <key>`.
- **Cartera x402**: pago por uso con créditos USDC desde una cartera en Base o Solana. No se requiere cuenta de Venice. Consulta la [guía de x402](/es/guides/integrations/x402-venice-api) para la configuración.

Ambos métodos comparten los mismos límites de uso y facturación (créditos de Venice).

## Formato de la solicitud

### Solicitud individual

```json
{
  "jsonrpc": "2.0",
  "method": "eth_chainId",
  "params": [],
  "id": 1
}
```

### Solicitud por lotes

Un array de hasta **100** objetos JSON-RPC 2.0. Cada elemento se valida de forma independiente; si algún método no está admitido, todo el lote se rechaza con `400` y se listan todos los nombres de método ofensivos en el mensaje de error.

```json
[
  { "jsonrpc": "2.0", "method": "eth_chainId", "params": [], "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 2 }
]
```

## Métodos admitidos y niveles de precio

Los métodos se clasifican en tres niveles de crédito. Los créditos consumidos por llamada = `baseCredits[chain] × methodTier`.

| Nivel | Multiplicador | Métodos de ejemplo |
|---|---|---|
| **Standard** | 1× | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas`, `net_version`, `web3_clientVersion`, métodos de bundler ERC-4337 (`eth_sendUserOperation`, `eth_estimateUserOperationGas`, etc.), extensiones de familia de cadena (`zks_*`, `linea_*`, `bor_*`, `starknet_*`) |
| **Advanced** | 2× | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction`, `debug_traceBlockByHash`, `txpool_inspect`, `txpool_status`, `arbtrace_*` |
| **Large** | 4× | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content`, `arbtrace_replayTransaction`, `arbtrace_replayBlockTransactions` |

### Créditos base por cadena

| baseCredits | Cadenas |
|---|---|
| **20** | Ethereum + todas las L2 EVM anteriores (Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast) y Starknet |
| **30** | zkSync Era |

### Ejemplos de coste

A `~$6,25 × 10⁻⁷` por crédito de Venice:

| Llamada | Créditos | Coste USD |
|---|---|---|
| `eth_call` en Ethereum (20 × 1×) | 20 | $0,0000125 |
| `trace_transaction` en Ethereum (20 × 2×) | 40 | $0,0000250 |
| `trace_replayTransaction` en Ethereum (20 × 4×) | 80 | $0,0000500 |
| `eth_call` en zkSync (30 × 1×) | 30 | $0,0000188 |

### No admitidos

- **Métodos solo WebSocket** (`eth_subscribe`, `eth_unsubscribe`) — este proxy es solo HTTP. Haz polling, o cambia a un proveedor WebSocket directo.
- **Métodos de filtro con estado** (`eth_newFilter`, `eth_getFilterChanges`, `eth_getFilterLogs`, `eth_uninstallFilter`, `eth_newBlockFilter`, `eth_newPendingTransactionFilter`) — el estado del filtro está fijado a un único backend upstream y se rompe silenciosamente en un proxy HTTP con balanceo de carga. Usa `eth_getLogs` (sin estado) en su lugar.
- **Métodos de minería / con claves** (`eth_sign`, `eth_accounts`, `eth_mining`, `eth_hashrate`, `eth_getWork`, `eth_submitWork`) — los endpoints de proveedores hosteados no guardan claves privadas de usuario, por lo que estos siempre fallan. Firma las transacciones en el cliente y envíalas mediante `eth_sendRawTransaction`.
- **Métodos no mapeados** — todo lo que no esté explícitamente en la allowlist devuelve `400`. Contacta con soporte para solicitar adiciones.

## Facturación por elemento de lote

Incluso cuando la respuesta HTTP es `200`, los elementos individuales del lote pueden devolver un campo `error` JSON-RPC (por ejemplo, un error de parámetros incorrectos o un método no compatible en la cadena destino). Venice factura estos elementos a **5 créditos cada uno** en lugar del nivel completo del método — una pequeña concesión para los errores habituales de "exploración de la API".

```json
// Solicitud por lotes:
[
  { "jsonrpc": "2.0", "method": "eth_chainId",   "params": [],         "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_getBalance","params": ["bad"],    "id": 2 }
]

// Respuesta (HTTP 200, X-Venice-RPC-Credits: 25):
[
  { "jsonrpc": "2.0", "id": 1, "result": "0x1" },
  { "jsonrpc": "2.0", "id": 2, "error": { "code": -32602, "message": "invalid params" } }
]
```

El primer elemento (éxito) factura 20 créditos, el segundo (error a nivel RPC) factura 5, suma `= 25`.

## Límites de uso

Tope de solicitudes por minuto por llamador autenticado:

| Nivel | Solicitudes/minuto |
|---|---|
| Standard | 100 |
| Staff | 1.000 |

Cuando se supera el tope, el endpoint devuelve `429` con un `customMessage` y las cabeceras de respuesta estándar `X-RateLimit-*`.

## Idempotencia

Establece la cabecera de solicitud `Idempotency-Key` a cualquier cadena que coincida con `[A-Za-z0-9_-]{1,255}` para permitir reintentos seguros. La respuesta se almacena en caché durante 24 horas indexada por `(user, idempotency-key)`:

- Repetir la **misma key con el mismo body** devuelve la respuesta cacheada y una cabecera de respuesta `Idempotent-Replayed: true`. No se toca el upstream y no se cobran créditos nuevos.
- Repetir la **misma key con un body distinto** devuelve `400` para evitar una corrupción silenciosa de estado. Elige una key nueva para solicitudes distintas.

## Cabeceras de respuesta

| Cabecera | Descripción |
|---|---|
| `X-Venice-RPC-Credits` | Créditos cobrados por esta solicitud. En solicitudes por lotes es la suma de los elementos. |
| `X-Venice-RPC-Cost-USD` | Coste en dólares con 8 decimales. Igual a `X-Venice-RPC-Credits × precio por crédito`. |
| `X-Request-ID` | ID de correlación de 32 caracteres. Inclúyelo en cualquier comunicación con soporte. |
| `Idempotent-Replayed` | Presente con valor `"true"` cuando la respuesta proviene de la caché de idempotencia. |
| `X-RateLimit-Limit` / `X-RateLimit-Remaining` / `X-RateLimit-Reset` | Solo se establecen en respuestas 429. |

## Registro forense para relays de transacciones

Cada llamada a `eth_sendRawTransaction` se registra del lado del servidor con el hash de la tx (keccak256 de los bytes en bruto), el slug de la red, el ID de solicitud y el ID de usuario del llamador. **No** retenemos el payload firmado en sí — el hash se puede recuperar del recibo on-chain. Este rastro de auditoría existe para que, si la API key de un cliente se ve comprometida y se usa para retransmitir transacciones ilícitas a través de nuestra infraestructura, podamos correlacionar la actividad on-chain con la cuenta responsable.

## Ejemplo

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Cabeceras de respuesta: `X-Venice-RPC-Credits: 20`, `X-Venice-RPC-Cost-USD: 0.00001250`, `X-Request-ID: <nanoid>`.
