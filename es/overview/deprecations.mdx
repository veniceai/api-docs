La API de Venice existe para dar a los desarrolladores acceso privado y sin restricciones a modelos de calidad de producción, libres de filtros ocultos o decisiones de caja negra.

A medida que los modelos mejoran, ocasionalmente retiramos los más antiguos en favor de alternativas más inteligentes, rápidas o capaces. Diseñamos estas transiciones para que sean predecibles y con poca fricción.

## Deprecaciones de modelos

Sabemos que las deprecaciones pueden ser disruptivas. Por eso buscamos deprecar solo cuando es necesario y diseñamos funcionalidades como las traits y los modelos con marca Venice para minimizar la disrupción.

Podemos deprecar un modelo cuando:

- Un modelo más nuevo ofrece una mejora clara para el mismo caso de uso
- El modelo ya no cumple nuestros estándares de rendimiento o fiabilidad
- Tiene un uso bajo de forma sostenida y seguir manteniéndolo fragmentaría la experiencia para los demás

## Proceso de deprecación

Cuando un modelo cumple los criterios de deprecación, anunciamos el cambio con 30–60 días de aviso. Los avisos de deprecación se publican a través del [changelog](https://featurebase.venice.ai/changelog) y nuestro [servidor de Discord](https://discord.gg/askvenice). Cuando llamas a un modelo deprecado durante el período de aviso, la respuesta de la API incluirá un aviso de deprecación.

Durante el período de aviso, el modelo permanece disponible, aunque en algunos casos podemos reducir la capacidad de infraestructura. Siempre proporcionamos un reemplazo recomendado y, cuando es necesario, ofrecemos guía de migración para facilitar la transición.

Tras la fecha de retirada, las solicitudes al modelo se redirigirán automáticamente a un modelo de potencia de procesamiento similar al mismo precio o inferior. Si el enrutamiento no es posible por razones técnicas o de seguridad, la API devolverá una respuesta 410 Gone. Si un modelo deprecado fue seleccionado mediante una trait (como `default_code`, `default_vision` o `fastest`), esa trait se reasignará a un reemplazo compatible.

Nunca eliminamos modelos en silencio ni alteramos el comportamiento sin versionado. Siempre sabrás qué se está ejecutando y cómo prepararte para lo que viene.

<Note>
Mejoras solo de rendimiento: podemos lanzar mejoras que preservan el comportamiento del modelo a la vez que mejoran el rendimiento, la latencia o la eficiencia de coste. Estas actualizaciones son retrocompatibles y no requieren ninguna acción por parte del cliente.
</Note>

Consulta el [seguidor de deprecación de modelos](#model-deprecation-tracker) más abajo. Para anuncios anteriores, revisa el [changelog](https://featurebase.venice.ai/changelog) y nuestro [servidor de Discord](https://discord.gg/askvenice).

## Cómo se seleccionan los modelos para la API de Venice

Seleccionamos cuidadosamente qué modelos poner a disposición en función del rendimiento, la fiabilidad y las necesidades reales de los desarrolladores. Para ser incluido, un modelo debe demostrar un rendimiento sólido, comportarse de forma consistente bajo endpoints compatibles con OpenAI y ofrecer una mejora clara sobre al menos uno de los modelos que ya admitimos.


Los modelos que estamos evaluando pueden lanzarse primero en [beta](/es/overview/beta-models) para recopilar comentarios y validar el rendimiento a escala.

No exponemos modelos que sean redundantes, no probados o que no estén listos para un uso constante en producción. Nuestro objetivo es mantener la API de Venice limpia, capaz y optimizada para lo que los desarrolladores realmente construyen.

Aprende más en [Deprecaciones de modelos](/es/overview/deprecations#model-deprecations) y <a href="/es/overview/models" target="_blank" rel="noopener noreferrer">Lista actual de modelos</a>.

## Versionado y alias

Todos los modelos de Venice se identifican mediante un ID único y permanente. Por ejemplo:

```venice-uncensored```
```zai-org-glm-4.7```
```zai-org-glm-5```
```qwen3-vl-235b-a22b```

Los IDs de modelo son estables. Si hay un cambio que rompa compatibilidad, lanzaremos un nuevo ID de modelo (por ejemplo, añadiendo una versión como v2). Si no hay cambios disruptivos, podemos actualizar el modelo existente y comunicaremos los cambios significativos.

Las traits ofrecen una abstracción estable para seleccionar modelos a la vez que dan a Venice la flexibilidad para mejorar la implementación subyacente. Los desarrolladores que prefieran acceso automático a los modelos recomendados más recientes pueden basarse en los alias por trait.

Para aplicaciones que requieren consistencia estricta y comportamiento predecible, recomendamos referenciar IDs de modelo fijos.

### Comprobar el estado de deprecación a través de la API

Puedes comprobar si un modelo está programado para retirada llamando al endpoint [Listar modelos](/es/api-reference/endpoint/models/list). Los modelos con fecha de retirada incluyen un objeto `deprecation` en su `model_spec`:

```json
{
  "id": "some-model-id",
  "model_spec": {
    "name": "Some Model",
    "privacy": "private",
    "deprecation": {
      "date": "2025-03-01T00:00:00.000Z"
    }
  },
  "type": "text",
  "object": "model",
  "owned_by": "venice.ai"
}
```

El objeto `deprecation` solo aparece cuando un modelo está programado para retirada. Puedes comprobar `if (model.model_spec.deprecation)` para saber si un modelo se está retirando, y usar la fecha ISO 8601 para avisar a los usuarios o planificar migraciones.
