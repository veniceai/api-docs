La API de Venice existe para dar a los desarrolladores acceso privado y sin restricciones a modelos de calidad de producción, libres de filtros ocultos o decisiones opacas.

A medida que los modelos mejoran, ocasionalmente retiramos los más antiguos en favor de alternativas más inteligentes, rápidas o capaces. Diseñamos estas transiciones para que sean predecibles y con poca fricción.

## Deprecaciones de modelos

Sabemos que las deprecaciones pueden ser disruptivas. Por eso intentamos deprecar solo cuando es necesario y diseñamos funciones como las características y los modelos con marca Venice para minimizar la disrupción.

Podemos deprecar un modelo cuando:

- Un modelo más nuevo ofrece una mejora clara para el mismo caso de uso
- El modelo ya no cumple nuestros estándares de rendimiento o fiabilidad
- Tiene un uso consistentemente bajo, y seguir manteniéndolo fragmentaría la experiencia para los demás

## Proceso de deprecación

Cuando un modelo cumple los criterios de deprecación, proporcionamos un aviso de 7 a 15 días antes de su retirada. Puedes hacer seguimiento de las próximas deprecaciones a través de:

- El [Tracker de deprecación de modelos](#model-deprecation-tracker) más abajo
- El endpoint de la API [List Models](/api-reference/endpoint/models/list) (los modelos con retirada programada incluyen un objeto `deprecation`)
- Nuestro [changelog](https://featurebase.venice.ai/changelog)
- Notificaciones por correo electrónico, enviadas automáticamente si has llamado al modelo en los últimos 15 días

Durante el periodo de aviso, las llamadas a un modelo deprecado incluirán una advertencia de deprecación en la respuesta de la API.

Durante el periodo de aviso, el modelo sigue disponible, aunque en algunos casos podemos reducir la capacidad de infraestructura. Siempre proporcionamos un reemplazo recomendado y, cuando es necesario, ofrecemos orientación de migración para facilitar la transición.

Después de la fecha de retirada, las solicitudes al modelo se enrutarán automáticamente a un modelo de potencia de procesamiento similar al mismo precio o inferior. Si el enrutamiento no es posible por razones técnicas o de seguridad, la API devolverá una respuesta 410 Gone. Si un modelo deprecado fue seleccionado mediante una característica (como `default_code`, `default_vision` o `fastest`), esa característica se reasignará a un reemplazo compatible.

Nunca retiramos modelos en silencio ni alteramos su comportamiento sin versionado. Siempre sabrás qué se está ejecutando y cómo prepararte para lo que viene.

<Note>
Mejoras solo de rendimiento: podemos lanzar mejoras que preservan el comportamiento del modelo mientras mejoran el rendimiento, la latencia o la eficiencia de coste. Estas actualizaciones son compatibles hacia atrás y no requieren acción del cliente.
</Note>

Consulta el [Tracker de deprecación de modelos](#model-deprecation-tracker) más abajo. Para anuncios anteriores, consulta el [changelog](https://featurebase.venice.ai/changelog).

## Cómo se seleccionan los modelos para la API de Venice

Seleccionamos cuidadosamente qué modelos hacer disponibles según rendimiento, fiabilidad y necesidades reales de los desarrolladores. Para ser incluido, un modelo debe demostrar un rendimiento sólido, comportarse de forma consistente con los endpoints compatibles con OpenAI y ofrecer una mejora clara sobre al menos uno de los modelos que ya admitimos.


Los modelos que estamos evaluando pueden lanzarse primero en [beta](/overview/beta-models) para recoger feedback y validar el rendimiento a gran escala.

No exponemos modelos que sean redundantes, no probados o no estén listos para un uso consistente en producción. Nuestro objetivo es mantener la API de Venice limpia, capaz y optimizada para lo que los desarrolladores realmente construyen.

Aprende más en [Deprecaciones de modelos](/overview/deprecations#model-deprecations) y <a href="/overview/models" target="_blank" rel="noopener noreferrer">Lista actual de modelos</a>.

## Versionado y aliases

Todos los modelos de Venice se identifican mediante un ID único y permanente. Por ejemplo:

```venice-uncensored```
```zai-org-glm-4.7```
```zai-org-glm-5```
```qwen3-vl-235b-a22b```

Los IDs de modelo son estables. Si hay un cambio disruptivo, lanzaremos un nuevo ID de modelo (por ejemplo, añadiendo una versión como v2). Si no hay cambios disruptivos, podemos actualizar el modelo existente y comunicaremos los cambios significativos.

Para ofrecer flexibilidad, Venice también mantiene aliases simbólicos, implementados mediante características, que apuntan al modelo predeterminado recomendado para una tarea dada:

Las características ofrecen una abstracción estable para seleccionar modelos al tiempo que dan a Venice flexibilidad para mejorar la implementación subyacente. Los desarrolladores que prefieren acceso automático a los modelos recomendados más recientes pueden confiar en aliases basados en características.

Para aplicaciones que requieren coherencia estricta y comportamiento predecible, recomendamos referenciar IDs de modelo fijos.

### Comprobar el estado de deprecación a través de la API

Puedes comprobar si un modelo está programado para retirada llamando al endpoint [List Models](/api-reference/endpoint/models/list). Los modelos con una fecha de retirada incluyen un objeto `deprecation` en su `model_spec`:

```json
{
  "id": "some-model-id",
  "model_spec": {
    "name": "Some Model",
    "privacy": "private",
    "deprecation": {
      "date": "2025-03-01T00:00:00.000Z"
    }
  },
  "type": "text",
  "object": "model",
  "owned_by": "venice.ai"
}
```

El objeto `deprecation` solo aparece cuando un modelo está programado para retirada. Puedes comprobar `if (model.model_spec.deprecation)` para saber si un modelo se está retirando, y usar la fecha ISO 8601 para advertir a los usuarios o planificar migraciones.
