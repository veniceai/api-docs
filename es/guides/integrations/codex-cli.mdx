Esta guía muestra cómo ejecutar OpenAI Codex CLI con Venice usando las rutas de configuración oficiales de Codex: `~/.codex/config.toml` (nivel de usuario) o `.codex/config.toml` (nivel de proyecto).

## Requisitos previos

- Una API key de Venice de [venice.ai/settings/api](https://venice.ai/settings/api)
- Codex CLI instalado y funcionando en tu máquina

---

## Configuración

<Steps>
  <Step title="Crea la ruta de configuración del proyecto">
    Desde la raíz de tu proyecto:

    ```bash
    mkdir -p .codex
    ```
  </Step>

  <Step title="Crea .codex/config.toml">
    Crea el archivo y pega la siguiente configuración:

    ```toml
    #:schema https://developers.openai.com/codex/config-schema.json

    model = "openai-gpt-54" # usa cualquier modelo de Venice
    model_provider = "venice"
    model_reasoning_effort = "high"
    personality = "pragmatic"
    sandbox_mode = "workspace-write"

    [model_providers.venice]
    name = "Venice"
    base_url = "https://api.venice.ai/api/v1/"
    experimental_bearer_token = "YOUR VENICE API KEY"
    wire_api = "responses"
    ```
  </Step>

  <Step title="Sustituye los dos placeholders">
    Actualiza:

    - `model` con el ID del modelo de Venice que quieras usar
    - `experimental_bearer_token` con tu API key real de Venice

    Puedes consultar los IDs de modelo disponibles en el [catálogo de modelos de texto](/es/models/text).
  </Step>

  <Step title="Ejecuta Codex CLI normalmente">
    Inicia Codex CLI desde el mismo proyecto. Cargará `.codex/config.toml` (para proyectos confiables) y enrutará las solicitudes a través de Venice.
  </Step>
</Steps>

---

## Rutas oficiales de config de Codex

- **Valores por defecto del usuario**: `~/.codex/config.toml`
- **Overrides de proyecto**: `.codex/config.toml` (se cargan solo para proyectos confiables)

Si quieres que la configuración de Venice se aplique en todas partes, pon la misma configuración en `~/.codex/config.toml`.

---

## Precedencia de la configuración (de mayor a menor)

1. Flags de CLI y overrides `--config`
2. Valores de perfil (`--profile <name>`)
3. Capas de config del proyecto (`.codex/config.toml`, gana el directorio más cercano)
4. Config de usuario (`~/.codex/config.toml`)
5. Config de sistema (`/etc/codex/config.toml`, Unix)
6. Valores por defecto integrados

---

## Notas

- Mantén tu API key privada y nunca subas claves reales a git.
- Codex ignora la configuración `.codex/` del proyecto cuando un proyecto está marcado como no confiable.
- Si cambias de modelo, solo actualiza el campo `model`.
- El ajuste `wire_api = "responses"` es obligatorio para esta configuración de proveedor.
