## ¿Por qué Venice para agentes on-chain?

| Capacidad | Lo que obtiene tu agente |
|---|---|
| **Inferencia** | Más de 230 modelos de texto, imagen, video, audio y embeddings a través de un endpoint compatible con OpenAI |
| **Crypto RPC** | Proxy JSON-RPC 2.0 a 10 cadenas EVM más Starknet (mainnet y testnets) |
| **Autenticación** | API key estándar o autenticación x402 con wallet (no se requiere cuenta de Venice) |
| **Financiación** | Autónoma: staking de VVV para DIEM diario. Navegador: recargas en USD o cripto vía dashboard |
| **Batching** | Hasta 100 llamadas JSON-RPC por solicitud, multi-cadena en paralelo |
| **Idempotencia** | Reintentos seguros con la cabecera `Idempotency-Key` |

## Autenticación

Elige el método de autenticación que corresponda a cómo se ejecuta tu agente.

| Método | Ideal para | Cómo funciona |
|---|---|---|
| **API key** | Agentes del lado del servidor, despliegues fijos | Cabecera `Authorization: Bearer <key>`. Obtén una clave en [venice.ai/settings/api](https://venice.ai/settings/api). |
| **Wallet x402** | Agentes autónomos, cripto-nativos o de vida corta | La wallet firma un mensaje Sign-In-With-X, paga por solicitud en USDC en Base o Solana. No se necesita cuenta de Venice. Consulta la [guía x402](/es/guides/integrations/x402-venice-api). |

Ambos métodos comparten los mismos límites de tasa y facturación en créditos de Venice.

<Tip>
Los agentes verdaderamente autónomos pueden mintear su propia API key stakeando VVV en Base. Consulta [Creación autónoma de API Key para agentes](/es/guides/getting-started/generating-api-key-agent).
</Tip>

## Inicio rápido de Crypto RPC

Envía cualquier método JSON-RPC 2.0 a `POST /crypto/rpc/{network}`.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

Respuesta:

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Las cabeceras de respuesta incluyen `X-Venice-RPC-Credits` (créditos cobrados), `X-Venice-RPC-Cost-USD` (coste en dólares) y `X-Request-ID` (ID de correlación).

### Redes soportadas

| Familia | Mainnet | Testnets |
|---|---|---|
| Ethereum | `ethereum-mainnet` | `ethereum-sepolia`, `ethereum-holesky` |
| Base | `base-mainnet` | `base-sepolia` |
| Arbitrum | `arbitrum-mainnet` | `arbitrum-sepolia` |
| Optimism | `optimism-mainnet` | `optimism-sepolia` |
| Polygon | `polygon-mainnet` | `polygon-amoy` |
| Linea | `linea-mainnet` | `linea-sepolia` |
| Avalanche C-Chain | `avalanche-mainnet` | `avalanche-fuji` |
| BNB Smart Chain | `bsc-mainnet` | `bsc-testnet` |
| Blast | `blast-mainnet` | `blast-sepolia` |
| zkSync Era | `zksync-mainnet` | `zksync-sepolia` |
| Starknet | `starknet-mainnet` | `starknet-sepolia` |

Usa [`GET /crypto/rpc/networks`](/es/api-reference/endpoint/crypto/networks) para la lista en vivo y autoritativa.

### Niveles de método

Los métodos se agrupan en tres niveles de créditos. Coste total = `baseCredits[chain] × methodTier`.

| Nivel | Multiplicador | Ejemplos |
|---|---|---|
| **Standard** | 1x | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas` |
| **Advanced** | 2x | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction` |
| **Large** | 4x | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content` |

Lista completa y detalle de precios en la [referencia de la API de Crypto RPC](/es/api-reference/endpoint/crypto/rpc).

## Recetas para agentes

Patrones comunes para agentes de IA que necesitan leer y escribir on-chain.

### Leer el saldo nativo de una wallet

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getBalance",
    "params": ["0xYourWalletAddress", "latest"],
    "id": 1
  }'
```

### Leer el saldo de un token ERC-20

Llama al selector `balanceOf(address)` con `eth_call`. El campo `data` es el selector de 4 bytes (`0x70a08231`) seguido de la dirección de la wallet rellenada a la izquierda hasta 32 bytes. Lo más fácil es dejar que una librería lo codifique:

```typescript
import { encodeFunctionData, parseAbi } from 'viem'

const data = encodeFunctionData({
  abi: parseAbi(['function balanceOf(address) view returns (uint256)']),
  args: ['0xWalletAddress'],
})

const response = await fetch('https://api.venice.ai/api/v1/crypto/rpc/base-mainnet', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.VENICE_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    jsonrpc: '2.0',
    method: 'eth_call',
    params: [{ to: '0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf', data }, 'latest'],
    id: 1,
  }),
})
```

La dirección de contrato anterior es VVV en Base. Sustitúyela por cualquier contrato ERC-20.

### Enviar una transacción firmada (ciclo de vida completo)

Venice nunca guarda tus claves privadas. El agente recopila los parámetros de la tx vía lecturas RPC, firma localmente con una librería como [viem](https://viem.sh) o [ethers](https://docs.ethers.org), y luego relaya el hex crudo a través de Venice.

<Steps>
  <Step title="Obtén el próximo nonce">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionCount","params":["0xAgentWallet","pending"],"id":1}'
    ```
    Usa `"pending"` para que envíos consecutivos no colisionen.
  </Step>
  <Step title="Obtén el precio del gas">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_gasPrice","params":[],"id":1}'
    ```
    Para cadenas EIP-1559, prefiere `eth_feeHistory` para calcular `maxFeePerGas` y `maxPriorityFeePerGas`.
  </Step>
  <Step title="Estima el gas">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"from":"0xAgentWallet","to":"0xRecipient","value":"0x0","data":"0x..."}],"id":1}'
    ```
  </Step>
  <Step title="Firma localmente">
    ```typescript
    import { privateKeyToAccount } from 'viem/accounts'
    import { base } from 'viem/chains'

    const account = privateKeyToAccount(process.env.AGENT_PRIVATE_KEY)

    const signed = await account.signTransaction({
      chainId: base.id,
      nonce,                  // del paso 1
      gas,                    // del paso 3
      maxFeePerGas,           // del paso 2 (fee history)
      maxPriorityFeePerGas,   // del paso 2 (fee history)
      to: '0xRecipient',
      value: 0n,
      data: '0x...',
    })
    ```
  </Step>
  <Step title="Envía a través de Venice">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Idempotency-Key: agent-tx-<id>" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_sendRawTransaction","params":["0xSignedHex"],"id":1}'
    ```
    Establece siempre `Idempotency-Key` en relays para que un fallo de red no pueda doblar la difusión.
  </Step>
  <Step title="Polling del recibo">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xTxHash"],"id":1}'
    ```
    Haz polling cada pocos segundos hasta que `result` sea no nulo. Comprueba `result.status` (`"0x1"` = éxito).
  </Step>
</Steps>

<Note>
Cada llamada `eth_sendRawTransaction` se registra del lado del servidor con el hash de la tx, la red, el ID de la solicitud y el ID del usuario que llama. La carga firmada en sí no se retiene. Este rastro de auditoría existe para que las claves comprometidas usadas para relays ilícitos puedan rastrearse hasta la cuenta responsable.
</Note>

### Batch de varias llamadas (verificación de portafolio multi-cadena)

Envía hasta 100 objetos JSON-RPC en una sola solicitud. Cada uno se valida y factura de forma independiente.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[
    { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 1 },
    { "jsonrpc": "2.0", "method": "eth_getBalance", "params": ["0xWallet", "latest"], "id": 2 },
    { "jsonrpc": "2.0", "method": "eth_gasPrice", "params": [], "id": 3 }
  ]'
```

Para lecturas multi-cadena (una llamada por cadena), emite solicitudes paralelas a diferentes endpoints `{network}`.

### Reintentos seguros con idempotencia

Establece la cabecera `Idempotency-Key` a cualquier cadena que coincida con `[A-Za-z0-9_-]{1,255}`. Venice cachea la respuesta durante 24 horas con clave `(usuario, key)`. Las reproducciones devuelven el resultado cacheado con `Idempotent-Replayed: true` y no cobran nada.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Idempotency-Key: agent-tx-2026-04-21-001" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_sendRawTransaction",
    "params": ["0xSignedRawTxHex"],
    "id": 1
  }'
```

Esto es crítico para relays de transacciones donde un fallo de red podría hacer que tu agente difunda dos veces la misma tx.

## Financiar la API key del agente

Una vez que el agente tiene una API key de Venice, necesita saldo gastable en la cuenta subyacente antes de que los endpoints de pago acepten la clave. Hay dos formas de poner saldo allí:

| Vía | ¿Autónoma? | Cómo funciona |
|---|---|---|
| **DIEM por staking de VVV** | Sí | Stakea VVV en el [Smart Contract de Staking de Venice](https://basescan.org/address/0x321b7ff75154472B18EDb199033fF4D116F340Ff#code) en Base. La asignación diaria de DIEM de la wallet es proporcional a su parte del pool de staking. La cuenta necesita al menos 0.1 DIEM acumulados antes de que cualquier DIEM sea gastable. DIEM se refresca a las 00:00 UTC. Para aumentar el gasto diario, stakea más VVV. |
| **Recarga en USD o cripto vía dashboard** | No (navegador) | Inicia sesión en [venice.ai](https://venice.ai) con la misma wallet (Sign-In-With-Ethereum), luego añade créditos en Settings, API. Tanto Stripe (tarjeta) como Coinbase (cripto) viven detrás de esa página y requieren un navegador. Los créditos no expiran. |

Para un agente que se ejecuta desatendido, **DIEM vía staking de VVV es hoy la única vía de financiación totalmente headless para una API key minteada**. Si el gasto diario del agente supera su asignación de DIEM, las opciones realistas son: stakear más VVV, o que un operador inicie sesión y recargue en USD o cripto.

### Staking autónomo de VVV y generación de clave

Un agente verdaderamente autónomo puede gestionar su propia wallet VVV en Base, stakearla y mintear su propia API key de Venice sin intervención humana. El flujo completo:

<Steps>
  <Step title="Adquiere VVV y ETH para gas">
    Envía VVV a la wallet del agente (o haz que el agente haga swap en [Aerodrome](https://aerodrome.finance) o [Uniswap](https://app.uniswap.org)), más una pequeña cantidad de ETH en Base para las dos transacciones de staking.
  </Step>
  <Step title="Stakea VVV">
    Llama a `approve` en el contrato de staking sobre el token VVV, luego `stake(amount)` en `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. El saldo de sVVV de la wallet se actualiza atómicamente con el stake.
  </Step>
  <Step title="Mintea una API key">
    `GET /api/v1/api_keys/generate_web3_key` devuelve un JWT que expira 15 minutos después de su emisión. Firma el token crudo con la wallet de staking, y luego `POST` la dirección, la firma y el token de vuelta. Venice devuelve una API key vinculada a la cuenta de usuario derivada de esa wallet.
  </Step>
</Steps>

El minting solo requiere un saldo sVVV distinto de cero, así que 1 VVV stakeado basta para recibir una clave. **Gastar** con la clave es una pregunta aparte, regida por la tabla de financiación de arriba.

Consulta [Creación autónoma de API Key para agentes](/es/guides/getting-started/generating-api-key-agent) para el recorrido completo con código y la referencia completa de errores.

## Autenticación x402 con wallet en 30 segundos

Si tu agente ya tiene una wallet de Base o Solana, sáltate la API key por completo. El SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) maneja el firmado Sign-In-With-X, las recargas y el seguimiento de saldo.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // omite si la wallet ya tiene saldo

const response = await venice.chat({
  model: 'kimi-k2-6',
  messages: [{ role: 'user', content: 'What is the latest block on Base?' }]
})
```

La misma autenticación de wallet funciona contra `/crypto/rpc/{network}` para lecturas y escrituras blockchain. Detalles completos del protocolo en la [guía x402](/es/guides/integrations/x402-venice-api).

## Precios

Crypto RPC se factura en créditos de Venice. Cada respuesta incluye `X-Venice-RPC-Credits` (créditos cobrados) y `X-Venice-RPC-Cost-USD` (coste en dólares) para que tu agente pueda rastrear el gasto por solicitud.

### Créditos base por cadena

| Créditos base | Cadenas |
|---|---|
| **20** | Ethereum, Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast, Starknet |
| **30** | zkSync Era |

### Ejemplos de coste

Precios observados para los niveles de método estándar, avanzado y grande:

| Llamada | Créditos | Coste USD |
|---|---|---|
| `eth_call` en Ethereum (20 × 1x) | 20 | ~$0.0000140 |
| `trace_transaction` en Ethereum (20 × 2x) | 40 | ~$0.0000280 |
| `trace_replayTransaction` en Ethereum (20 × 4x) | 80 | ~$0.0000560 |
| `eth_call` en zkSync (30 × 1x) | 30 | ~$0.0000210 |

Confía siempre en la cabecera de respuesta `X-Venice-RPC-Cost-USD` para el coste autoritativo. Los elementos con error en solicitudes batch se facturan a 5 créditos cada uno.

### Límites de tasa

| Nivel | Solicitudes por minuto |
|---|---|
| Estándar | 100 |
| Staff | 1,000 |

Al superarse, el endpoint devuelve `429` con cabeceras de respuesta estándar `X-RateLimit-*`.

## Manejo de errores

Respuestas HTTP comunes que tu agente debería manejar:

| Estado | Significado | Qué hacer |
|---|---|---|
| `400` | Método JSON-RPC no soportado o no mapeado, o batch malformado | Verifica el método contra la [allowlist](/es/api-reference/endpoint/crypto/rpc). El cuerpo del error nombra el método infractor. |
| `400` | Reproducción de una `Idempotency-Key` con un cuerpo distinto | Usa una clave nueva para solicitudes distintas. |
| `402` | Sin cabecera de autenticación (el cuerpo de la respuesta incluye `authOptions` listando ambas vías de autenticación soportadas), o sin créditos con una cabecera de autenticación válida | Si no hay autenticación: adjunta `Authorization: Bearer ...` o la cabecera x402 `X-Sign-In-With-X`. Si sin créditos: con una clave Bearer, financia la cuenta (DIEM, USD o recarga vía dashboard); con autenticación x402, llama directamente a `POST /api/v1/x402/top-up`. |
| `429` | Límite de tasa alcanzado (100 req/min estándar, 1.000 req/min staff) | Respeta `X-RateLimit-Reset` y haz backoff. Agrupa hasta 100 llamadas por solicitud para amortizar el límite. |
| `5xx` | Hipo del nodo RPC upstream | Reintenta con la misma `Idempotency-Key` para evitar doble cobro. |

Los errores por elemento en batch (por ejemplo, params inválidos en una de N llamadas) vuelven dentro de una respuesta `200 OK` con un campo JSON-RPC `error` en el elemento infractor. Esos elementos se facturan a 5 créditos cada uno.

## No soportado

Estas categorías de métodos se rechazan intencionalmente:

- **Solo WebSocket** (`eth_subscribe`, `eth_unsubscribe`): el proxy es solo HTTP. Haz polling en su lugar.
- **Filtros con estado** (`eth_newFilter`, `eth_getFilterChanges`, etc.): el estado del filtro está fijado a un único backend y se rompe en un proxy con balanceo de carga. Usa `eth_getLogs` en su lugar.
- **Métodos con llaves** (`eth_sign`, `eth_accounts`, `eth_mining`): los proveedores hospedados no guardan las claves del usuario. Firma del lado del cliente y envía vía `eth_sendRawTransaction`.
- **Métodos no mapeados**: cualquier cosa no permitida devuelve `400`. Contacta a soporte para solicitar adiciones.
