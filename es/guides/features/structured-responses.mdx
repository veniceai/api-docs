Venice ahora incluye salidas estructuradas mediante el campo "response\_format" disponible en la API. Este campo te permite generar respuestas a tus prompts que sigan un formato específico predefinido. Con este nuevo método, los modelos tienen menos probabilidades de inventar claves o valores incorrectos en la respuesta, algo más frecuente cuando se intentaba lograr mediante manipulación del system prompt o function calling.

El campo "response\_format" de salida estructurada utiliza el formato de la API de OpenAI y se describe con más detalle en la guía de OpenAI [aquí](https://platform.openai.com/docs/guides/structured-outputs). OpenAI también publicó un artículo de introducción al uso de salidas estructuradas en la API específicamente [aquí](https://openai.com/index/introducing-structured-outputs-in-the-api/). Como es una funcionalidad avanzada, hay algunos "detalles a tener en cuenta" al final de esta página que deberías seguir.

Esta funcionalidad no está disponible de forma nativa para todos los modelos. Consulta la sección de modelos [aquí](https://docs.venice.ai/api-reference/endpoint/models/list?playground=open) y busca "supportsResponseSchema" para los modelos aplicables.

```json
    {
      "id": "venice-uncensored",
      "type": "text",
      "object": "model",
      "created": 1726869022,
      "owned_by": "venice.ai",
      "model_spec": {
        "availableContextTokens": 32768,
        "capabilities": {
          "supportsFunctionCalling": true,
          "supportsResponseSchema": true,
          "supportsWebSearch": true
        },
```

### Cómo usar las respuestas estructuradas

Para usar correctamente "response\_format" puedes definir tu esquema con varias "properties", representando categorías de salidas, cada una con tipos de datos configurados individualmente. Estos objetos pueden anidarse para crear estructuras de salida más avanzadas.

Este es un ejemplo de una llamada a la API usando response\_format para explicar el proceso paso a paso de resolver una ecuación matemática.

Puedes ver que las properties se han configurado para requerir tanto "steps" como "final\_answer" en la respuesta. Dentro del anidamiento, la categoría steps consta de "explanation" y "output", cada una como string.

```json
curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header 'Authorization: Bearer <api-key>' \
  --header 'Content-Type: application/json' \
  --data '{
  "model": "venice-uncensored",
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful math tutor."
    },
    {
      "role": "user",
      "content": "solve 8x + 31 = 2"
    }
  ],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "math_response",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "steps": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "explanation": {
                  "type": "string"
                },
                "output": {
                  "type": "string"
                }
              },
              "required": ["explanation", "output"],
              "additionalProperties": false
            }
          },
          "final_answer": {
            "type": "string"
          }
        },
        "required": ["steps", "final_answer"],
        "additionalProperties": false
      }
    }
  }
}

```

Esta es la respuesta recibida del modelo. Puedes ver que la estructura siguió los requisitos proporcionando primero los "steps" con la "explanation" y el "output" de cada paso, y después la "final answer".

```json
{
  "steps": [
    {
      "explanation": "Subtract 31 from both sides to isolate the term with x.",
      "output": "8x + 31 - 31 = 2 - 31"
    },
    {
      "explanation": "This simplifies to 8x = -29.",
      "output": "8x = -29"
    },
    {
      "explanation": "Divide both sides by 8 to solve for x.",
      "output": "x = -29 / 8"
    }
  ],
  "final_answer": "x = -29 / 8"
}

```

Aunque este es un ejemplo sencillo, se puede extrapolar a casos de uso más avanzados como: extracción de datos, ejercicios de cadena de pensamiento, generación de UI, categorización de datos y muchos otros.

### Detalles a tener en cuenta

Estos son algunos requisitos clave que recordar al usar Structured Outputs vía response\_format:

* Las solicitudes iniciales con response\_format pueden tardar más en generar una respuesta. Las solicitudes posteriores no sufrirán la misma latencia que la inicial.

* Para consultas grandes, el modelo puede no completar si se alcanzan `max_tokens` o el timeout del modelo, o si se infringen los límites de uso.

* Un formato de esquema incorrecto provocará errores al completar, normalmente por timeout.

* Aunque response\_format asegura que el modelo emita una forma concreta, no garantiza que la información dentro sea correcta. El contenido viene determinado por el prompt y el rendimiento del modelo.

* Las salidas estructuradas vía response\_format no son compatibles con llamadas paralelas a funciones.

* Importante: todos los campos o parámetros deben incluir una etiqueta `required`. Para hacer opcional un campo, añade una opción `null` dentro del `type` del campo, así: `"type": ["string", "null"]`.

* Es posible hacer opcionales los campos dando una opción `null` dentro del campo requerido para permitir una respuesta vacía.

* Importante: `additionalProperties` debe estar en false para que response\_format funcione correctamente.

* Importante: `strict` debe estar en true para que response\_format funcione correctamente.
