El prompt caching almacena los tokens de entrada ya procesados para que las solicitudes posteriores con prefijos idénticos puedan reutilizarlos en lugar de procesarlos de nuevo. Esto reduce la latencia (hasta un 80 % en prompts largos) y los costes (hasta un 90 % de descuento sobre los tokens cacheados).

Venice gestiona la caché automáticamente para los modelos compatibles, pero comprender cómo implementa la caché cada proveedor te ayuda a maximizar la tasa de aciertos y minimizar costes.

## Cómo funciona la caché

La caché funciona por **coincidencia de prefijos**: el sistema guarda tokens procesados y los reutiliza cuando solicitudes posteriores empiezan con el mismo contenido.

Considera un chatbot con un system prompt de 2.000 tokens:

<Steps>
  <Step title="Solicitud 1">
    System prompt (2.000 tokens) + mensaje de usuario (50 tokens)
    
    **Procesados**: 2.050 tokens · **De caché**: 0 tokens
    
    Prefijo escrito en la caché.
  </Step>
  <Step title="Solicitud 2">
    System prompt (2.000 tokens) + mensaje de usuario (80 tokens)
    
    **Procesados**: 80 tokens · **De caché**: 2.000 tokens
  </Step>
  <Step title="Solicitud 3">
    System prompt (2.000 tokens) + mensaje de usuario (120 tokens)
    
    **Procesados**: 120 tokens · **De caché**: 2.000 tokens
  </Step>
</Steps>

**Total sin caché**: 2.050 + 2.080 + 2.120 = 6.250 tokens a precio completo

**Total con caché**: 2.050 + 80 + 120 = 2.250 tokens a precio completo, 4.000 tokens con tarifa de descuento

<Warning>
La caché solo funciona sobre el **prefijo**. Cualquier cambio al inicio de tu prompt invalida la caché para todo lo que viene después. Pon siempre el contenido estático (system prompt, documentos, ejemplos) antes del contenido dinámico (mensajes de usuario).
</Warning>

## Comportamiento específico del proveedor

Venice normaliza la caché entre proveedores. Para la mayoría de modelos, la caché es automática. Solo envía tus solicitudes y revisa la respuesta para ver las estadísticas de caché. **Claude** requiere marcadores de caché explícitos a nivel de protocolo, pero Venice los añade automáticamente para system prompts e historial de conversación.

El comportamiento de la caché lo controla en última instancia cada proveedor y puede cambiar, así que consulta sus docs para los últimos detalles.

### Claude Opus 4.5 (Anthropic)

Claude requiere breakpoints de caché explícitos a nivel de protocolo. Venice lo gestiona automáticamente:

- Los **system prompts** se cachean automáticamente
- El **historial de conversación** se cachea colocando un breakpoint en el penúltimo mensaje de usuario

Esto significa que el historial de conversación se lee de la caché y solo el último turno se procesa como entrada nueva:

| Turno | Tokens del prompt | Lectura de caché | Escritura de caché | Ahorro |
|------|---------------|------------|-------------|---------|
| 1 | 10.979 | 0 | 10.938 | Primera escritura |
| 2 | 11.031 | 10.938 | 31 | 99,7 % en caché |
| 3 | 11.062 | 10.969 | 52 | 99,5 % en caché |

**Detalles adicionales:**

- **Hasta 4 breakpoints por solicitud**: el sistema usa el prefijo coincidente más largo
- **La clave de caché es byte-exact**: cambios de espacios en blanco, codificaciones de imagen distintas o herramientas reordenadas rompen el acierto
- **Límites de uso conscientes de la caché**: los tokens cacheados no cuentan para tu límite ITPM, permitiendo mayor throughput efectivo
- **Premium del 25 % en escritura**: la primera solicitud cuesta más, pero hay un 90 % de ahorro en lecturas posteriores

#### Control manual de caché

Para casos especiales como cachear un documento grande en el primer turno, puedes añadir breakpoints explícitos:

```json
{
  "messages": [
    {
      "role": "system",
      "content": [{
        "type": "text",
        "text": "You are a legal assistant...",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    {
      "role": "user", 
      "content": [{
        "type": "text",
        "text": "[Long contract document...]",
        "cache_control": { "type": "ephemeral" }
      }]
    },
    { "role": "assistant", "content": "I've reviewed the contract." },
    { "role": "user", "content": "What are the termination clauses?" }
  ]
}
```

Esto asegura que tanto el system prompt como el documento se cacheen desde la primera solicitud. Para conversaciones típicas, no necesitas marcadores manuales.

### Todos los demás modelos

La caché es **automática**. No se necesitan parámetros especiales. Solo asegúrate de que tus prompts superen ~1.024 tokens y usa `prompt_cache_key` para un enrutamiento consistente.

## Parámetros de la solicitud

| Parámetro | Tipo | Modelos | Descripción |
|-----------|------|--------|-------------|
| `prompt_cache_key` | string | Todos | Pista de enrutamiento para afinidad de caché. Las solicitudes con la misma clave tienen más probabilidad de caer en el mismo servidor con caché caliente. |
| `cache_control` | object | Claude | Marca bloques de contenido para cachear. Consulta la sección de Claude Opus 4.5. |

### prompt_cache_key

Para conversaciones o flujos agénticos, usa un `prompt_cache_key` consistente para mejorar la tasa de aciertos:

```json
{
  "model": "claude-opus-4-5",
  "prompt_cache_key": "session-abc-123",
  "messages": [...]
}
```

Esto enruta las solicitudes a servidores que probablemente ya tengan tu contexto cacheado. Usa un ID de sesión, ID de conversación o ID de usuario como clave.

## Campos de la respuesta

El objeto `usage` de la respuesta incluye estadísticas de caché:

```json
{
  "usage": {
    "prompt_tokens": 5500,
    "completion_tokens": 200,
    "total_tokens": 5700,
    "prompt_tokens_details": {
      "cached_tokens": 5000,
      "cache_creation_input_tokens": 0
    }
  }
}
```

| Campo | Descripción |
|-------|-------------|
| `prompt_tokens` | Total de tokens de entrada en la solicitud |
| `prompt_tokens_details.cached_tokens` | Tokens servidos desde caché (facturados con descuento) |
| `prompt_tokens_details.cache_creation_input_tokens` | Tokens escritos en caché (pueden incurrir en premium en Claude) |

**Desglose de facturación** (usando Claude Opus 4.5 como ejemplo):
- 5.000 tokens cacheados × \$0,60/1M = \$0,003
- 500 tokens no cacheados × \$6,00/1M = \$0,003
- Total: \$0,006 (frente a \$0,033 sin caché, ahorro del 82 %)

## Mejores prácticas

### Estructura los prompts para cachear

Coloca el contenido estático al principio y el dinámico al final.

**Buena estructura**

| Posición | Contenido | ¿Cacheado? |
|----------|---------|---------|
| 1 | Instrucciones del sistema | Sí |
| 2 | Documentos de referencia | Sí |
| 3 | Ejemplos few-shot | Sí |
| 4 | Consulta del usuario | No |

**Mala estructura**

| Posición | Contenido | ¿Cacheado? |
|----------|---------|---------|
| 1 | Timestamp actual | No (invalida todo lo que sigue) |
| 2 | Instrucciones del sistema | No |
| 3 | Consulta del usuario | No |

### Mantén los prefijos byte-idénticos

Las claves de caché se calculan a partir de secuencias exactas de bytes. Incluso diferencias triviales rompen los aciertos:

- Espacios en blanco o saltos de línea distintos
- Timestamps o IDs de solicitud en los prompts
- Orden aleatorio de ejemplos few-shot
- Distinto formato del mismo contenido

### Cumple los umbrales mínimos de tokens

Si tus prompts están por debajo del mínimo (típicamente 1.024 tokens), la caché no se activará. Para prompts pequeños, considera:

- Añadir más contexto o ejemplos para llegar al umbral
- Empaquetar varias solicitudes pequeñas en prompts agrupados
- Aceptar que la caché no se aplicará para consultas simples

### Usa prompt_cache_key para conversaciones

Para conversaciones en curso, establece un `prompt_cache_key` consistente:

```json
// Turno 1
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turno 2
{ "prompt_cache_key": "conv-xyz", "messages": [...] }

// Turno 3
{ "prompt_cache_key": "conv-xyz", "messages": [...] }
```

Esto aumenta la probabilidad de que todos los turnos caigan en el mismo servidor con caché caliente.

### Monitoriza el rendimiento de la caché

Rastrea estas métricas:
- **Tasa de aciertos de caché**: `cached_tokens / prompt_tokens`
- **Ahorro de coste**: compara el coste real frente al coste sin caché
- **Reducción de latencia**: tiempo hasta el primer token con vs. sin aciertos de caché

Si `cached_tokens` es constantemente 0:
1. Los prompts pueden estar por debajo del umbral de tokens mínimo
2. Los prompts pueden estar cambiando entre solicitudes
3. Las solicitudes pueden estar cayendo en servidores distintos (usa `prompt_cache_key`)
4. La caché puede haber expirado (solicitudes demasiado infrecuentes)

### Considera la economía de la caché

**Premium de escritura de caché de Claude Opus 4.5**: la primera solicitud cuesta un 25 % más, pero hay un 90 % de ahorro en lecturas posteriores.

| Escenario | ¿Compensa el premium de escritura? |
|----------|-------------------------------|
| 1 solicitud con este prompt | No (pagas un 25 % más, sin beneficio) |
| 2+ solicitudes con el mismo prefijo | Sí (punto de equilibrio en la 2ª solicitud) |
| Prompts que cambian rápido | No (costes de escritura constantes) |
| System prompt estable, muchas consultas | Sí (amortizado en muchas lecturas) |

## Vida útil de la caché

Las cachés expiran tras un periodo de inactividad (típicamente 5-10 minutos). Esto significa:

| Patrón de tráfico | Beneficio de la caché |
|-----------------|-----------------|
| Solicitudes continuas (huecos < 5 min) | Alto: la caché permanece caliente |
| Tráfico a ráfagas (huecos > 10 min) | Limitado: la caché expira entre ráfagas |
| Solicitudes esporádicas (horas de separación) | Ninguno: la caché siempre fría |

## Caché con herramientas y funciones

Las definiciones de funciones pueden cachearse junto a los system prompts:

```json
{
  "model": "claude-opus-4-5",
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_database",
        "description": "Search the product database",
        "parameters": { ... }
      }
    }
  ],
  "messages": [
    {
      "role": "system",
      "content": [
        {
          "type": "text",
          "text": "You are a shopping assistant...",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    ...
  ]
}
```

Las definiciones de herramientas pasan a formar parte del prefijo cacheado. Si tienes muchas herramientas, esto puede reducir significativamente los costes por solicitud.

## Caché con imágenes y documentos

Para modelos de visión, las imágenes pueden incluirse en el contenido cacheado:

```json
{
  "model": "claude-opus-4-5",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": { "url": "data:image/png;base64,..." }
        },
        {
          "type": "text",
          "text": "This is the floor plan. I'll ask several questions about it.",
          "cache_control": { "type": "ephemeral" }
        }
      ]
    },
    {
      "role": "assistant",
      "content": "I can see the floor plan. What would you like to know?"
    },
    {
      "role": "user",
      "content": "How many bedrooms are there?"
    }
  ]
}
```

La imagen y el contexto inicial se cachean, así que las preguntas de seguimiento sobre la misma imagen no la reprocesan.

## Solución de problemas

<Accordion title="cached_tokens siempre es 0">

| Causa | Solución |
|-------|----------|
| Prompt demasiado corto | Asegúrate de que el prompt supere ~1.024 tokens (4.000 para Claude) |
| Prefijo cambiado | Revisa contenido dinámico al inicio de tu prompt |
| Primera solicitud | Esperado: la primera solicitud escribe en caché, las siguientes leen |
| Caché expirada | Reduce el tiempo entre solicitudes a menos de 5 minutos |
| Servidores distintos | Añade `prompt_cache_key` para enrutar consistentemente |

</Accordion>

<Accordion title="cache_creation_input_tokens en cada solicitud">

| Causa | Solución |
|-------|----------|
| Prompt cambiando | Elimina timestamps, IDs de solicitud u otro contenido dinámico del prefijo |
| Falta cache_control | Para Claude, asegúrate de que el marcador `cache_control` esté presente en los bloques de contenido |
| Por debajo del umbral | Los prompts bajo el recuento mínimo no activan la caché |
| Único mensaje de usuario | Esperado en el primer turno. La caché crece con el historial de conversación. |

</Accordion>

<Accordion title="Costes mayores de lo esperado">

| Causa | Solución |
|-------|----------|
| Premium de escritura de caché | Claude cobra un 25 % más por escrituras. Solo compensa si reutilizas el prompt. |
| Baja reutilización | Si cada prompt es único, pagas el coste de escritura sin beneficios de lectura |
| Mala estructura del prompt | Mueve el contenido dinámico al final para que el prefijo se mantenga estable |

</Accordion>
