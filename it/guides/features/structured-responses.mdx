Venice ora include gli output strutturati tramite "response\_format" come campo disponibile nell'API. Questo campo ti permette di generare risposte ai tuoi prompt che seguono un formato specifico predefinito. Con questo nuovo metodo, i modelli hanno meno probabilità di allucinare chiavi o valori errati nella risposta, cosa più frequente quando si tentava tramite manipolazione di system prompt o tramite function calling.

Il campo "response\_format" per gli output strutturati utilizza il formato dell'API OpenAI, ed è descritto più in dettaglio nella guida OpenAI [qui](https://platform.openai.com/docs/guides/structured-outputs). OpenAI ha anche pubblicato un articolo introduttivo sull'uso degli output strutturati nell'API specificamente [qui](https://openai.com/index/introducing-structured-outputs-in-the-api/). Trattandosi di funzionalità avanzata, ci sono alcune "gotcha" in fondo a questa pagina che dovresti seguire.

Questa funzionalità non è nativamente disponibile per tutti i modelli. Fai riferimento alla sezione modelli [qui](https://docs.venice.ai/api-reference/endpoint/models/list?playground=open), e cerca "supportsResponseSchema" per i modelli applicabili.

```json
    {
      "id": "venice-uncensored",
      "type": "text",
      "object": "model",
      "created": 1726869022,
      "owned_by": "venice.ai",
      "model_spec": {
        "availableContextTokens": 32768,
        "capabilities": {
          "supportsFunctionCalling": true,
          "supportsResponseSchema": true,
          "supportsWebSearch": true
        },
```

### Come usare le risposte strutturate

Per usare correttamente "response\_format" puoi definire il tuo schema con varie "properties", che rappresentano categorie di output, ciascuna con tipi di dato configurati individualmente. Questi oggetti possono essere annidati per creare strutture di output più avanzate.

Ecco un esempio di chiamata API che usa response\_format per spiegare passo-passo il processo di risoluzione di un'equazione matematica.

Puoi vedere che le properties sono state configurate per richiedere sia "steps" che "final\_answer" nella risposta. All'interno dell'annidamento, la categoria steps è composta sia da una "explanation" sia da un "output", ognuno come stringa.

```json
curl --request POST \
  --url https://api.venice.ai/api/v1/chat/completions \
  --header 'Authorization: Bearer <api-key>' \
  --header 'Content-Type: application/json' \
  --data '{
  "model": "venice-uncensored",
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful math tutor."
    },
    {
      "role": "user",
      "content": "solve 8x + 31 = 2"
    }
  ],
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "math_response",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "steps": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "explanation": {
                  "type": "string"
                },
                "output": {
                  "type": "string"
                }
              },
              "required": ["explanation", "output"],
              "additionalProperties": false
            }
          },
          "final_answer": {
            "type": "string"
          }
        },
        "required": ["steps", "final_answer"],
        "additionalProperties": false
      }
    }
  }
}

```

Ecco la risposta ricevuta dal modello. Puoi vedere che la struttura ha seguito i requisiti, fornendo prima gli "steps" con "explanation" e "output" di ciascun passo, e poi la "final answer".

```json
{
  "steps": [
    {
      "explanation": "Subtract 31 from both sides to isolate the term with x.",
      "output": "8x + 31 - 31 = 2 - 31"
    },
    {
      "explanation": "This simplifies to 8x = -29.",
      "output": "8x = -29"
    },
    {
      "explanation": "Divide both sides by 8 to solve for x.",
      "output": "x = -29 / 8"
    }
  ],
  "final_answer": "x = -29 / 8"
}

```

Sebbene questo sia un esempio semplice, può essere esteso a casi d'uso più avanzati come: estrazione di dati, esercizi di chain-of-thought, generazione di UI, categorizzazione di dati e molti altri.

### Gotcha

Ecco alcuni requisiti chiave da tenere a mente quando si usano gli output strutturati tramite response\_format:

* Le richieste iniziali che usano response\_format potrebbero richiedere più tempo per generare una risposta. Le richieste successive non subiranno la stessa latenza della richiesta iniziale.

* Per query più grandi, il modello può non riuscire a completare se vengono raggiunti `max_tokens` o il timeout del modello, o se vengono violati i rate limit.

* Un formato di schema errato genererà errori al completamento, di solito per timeout.

* Anche se response\_format garantisce che il modello produrrà output in un modo particolare, non garantisce che il modello fornisca informazioni corrette al suo interno. Il contenuto è guidato dal prompt e dalle prestazioni del modello.

* Gli output strutturati tramite response\_format non sono compatibili con le chiamate di funzione parallele.

* Importante: Tutti i campi o parametri devono includere un tag `required`. Per rendere un campo opzionale devi aggiungere un'opzione `null` all'interno del `type` del campo, così: `"type": ["string", "null"]`&#x20;

* È possibile rendere i campi opzionali fornendo un'opzione `null` all'interno del campo required per permettere una risposta vuota.

* Importante: `additionalProperties` deve essere impostato a false perché response\_format funzioni correttamente.

* Importante: `strict` deve essere impostato a true perché response\_format funzioni correttamente.
