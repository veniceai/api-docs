## Perché Venice per agenti on-chain?

| Capacità | Cosa ottiene il tuo agente |
|---|---|
| **Inferenza** | 230+ modelli di testo, immagini, video, audio ed embedding tramite un endpoint compatibile OpenAI |
| **Crypto RPC** | Proxy JSON-RPC 2.0 verso 10 catene EVM più Starknet (mainnet e testnet) |
| **Autenticazione** | API key standard o autenticazione wallet x402 (nessun account Venice richiesto) |
| **Finanziamento** | Autonomo: staking VVV per DIEM giornaliero. Browser: top-up USD o crypto via dashboard |
| **Batching** | Fino a 100 chiamate JSON-RPC per richiesta, multi-chain in parallelo |
| **Idempotenza** | Retry sicuri con header `Idempotency-Key` |

## Autenticazione

Scegli il metodo di autenticazione che corrisponde al modo in cui il tuo agente viene eseguito.

| Metodo | Ideale per | Come funziona |
|---|---|---|
| **API key** | Agenti server-side, deployment fissi | Header `Authorization: Bearer <key>`. Ottieni una key da [venice.ai/settings/api](https://venice.ai/settings/api). |
| **Wallet x402** | Agenti autonomi, crypto-nativi o a breve durata | Il wallet firma un messaggio Sign-In-With-X, paga per richiesta in USDC su Base o Solana. Nessun account Venice richiesto. Vedi la [guida x402](/it/guides/integrations/x402-venice-api). |

Entrambi i metodi condividono gli stessi rate limit e la stessa fatturazione in crediti Venice.

<Tip>
Gli agenti veramente autonomi possono coniare la propria API key mettendo VVV in staking su Base. Vedi [Creazione autonoma di API key per agenti](/it/guides/getting-started/generating-api-key-agent).
</Tip>

## Quickstart Crypto RPC

Invia qualsiasi metodo JSON-RPC 2.0 a `POST /crypto/rpc/{network}`.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

Risposta:

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Gli header di risposta includono `X-Venice-RPC-Credits` (crediti addebitati), `X-Venice-RPC-Cost-USD` (costo in dollari) e `X-Request-ID` (ID di correlazione).

### Fasce di metodi

I metodi sono raggruppati in tre fasce di credito. Costo totale = `baseCredits[chain] × methodTier`.

| Fascia | Moltiplicatore | Esempi |
|---|---|---|
| **Standard** | 1x | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas` |
| **Advanced** | 2x | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction` |
| **Large** | 4x | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content` |

Elenco completo e dettagli sui prezzi nel [riferimento API Crypto RPC](/it/api-reference/endpoint/crypto/rpc).

## Ricette per agenti

Pattern comuni per agenti AI che devono leggere e scrivere on-chain.

### Leggere il saldo nativo di un wallet

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getBalance",
    "params": ["0xYourWalletAddress", "latest"],
    "id": 1
  }'
```

### Leggere il saldo di un token ERC-20

Chiama il selettore `balanceOf(address)` con `eth_call`. Il campo `data` è il selettore di 4 byte (`0x70a08231`) seguito dall'indirizzo del wallet left-padded a 32 byte. Più semplice farlo codificare da una libreria:

```typescript
import { encodeFunctionData, parseAbi } from 'viem'

const data = encodeFunctionData({
  abi: parseAbi(['function balanceOf(address) view returns (uint256)']),
  args: ['0xWalletAddress'],
})

const response = await fetch('https://api.venice.ai/api/v1/crypto/rpc/base-mainnet', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${process.env.VENICE_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    jsonrpc: '2.0',
    method: 'eth_call',
    params: [{ to: '0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf', data }, 'latest'],
    id: 1,
  }),
})
```

L'indirizzo del contratto sopra è VVV su Base. Sostituiscilo con qualsiasi contratto ERC-20.

### Inviare una transazione firmata (ciclo completo)

Venice non detiene mai le tue chiavi private. L'agente raccoglie i parametri della tx tramite letture RPC, firma localmente con una libreria come [viem](https://viem.sh) o [ethers](https://docs.ethers.org), poi inoltra l'hex raw attraverso Venice.

<Steps>
  <Step title="Ottieni il prossimo nonce">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionCount","params":["0xAgentWallet","pending"],"id":1}'
    ```
    Usa `"pending"` in modo che invii consecutivi non collidano.
  </Step>
  <Step title="Ottieni il gas price">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_gasPrice","params":[],"id":1}'
    ```
    Per le catene EIP-1559, preferisci `eth_feeHistory` per calcolare `maxFeePerGas` e `maxPriorityFeePerGas`.
  </Step>
  <Step title="Stima il gas">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"from":"0xAgentWallet","to":"0xRecipient","value":"0x0","data":"0x..."}],"id":1}'
    ```
  </Step>
  <Step title="Firma localmente">
    ```typescript
    import { privateKeyToAccount } from 'viem/accounts'
    import { base } from 'viem/chains'

    const account = privateKeyToAccount(process.env.AGENT_PRIVATE_KEY)

    const signed = await account.signTransaction({
      chainId: base.id,
      nonce,                  // from step 1
      gas,                    // from step 3
      maxFeePerGas,           // from step 2 (fee history)
      maxPriorityFeePerGas,   // from step 2 (fee history)
      to: '0xRecipient',
      value: 0n,
      data: '0x...',
    })
    ```
  </Step>
  <Step title="Invia tramite Venice">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Idempotency-Key: agent-tx-<id>" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_sendRawTransaction","params":["0xSignedHex"],"id":1}'
    ```
    Imposta sempre `Idempotency-Key` sui relay in modo che un blip di rete non possa fare doppio broadcast.
  </Step>
  <Step title="Polling per la receipt">
    ```bash
    curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
      -H "Authorization: Bearer $VENICE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xTxHash"],"id":1}'
    ```
    Poll ogni pochi secondi finché `result` non è non-null. Controlla `result.status` (`"0x1"` = success).
  </Step>
</Steps>

<Note>
Ogni chiamata `eth_sendRawTransaction` viene registrata lato server con tx hash, rete, request ID e ID dell'utente chiamante. Il payload firmato stesso non viene conservato. Questa traccia di audit esiste in modo che chiavi compromesse usate per relay illeciti possano essere tracciate fino all'account responsabile.
</Note>

### Batch di più chiamate (controllo portfolio multi-chain)

Invia fino a 100 oggetti JSON-RPC in una sola richiesta. Ciascuno è validato e fatturato indipendentemente.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[
    { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 1 },
    { "jsonrpc": "2.0", "method": "eth_getBalance", "params": ["0xWallet", "latest"], "id": 2 },
    { "jsonrpc": "2.0", "method": "eth_gasPrice", "params": [], "id": 3 }
  ]'
```

Per letture multi-chain (una chiamata per catena), invia richieste parallele a diversi endpoint `{network}`.

### Retry sicuri con idempotenza

Imposta l'header `Idempotency-Key` su qualsiasi stringa che corrisponda a `[A-Za-z0-9_-]{1,255}`. Venice mette in cache la risposta per 24 ore con chiave `(user, key)`. I replay restituiscono il risultato in cache con `Idempotent-Replayed: true` e non addebitano nulla.

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/base-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Idempotency-Key: agent-tx-2026-04-21-001" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_sendRawTransaction",
    "params": ["0xSignedRawTxHex"],
    "id": 1
  }'
```

È critico per i relay di transazioni dove un blip di rete potrebbe altrimenti far sì che il tuo agente trasmetta due volte la stessa tx.

## Finanziare l'API key dell'agente

Una volta che l'agente ha una API key Venice, ha bisogno di saldo spendibile sull'account sottostante prima che gli endpoint a pagamento accettino la key. Ci sono due modi per mettere saldo lì:

| Percorso | Autonomo? | Come funziona |
|---|---|---|
| **DIEM da staking VVV** | Sì | Metti VVV in staking nello [Smart Contract di Staking Venice](https://basescan.org/address/0x321b7ff75154472B18EDb199033fF4D116F340Ff#code) su Base. L'allocazione giornaliera di DIEM del wallet è proporzionale alla sua quota nel pool di staking. L'account deve aver accumulato almeno 0.1 DIEM prima che qualsiasi DIEM sia spendibile. DIEM si aggiorna alle 00:00 UTC. Per aumentare la spesa giornaliera, metti più VVV in staking. |
| **Top-up USD o crypto via dashboard** | No (browser) | Accedi a [venice.ai](https://venice.ai) con lo stesso wallet (Sign-In-With-Ethereum), poi aggiungi crediti in Settings, API. Sia Stripe (carta) sia Coinbase (crypto) vivono dietro quella pagina e richiedono un browser. I crediti non scadono mai. |

Per un agente che gira non presidiato, **DIEM via staking VVV è oggi l'unica via di finanziamento completamente headless per una API key coniata**. Se la spesa giornaliera dell'agente supera l'allocazione DIEM, le opzioni realistiche sono: mettere più VVV in staking, o far accedere un operatore e fare top-up in USD o crypto.

### Staking VVV autonomo e generazione key

Un agente veramente autonomo può gestire il proprio wallet VVV su Base, metterlo in staking e coniare la propria API key Venice senza intervento umano. Il flusso completo:

<Steps>
  <Step title="Acquisisci VVV ed ETH per il gas">
    Invia VVV al wallet dell'agente (o fagli eseguire uno swap su [Aerodrome](https://aerodrome.finance) o [Uniswap](https://app.uniswap.org)), più una piccola quantità di ETH su Base per le due transazioni di staking.
  </Step>
  <Step title="Stake VVV">
    `approve` il contratto di staking sul token VVV, poi `stake(amount)` su `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. Il saldo sVVV del wallet si aggiorna atomicamente con lo stake.
  </Step>
  <Step title="Conia una API key">
    `GET /api/v1/api_keys/generate_web3_key` restituisce un JWT che scade 15 minuti dopo l'emissione. Firma il token raw con il wallet di staking, poi `POST` address, signature e token indietro. Venice restituisce una API key legata all'account utente derivato da quel wallet.
  </Step>
</Steps>

Il mint richiede solo un saldo sVVV non nullo, quindi 1 VVV in staking è sufficiente per ricevere una key. La **spesa** con la key è una questione separata, regolata dalla tabella di finanziamento sopra.

Vedi [Creazione autonoma di API key per agenti](/it/guides/getting-started/generating-api-key-agent) per la procedura completa con codice e l'intero riferimento errori.

## Auth wallet x402 in 30 secondi

Se il tuo agente ha già un wallet Base o Solana, salta del tutto l'API key. L'SDK [`venice-x402-client`](https://github.com/veniceai/x402-client) gestisce firma Sign-In-With-X, top-up e tracciamento del saldo.

```bash
npm install venice-x402-client
```

```typescript
import { VeniceClient } from 'venice-x402-client'

const venice = new VeniceClient(process.env.WALLET_KEY)

await venice.topUp(10) // skip if the wallet already has balance

const response = await venice.chat({
  model: 'kimi-k2-6',
  messages: [{ role: 'user', content: 'What is the latest block on Base?' }]
})
```

La stessa autenticazione wallet funziona su `/crypto/rpc/{network}` per letture e scritture blockchain. Dettagli completi del protocollo nella [guida x402](/it/guides/integrations/x402-venice-api).

## Prezzi

Crypto RPC è fatturato in crediti Venice. Ogni risposta include `X-Venice-RPC-Credits` (crediti addebitati) e `X-Venice-RPC-Cost-USD` (costo in dollari) in modo che il tuo agente possa tracciare la spesa per richiesta.

### Crediti base per catena

| Crediti base | Catene |
|---|---|
| **20** | Ethereum, Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast, Starknet |
| **30** | zkSync Era |

### Esempi di costo

Prezzi osservati per le fasce di metodo standard, advanced e large:

| Chiamata | Crediti | Costo USD |
|---|---|---|
| `eth_call` su Ethereum (20 × 1x) | 20 | ~$0.0000140 |
| `trace_transaction` su Ethereum (20 × 2x) | 40 | ~$0.0000280 |
| `trace_replayTransaction` su Ethereum (20 × 4x) | 80 | ~$0.0000560 |
| `eth_call` su zkSync (30 × 1x) | 30 | ~$0.0000210 |

Affidati sempre all'header di risposta `X-Venice-RPC-Cost-USD` per il costo autoritativo. Gli elementi in errore nelle richieste batch sono fatturati a un flat di 5 crediti ciascuno.

### Rate limit

| Fascia | Richieste al minuto |
|---|---|
| Standard | 100 |
| Staff | 1.000 |

Quando superati, l'endpoint restituisce `429` con gli header di risposta standard `X-RateLimit-*`.

## Gestione degli errori

Risposte HTTP comuni che il tuo agente dovrebbe gestire:

| Stato | Significato | Cosa fare |
|---|---|---|
| `400` | Metodo JSON-RPC non supportato o non mappato, o batch malformato | Verifica il metodo contro l'[allowlist](/it/api-reference/endpoint/crypto/rpc). Il body dell'errore nomina il metodo problematico. |
| `400` | Replay di un `Idempotency-Key` con body diverso | Usa una key nuova per richieste distinte. |
| `402` | Nessun header di autenticazione (il body della risposta include `authOptions` che elenca entrambi i percorsi di auth supportati), o crediti esauriti con un header di auth valido | Se nessun auth: allega `Authorization: Bearer ...` o l'header `X-Sign-In-With-X` x402. Se crediti esauriti: con una Bearer key, finanzia l'account (DIEM, USD o top-up dashboard); con auth x402, chiama direttamente `POST /api/v1/x402/top-up`. |
| `429` | Rate limit raggiunto (100 req/min standard, 1.000 req/min staff) | Onora `X-RateLimit-Reset` e fai backoff. Fai batch fino a 100 chiamate per richiesta per ammortizzare il limite. |
| `5xx` | Problema del nodo RPC upstream | Ritenta con lo stesso `Idempotency-Key` per evitare doppio addebito. |

Gli errori per singolo elemento di batch (es. params invalidi su una delle N chiamate) tornano dentro una risposta `200 OK` con un campo `error` JSON-RPC sull'elemento problematico. Questi elementi sono fatturati a un flat di 5 crediti ciascuno.

## Non supportati

Queste categorie di metodi sono intenzionalmente rifiutate:

- **Solo WebSocket** (`eth_subscribe`, `eth_unsubscribe`): il proxy è solo HTTP. Esegui invece il polling.
- **Filtri stateful** (`eth_newFilter`, `eth_getFilterChanges`, ecc.): lo stato dei filtri è legato a un singolo backend e si rompe su un proxy load-balanced. Usa invece `eth_getLogs`.
- **Metodi che detengono chiavi** (`eth_sign`, `eth_accounts`, `eth_mining`): i provider hosted non detengono chiavi utente. Firma lato client e invia con `eth_sendRawTransaction`.
- **Metodi non mappati**: qualsiasi cosa non in allowlist restituisce `400`. Contatta il supporto per richiedere aggiunte.
