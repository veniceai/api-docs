Un agente AI che controlla un wallet su Base puÃ² coniare la propria API key Venice senza alcun intervento umano. L'agente acquisisce VVV, lo mette in staking, firma un token di validazione a breve durata emesso da Venice e invia il token firmato per ricevere una nuova API key legata al wallet di staking.

Questa guida illustra l'intero flusso end-to-end e copre le opzioni di finanziamento per pagare effettivamente l'inferenza una volta coniata la key.

## Prerequisiti

* Un wallet EVM su Base controllato dall'agente (chiave privata in una variabile d'ambiente o secret manager).
* Una piccola quantitÃ  di ETH su Base per il gas (lo staking sono due transazioni: `approve` poi `stake`).
* Una quantitÃ  non nulla di VVV da mettere in staking. L'endpoint di mint richiede solo che il wallet abbia un saldo sVVV non nullo, quindi 1 VVV Ã¨ sufficiente per coniare una key. Vedi [Pagare l'inferenza](#paying-for-inference) per cosa serve effettivamente per chiamare endpoint a pagamento.

<Tip>
  Usa un wallet dedicato per l'agente piuttosto che un wallet di tesoreria. La chiave privata del wallet firma ogni richiesta di token a Venice, quindi il suo blast radius dovrebbe essere piccolo.
</Tip>

## Passi

<Steps>
  <Step title="Acquisisci VVV">
    Invia VVV al wallet dell'agente, oppure fai eseguire all'agente uno swap su un DEX come [Aerodrome](https://aerodrome.finance/swap?from=eth\&to=0xacfe6019ed1a7dc6f7b508c02d1b04ec88cc21bf\&chain0=8453\&chain1=8453) o [Uniswap](https://app.uniswap.org/swap?chain=base\&inputCurrency=NATIVE\&outputCurrency=0xacfe6019ed1a7dc6f7b508c02d1b04ec88cc21bf).

    Contratto del token VVV su Base: `0xacfE6019Ed1A7Dc6f7B508C02d1b04ec88cC21bf`
  </Step>

  <Step title="Metti VVV in staking con Venice">
    Metti il VVV in staking nello [Smart Contract di Staking Venice](https://basescan.org/address/0x321b7ff75154472b18edb199033ff4d116f340ff#code) all'indirizzo `0x321b7ff75154472B18EDb199033fF4D116F340Ff`. Sono due transazioni:

    1. `approve(spender, amount)` sul token VVV, dove `spender` Ã¨ il contratto di staking.
    2. `stake(amount)` sul contratto di staking.

    <Frame as="div">
      <img src="/images/guides/SC-Stake.png" alt="Staking nello Smart Contract" />
    </Frame>

    Quando la seconda transazione viene confermata, il saldo VVV del wallet diminuisce e il saldo sVVV aumenta dello stesso importo. L'endpoint di mint legge il saldo sVVV per confermare che il wallet Ã¨ in staking.
  </Step>

  <Step title="Richiedi un token di validazione">
    Chiama `GET /api/v1/api_keys/generate_web3_key` per ottenere un token a breve durata firmato da Venice. L'endpoint non Ã¨ autenticato.

    ```bash
    curl --request GET \
      --url https://api.venice.ai/api/v1/api_keys/generate_web3_key
    ```

    La risposta contiene un campo `token`. Il token scade 15 minuti dopo l'emissione, quindi firmalo e invialo molto prima.
  </Step>

  <Step title="Firma il token con il wallet di staking">
    Firma la stringa raw del token con il wallet che detiene il VVV in staking. Ãˆ un `personal_sign` standard sui byte del token. Sia `ethers.Wallet.signMessage(token)` sia `account.signMessage({ message: token })` di `viem` producono la firma corretta.
  </Step>

  <Step title="Conia l'API key">
    Esegui `POST` con address, signature e token allo stesso endpoint, insieme al tipo di key che vuoi.

    ```bash
    curl --request POST \
      --url https://api.venice.ai/api/v1/api_keys/generate_web3_key \
      --header 'Content-Type: application/json' \
      --data '{
        "address": "<wallet address>",
        "signature": "<signed token>",
        "token": "<unsigned token>",
        "apiKeyType": "INFERENCE",
        "description": "Agent key minted on <date>"
      }'
    ```

    Campi obbligatori: `address`, `signature`, `token`, `apiKeyType` (`INFERENCE` o `ADMIN`).

    Campi opzionali: `description`, `expiresAt`, `consumptionLimit` (limita la spesa totale su questa key, denominata in `usd`, `vcu` o `diem`).

    In caso di successo la risposta contiene la stringa `apiKey` coniata. Salvala nel secret store dell'agente e usala come un normale Bearer token (`Authorization: Bearer <key>`).
  </Step>
</Steps>

## Esempio end-to-end

L'esempio sotto usa un wallet reale da una variabile d'ambiente invece di uno generato a caso. Un wallet casuale non ha VVV in staking e il mint verrÃ  rifiutato con l'errore `Wallet has no staked VVV on Base`.

```typescript
import { ethers } from "ethers"

const wallet = new ethers.Wallet(process.env.WALLET_PRIVATE_KEY!)
const address = wallet.address

const tokenResponse = await fetch("https://api.venice.ai/api/v1/api_keys/generate_web3_key")
const { data: { token } } = await tokenResponse.json()

const signature = await wallet.signMessage(token)

const mintResponse = await fetch("https://api.venice.ai/api/v1/api_keys/generate_web3_key", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    address,
    signature,
    token,
    apiKeyType: "INFERENCE",
    description: "Agent key",
  }),
})

const result = await mintResponse.json()
if (!mintResponse.ok) {
  throw new Error(`Mint failed: ${result.error}`)
}

console.log("Minted key:", result.data.apiKey)
```

## Riferimento errori

L'endpoint restituisce messaggi di errore specifici e azionabili. Mappali nell'agente in modo che possa decidere se ritentare, richiedere un nuovo token o fermarsi.

| Stato | Il messaggio di errore contiene | Cosa significa | Cosa fare |
| --- | --- | --- | --- |
| `400` | `Invalid wallet address` | Il campo `address` non Ã¨ un indirizzo EVM valido. | Correggi l'indirizzo e reinvia. |
| `400` | `JWT has expired` | Il token di validazione Ã¨ scaduto prima della firma e dell'invio. | Richiedi un nuovo token, firmalo e invialo immediatamente. |
| `400` | `JWT signature is invalid` | Il token non Ã¨ stato firmato da Venice (probabilmente manomesso o fabbricato). | Usa sempre un token fresco dall'endpoint `GET`. |
| `400` | `JWT claims are invalid` | L'issuer o l'audience del token non corrisponde a quanto Venice si aspetta. | Usa il token non modificato restituito dall'endpoint `GET`. |
| `400` | `JWT is malformed` | Il `token` inviato non Ã¨ un JWT. | Assicurati di inviare esattamente la stringa `token` restituita dall'endpoint `GET`. |
| `400` | `Wallet signature does not match` | La `signature` non corrisponde all'`address` per il dato `token`. | Firma i byte raw del token con il wallet che possiede `address`. |
| `400` | `Could not verify wallet signature` | La chiamata RPC per verificare la firma Ã¨ fallita (transitorio). | Ritenta con backoff. |
| `400` | `Wallet has no staked VVV on Base` | Il wallet ha saldo sVVV pari a zero. | Metti prima VVV in staking, poi ritenta. |

## Pagare l'inferenza

Coniare una key ed essere in grado di chiamare endpoint a pagamento sono due cose separate. Una key appena coniata autentica correttamente ma non puÃ² chiamare endpoint a pagamento (come `/chat/completions`) finchÃ© l'account del wallet non ha un saldo spendibile.

La key coniata puÃ² spendere dall'account utente in questo ordine di prioritÃ : DIEM, poi crediti bundled, poi USD.

| Fonte di finanziamento | Autonoma? | Come |
| --- | --- | --- |
| **DIEM dallo staking VVV** | SÃ¬ | L'allocazione giornaliera di DIEM del wallet Ã¨ proporzionale alla sua quota nel pool di staking. L'account ha bisogno di almeno 0.1 DIEM in staking perchÃ© il DIEM sia spendibile. Stake piÃ¹ grandi guadagnano proporzionalmente piÃ¹ DIEM giornaliero, aggiornato ogni epoca (00:00 UTC). |
| **USD via Stripe** | No (browser) | Accedi a venice.ai con lo stesso wallet (Sign-In-With-Ethereum). La dashboard trova il record utente esistente. Aggiungi crediti in Settings, API. |
| **Abbonamento crypto Coinbase** | No (browser) | Stesso login con wallet, poi sottoscrivi tramite la dashboard. Il flusso reindirizza a Coinbase Commerce per il pagamento effettivo, quindi non puÃ² essere pilotato da uno script. |
| **Onramp Coinbase** | No (browser) | Stesso login con wallet, poi usa il widget di onramp nella dashboard. Ospitato sull'UI di Coinbase. |

Se l'agente ha bisogno di un percorso di finanziamento completamente crypto-nativo e headless, le opzioni piÃ¹ pulite sono:

1. **Mettere in staking piÃ¹ VVV** in modo che l'allocazione giornaliera di DIEM copra la spesa dell'agente. La key coniata lo recupera automaticamente.
2. **Usare il [flusso wallet x402](/it/guides/integrations/x402-venice-api) invece dell'API key.** Con x402 l'agente firma un messaggio Sign-In-With-X per richiesta, ricarica direttamente con USDC su Base o Solana tramite `POST /api/v1/x402/top-up`, e paga per richiesta. Il saldo USDC x402 Ã¨ legato al wallet, non all'utente, quindi non appare come saldo per la Bearer key coniata, ma permette allo stesso wallet di pagare l'inferenza in modo programmatico.
