I rate limit variano per modello e tier. I limiti predefiniti riportati sotto sono un riferimento utile, ma l'endpoint API `/api_keys/rate_limits` è il modo canonico per recuperare i tuoi limiti correnti. Puoi controllare i tuoi limiti esatti in qualsiasi momento:

<CardGroup cols={2}>
  <Card title="Visualizza i tuoi limiti" icon="gauge-high" href="/it/api-reference/endpoint/api_keys/rate_limits?playground=open">
    Playground interattivo
  </Card>
  <Card title="Log dei rate limit" icon="clock-rotate-left" href="/it/api-reference/endpoint/api_keys/rate_limit_logs?playground=open">
    Vedi quali richieste hanno raggiunto i limiti
  </Card>
</CardGroup>

```bash
curl https://api.venice.ai/api/v1/api_keys/rate_limits \
  -H "Authorization: Bearer $VENICE_API_KEY"
```

## Gestione degli errori

Le richieste fallite (500, 503, 429) dovrebbero essere ripetute con backoff esponenziale.

Per gli errori 429 in particolare, controlla l'header `x-ratelimit-reset-requests` per il timestamp Unix esatto in cui puoi ritentare. La maggior parte delle librerie HTTP ha meccanismi di retry integrati che gestiscono questo automaticamente.

### Protezione anti-abuso

Se generi più di 20 richieste fallite in 30 secondi, l'API bloccherà ulteriori richieste per 30 secondi:

```
Too many failed attempts (> 20) resulting in a non-success status code. Please wait 30s and try again.
```

## Header di risposta

Ogni risposta include questi header:

| Header | Descrizione |
|:-------|:------------|
| `x-ratelimit-limit-requests` | Massimo numero di richieste consentite nella finestra corrente |
| `x-ratelimit-remaining-requests` | Richieste rimanenti nella finestra corrente |
| `x-ratelimit-reset-requests` | Timestamp Unix di quando la finestra si resetta |
| `x-ratelimit-limit-tokens` | Massimo numero di token consentiti al minuto |
| `x-ratelimit-remaining-tokens` | Token rimanenti nel minuto corrente |
| `x-ratelimit-reset-tokens` | Secondi fino al reset del limite di token |
