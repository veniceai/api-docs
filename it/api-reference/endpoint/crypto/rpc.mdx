Inoltra una richiesta JSON-RPC 2.0 (singola o batch) a un nodo blockchain supportato. Supporta sia l'autenticazione tramite API key sia tramite wallet x402. La fatturazione è a credito, denominata sul tuo saldo Venice — una sola credenziale, una sola fattura, tutte le catene elencate sotto.

## Autenticazione

Questo endpoint supporta due metodi di autenticazione:

- **API Key**: Autenticazione standard con Bearer token tramite l'header `Authorization: Bearer <key>`.
- **Wallet x402**: Pay-as-you-go con crediti USDC da un wallet su Base o Solana. Non è richiesto un account Venice. Consulta la [guida x402](/it/guides/integrations/x402-venice-api) per la configurazione.

Entrambi i metodi condividono gli stessi rate limit e la stessa fatturazione (crediti Venice).

## Strutture di richiesta

### Richiesta singola

```json
{
  "jsonrpc": "2.0",
  "method": "eth_chainId",
  "params": [],
  "id": 1
}
```

### Richiesta batch

Un array di massimo **100** oggetti JSON-RPC 2.0. Ogni elemento viene validato in modo indipendente; se un metodo non è supportato, l'intero batch viene rifiutato con `400` e ogni nome di metodo non valido è elencato nel messaggio di errore.

```json
[
  { "jsonrpc": "2.0", "method": "eth_chainId", "params": [], "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_blockNumber", "params": [], "id": 2 }
]
```

## Metodi supportati e tariffe a credito

I metodi sono classificati in tre fasce di credito. Crediti consumati per chiamata = `baseCredits[chain] × methodTier`.

| Fascia | Moltiplicatore | Metodi di esempio |
|---|---|---|
| **Standard** | 1× | `eth_call`, `eth_getBalance`, `eth_blockNumber`, `eth_sendRawTransaction`, `eth_getLogs`, `eth_getTransactionReceipt`, `eth_estimateGas`, `net_version`, `web3_clientVersion`, metodi bundler ERC-4337 (`eth_sendUserOperation`, `eth_estimateUserOperationGas`, ecc.), estensioni delle famiglie di catene (`zks_*`, `linea_*`, `bor_*`, `starknet_*`) |
| **Advanced** | 2× | `trace_block`, `trace_call`, `trace_transaction`, `debug_traceCall`, `debug_traceTransaction`, `debug_traceBlockByHash`, `txpool_inspect`, `txpool_status`, `arbtrace_*` |
| **Large** | 4× | `trace_replayBlockTransactions`, `trace_replayTransaction`, `txpool_content`, `arbtrace_replayTransaction`, `arbtrace_replayBlockTransactions` |

### Crediti base per catena

| baseCredits | Catene |
|---|---|
| **20** | Ethereum + tutti gli L2 EVM sopra (Base, Optimism, Arbitrum, Polygon, Linea, Avalanche, BSC, Blast) e Starknet |
| **30** | zkSync Era |

### Esempi di costo

Al prezzo Venice di `~$6,25 × 10⁻⁷` per credito:

| Chiamata | Crediti | Costo USD |
|---|---|---|
| `eth_call` su Ethereum (20 × 1×) | 20 | $0,0000125 |
| `trace_transaction` su Ethereum (20 × 2×) | 40 | $0,0000250 |
| `trace_replayTransaction` su Ethereum (20 × 4×) | 80 | $0,0000500 |
| `eth_call` su zkSync (30 × 1×) | 30 | $0,0000188 |

### Non supportati

- **Metodi solo WebSocket** (`eth_subscribe`, `eth_unsubscribe`) — questo proxy è solo HTTP. Esegui il polling oppure passa a un provider WebSocket diretto.
- **Metodi stateful per i filtri** (`eth_newFilter`, `eth_getFilterChanges`, `eth_getFilterLogs`, `eth_uninstallFilter`, `eth_newBlockFilter`, `eth_newPendingTransactionFilter`) — lo stato dei filtri è legato a un singolo backend upstream e si interrompe silenziosamente su un proxy HTTP load-balanced. Usa invece `eth_getLogs` (stateless).
- **Metodi miner / con chiavi private** (`eth_sign`, `eth_accounts`, `eth_mining`, `eth_hashrate`, `eth_getWork`, `eth_submitWork`) — gli endpoint dei provider ospitati non detengono le chiavi private degli utenti, quindi questi vanno sempre in errore. Firma le transazioni lato client e invia con `eth_sendRawTransaction`.
- **Metodi non mappati** — qualsiasi metodo non esplicitamente in allowlist restituisce `400`. Contatta il supporto per richiedere aggiunte.

## Fatturazione per singolo elemento nei batch

Anche quando la risposta HTTP è `200`, i singoli elementi del batch possono tornare con un campo `error` JSON-RPC (per esempio un errore di parametri non validi o un metodo non supportato sulla catena di destinazione). Venice fattura questi elementi a **5 crediti ciascuno** invece della tariffa piena del metodo — una piccola concessione per i tipici errori di "esplorazione dell'API".

```json
// Richiesta batch:
[
  { "jsonrpc": "2.0", "method": "eth_chainId",   "params": [],         "id": 1 },
  { "jsonrpc": "2.0", "method": "eth_getBalance","params": ["bad"],    "id": 2 }
]

// Risposta (HTTP 200, X-Venice-RPC-Credits: 25):
[
  { "jsonrpc": "2.0", "id": 1, "result": "0x1" },
  { "jsonrpc": "2.0", "id": 2, "error": { "code": -32602, "message": "invalid params" } }
]
```

Il primo elemento (success) fattura 20 crediti, il secondo (errore a livello RPC) ne fattura 5, totale `= 25`.

## Rate limit

Limite di richieste al minuto per chiamante autenticato:

| Fascia | Richieste/minuto |
|---|---|
| Standard | 100 |
| Staff | 1.000 |

Quando il limite viene superato, l'endpoint restituisce `429` con un `customMessage` e gli header di risposta standard `X-RateLimit-*`.

## Idempotenza

Imposta l'header di richiesta `Idempotency-Key` su qualsiasi stringa che corrisponda a `[A-Za-z0-9_-]{1,255}` per abilitare i retry sicuri. La risposta viene memorizzata in cache per 24 ore con chiave `(user, idempotency-key)`:

- Riproporre la **stessa key con lo stesso body** restituisce la risposta in cache e un header di risposta `Idempotent-Replayed: true`. L'upstream non viene toccato e non vengono addebitati nuovi crediti.
- Riproporre la **stessa key con un body diverso** restituisce `400` per evitare una corruzione silenziosa dello stato. Scegli una nuova key per richieste distinte.

## Header di risposta

| Header | Descrizione |
|---|---|
| `X-Venice-RPC-Credits` | Crediti addebitati per questa richiesta. Nei batch è la somma degli elementi. |
| `X-Venice-RPC-Cost-USD` | Costo in dollari a 8 decimali. Pari a `X-Venice-RPC-Credits × prezzo per credito`. |
| `X-Request-ID` | ID di correlazione di 32 caratteri. Includilo in qualsiasi corrispondenza con il supporto. |
| `Idempotent-Replayed` | Presente con valore `"true"` quando la risposta proviene dalla cache di idempotenza. |
| `X-RateLimit-Limit` / `X-RateLimit-Remaining` / `X-RateLimit-Reset` | Impostati solo sulle risposte 429. |

## Logging forensico per i relay di transazioni

Ogni chiamata a `eth_sendRawTransaction` viene registrata lato server con l'hash della tx (keccak256 dei byte raw), lo slug della rete, l'ID della richiesta e l'ID dell'utente chiamante. **Non** conserviamo il payload firmato stesso — l'hash è recuperabile dalla receipt on-chain. Questa traccia di audit esiste in modo che, se l'API key di un cliente viene compromessa e usata per relay di transazioni illecite tramite la nostra infrastruttura, possiamo correlare l'attività on-chain con l'account responsabile.

## Esempio

```bash
curl https://api.venice.ai/api/v1/crypto/rpc/ethereum-mainnet \
  -H "Authorization: Bearer $VENICE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_chainId",
    "params": [],
    "id": 1
  }'
```

```json
{ "jsonrpc": "2.0", "id": 1, "result": "0x1" }
```

Header di risposta: `X-Venice-RPC-Credits: 20`, `X-Venice-RPC-Cost-USD: 0.00001250`, `X-Request-ID: <nanoid>`.
