A API Venice existe para dar a desenvolvedores acesso privado e irrestrito a modelos de nível de produção, livres de filtros ocultos ou decisões de caixa-preta.

À medida que os modelos melhoram, ocasionalmente retiramos os mais antigos em favor de alternativas mais inteligentes, mais rápidas ou mais capazes. Projetamos essas transições para serem previsíveis e de baixo atrito.

## Depreciações de modelo

Sabemos que as depreciações podem ser disruptivas. É por isso que buscamos depreciar apenas quando necessário, e projetamos recursos como traits e modelos com a marca Venice para minimizar a disrupção.

Podemos depreciar um modelo quando:

- Um modelo mais novo oferece uma melhoria clara para o mesmo caso de uso
- O modelo não atende mais aos nossos padrões de desempenho ou confiabilidade
- Apresenta uso consistentemente baixo, e continuar a oferecê-lo fragmentaria a experiência para todos os outros

## Processo de depreciação

Quando um modelo atende aos critérios de depreciação, anunciamos a mudança com 30–60 dias de antecedência. As notificações de depreciação são publicadas no [changelog](https://featurebase.venice.ai/changelog) e no nosso [servidor Discord](https://discord.gg/askvenice). Quando você chama um modelo depreciado durante o período de aviso, a resposta da API incluirá um aviso de depreciação.

Durante o período de aviso, o modelo permanece disponível, embora em alguns casos possamos reduzir a capacidade de infraestrutura. Sempre fornecemos uma substituição recomendada e, quando necessário, oferecemos orientação para migração para ajudar na transição.

Após a data de descontinuação, as requisições ao modelo serão automaticamente roteadas para um modelo de poder de processamento similar pelo mesmo preço ou menor. Se o roteamento não for possível por razões técnicas ou de segurança, a API retornará uma resposta 410 Gone. Se um modelo depreciado foi selecionado via uma trait (como `default_code`, `default_vision` ou `fastest`), essa trait será reatribuída a uma substituição compatível.

Nunca removemos modelos silenciosamente nem alteramos comportamento sem versionamento. Você sempre saberá o que está em execução e como se preparar para o que vem em seguida.

<Note>
Upgrades apenas de desempenho: Podemos lançar melhorias que preservam o comportamento do modelo enquanto melhoram desempenho, latência ou eficiência de custo. Essas atualizações são compatíveis com versões anteriores e não exigem ação do cliente.
</Note>

Veja o [Rastreador de Depreciações de Modelo](#model-deprecation-tracker) abaixo. Para anúncios anteriores, consulte o [changelog](https://featurebase.venice.ai/changelog) e o nosso [servidor Discord](https://discord.gg/askvenice).

## Como os modelos são selecionados para a API Venice

Selecionamos cuidadosamente quais modelos disponibilizar com base em desempenho, confiabilidade e necessidades reais de desenvolvedores. Para ser incluído, um modelo deve demonstrar desempenho forte, comportar-se consistentemente em endpoints compatíveis com OpenAI e oferecer uma melhoria clara em relação a pelo menos um dos modelos que já suportamos.


Modelos que estamos avaliando podem ser lançados inicialmente em [beta](/pt-BR/overview/beta-models) para coletar feedback e validar desempenho em escala.

Não expomos modelos que sejam redundantes, não comprovados ou que não estejam prontos para uso consistente em produção. Nosso objetivo é manter a API Venice limpa, capaz e otimizada para o que os desenvolvedores realmente constroem.

Saiba mais em [Depreciações de modelo](/pt-BR/overview/deprecations#model-deprecations) e <a href="/pt-BR/overview/models" target="_blank" rel="noopener noreferrer">Lista atual de modelos</a>.

## Versionamento e aliases

Todos os modelos Venice são identificados por um ID único e permanente. Por exemplo:

```venice-uncensored```
```zai-org-glm-4.7```
```zai-org-glm-5```
```qwen3-vl-235b-a22b```

Os IDs de modelo são estáveis. Se houver uma mudança incompatível, lançaremos um novo ID de modelo (por exemplo, adicionando uma versão como v2). Se não houver mudanças incompatíveis, podemos atualizar o modelo existente e comunicaremos mudanças significativas.

Para oferecer flexibilidade, a Venice também mantém aliases simbólicos, implementados por meio de traits, que apontam para o modelo padrão recomendado para uma determinada tarefa:

<div id="traits-list-placeholder"></div>

As traits oferecem uma abstração estável para selecionar modelos, ao mesmo tempo em que dão à Venice a flexibilidade de melhorar a implementação subjacente. Desenvolvedores que preferem acesso automático aos modelos recomendados mais recentes podem confiar em aliases baseados em traits.

Para aplicações que exigem consistência estrita e comportamento previsível, recomendamos referenciar IDs de modelo fixos.

### Verificando o status de depreciação via API

Você pode verificar se um modelo está agendado para retirada chamando o endpoint [List Models](/pt-BR/api-reference/endpoint/models/list). Modelos com data de retirada incluem um objeto `deprecation` em seu `model_spec`:

```json
{
  "id": "some-model-id",
  "model_spec": {
    "name": "Some Model",
    "privacy": "private",
    "deprecation": {
      "date": "2025-03-01T00:00:00.000Z"
    }
  },
  "type": "text",
  "object": "model",
  "owned_by": "venice.ai"
}
```

O objeto `deprecation` só aparece quando um modelo está agendado para retirada. Você pode verificar com `if (model.model_spec.deprecation)` para saber se um modelo está sendo retirado e usar a data ISO 8601 para avisar os usuários ou planejar migrações.
